## 5. False Positive Estimation

### Method: Description keyword scan

Checked whether SWE-classified descriptions mention software/code/programming-related terms:

| Tier | Total | Has SW Signal | No SW Signal | No-SW % |
|---|---|---|---|---|
| regex | 47,585 | 41,859 | 5,726 | 12.0% |
| title_lookup_llm | 3,948 | 3,151 | 797 | 20.2% |
| embedding_high | 794 | 668 | 126 | 15.9% |
| **Total** | **52,327** | **45,678** | **6,649** | **12.7%** |

Not all rows without software keywords are false positives -- some postings use domain-specific language without explicitly saying "software". But this gives an upper bound on false-positive risk.

### Suspicious title categories within SWE:

| Title Pattern | SWE Count | Mentions "software" | Mentions mfg/hardware |
|---|---|---|---|
| %automation engineer% | 550 | 50% | 35% |
| %reliability engineer% | 989 | 47% | 26% |
| %test engineer% | 699 | 67% | 56% |
| %quality assurance% | 278 | 53% | 36% |

**Automation engineer** and **reliability engineer** are the highest false-positive risk categories. Roughly 1/3 to 1/2 of these mention manufacturing/hardware in their descriptions, suggesting they are industrial rather than software roles.

### Estimated false positive rate:
- **Regex tier (90.9% of SWE):** ~3-5% false positives. Regex patterns like "software engineer" are highly precise.
- **Title_lookup_llm tier (7.5%):** ~20-25% false positives. This tier includes "automation engineer", "reliability engineer", "test engineer" which conflate software and hardware/manufacturing roles.
- **Embedding_high tier (1.5%):** ~10-15% false positives. Mixed quality.
- **Overall weighted estimate:** ~5-7% false positives (~2,600-3,700 rows out of 52,327).

## 6. False Negative Estimation

895 non-SWE rows have strong SWE title signals ("software engineer", "software developer", "full stack", "backend engineer", "frontend engineer").

**Breakdown:**
| Category | Count | Notes |
|---|---|---|
| Management (Director/Manager of SW Eng.) | 605 | Intentional exclusion -- management roles, not IC developers |
| Other (miscellaneous) | 212 | Mixed -- some may be genuine FN |
| Intern | 59 | Some may be legitimate exclusions |
| Advisory | 4 | Consultants, not developers |

**Key insight:** 605/895 (67.6%) of potential false negatives are **management roles** (e.g., "Director, Software Engineering", "Manager, Software Engineering"). These are *intentionally excluded* from SWE because they are not IC development roles. This is a correct design decision for a study of SWE posting requirements.

The remaining ~230 potential false negatives are a mix of edge cases (interns, advisory, misclassified). Against 52K SWE rows, this represents a **false negative rate of ~0.4-0.5%**, which is negligible.

## 8. Boundary Role Analysis Across Periods

This is the most analytically important part of the audit. Do boundary roles shift classification across periods?

| Role | Arshkon SWE% | Asaniczka SWE% | Scraped SWE% | Stable? |
|---|---|---|---|---|
| Data Engineer | 91.6% | 85.5% | 92.8% | Yes |
| ML Engineer | 90.9% | 96.7% | 93.3% | Yes |
| Machine Learning Eng. | 100% | 91.5% | 98.3% | Yes |
| DevOps | 97.7% | 99.3% | 99.1% | Yes |
| Site Reliability | 100% | 99.3% | 99.8% | Yes |
| Platform Engineer | 90.9% | 86.3% | 85.0% | Mostly |
| Cloud Engineer | 98.1% | 96.3% | 90.8% | Mild shift |
| Security Engineer | **1.6%** | **0.4%** | **3.3%** | Stable (non-SWE) |
| QA Engineer | 32.3% | 60.5% | 42.3% | **Unstable** |
| Test Engineer | 32.3% | 32.9% | 24.5% | Mostly |
| Solutions Architect | 2.3% | 0.5% | 0.1% | Stable (non-SWE) |

### Key findings:

1. **Core SWE roles are stably classified:** Data Engineer, ML Engineer, DevOps, SRE all classify as SWE at >85% across all periods. These are safe to include in SWE analyses.

2. **Security Engineer is consistently non-SWE:** 0.4-3.3% SWE classification. The security engineer boundary is stable and correctly placed on the SWE-adjacent side. The small uptick in scraped (3.3%) is worth monitoring but not concerning.

3. **QA Engineer is the most unstable boundary:** Classification as SWE ranges from 32% to 61% across sources. This means QA engineers are being classified inconsistently, and any temporal analysis comparing QA inclusion rates could create spurious trends.

4. **Cloud Engineer shows mild reclassification drift:** From 98.1% SWE (arshkon) to 90.8% SWE (scraped). This 7pp shift could reflect real changes in how cloud engineer roles are described, or could be a classifier artifact.

5. **Platform Engineer is borderline stable** with 85-91% SWE classification. The boundary is placed correctly (mostly SWE), but ~10-15% of platform engineers are classified as adjacent.

### Risk assessment for temporal analysis:

The unstable QA/test engineer boundary is the main concern. These roles represent:
- QA Engineer: 31-189 rows per source (small)
- Test Engineer: 54-414 rows per source (moderate)

Together, QA/test engineers represent **~1,000-2,000 rows within SWE**, or roughly 2-4% of the SWE sample. An unstable 20-30pp swing in their classification rate could shift SWE counts by ~200-600 rows. This is small relative to 52K SWE rows but could matter for compositional analyses.

**Recommendation:** For sensitivity analysis, report results with and without QA/test engineer titles.

## 10. Surprises and Unexpected Patterns

1. **"Reliability Engineer" is a major contamination risk.** 989 SWE-classified reliability engineers, but only 47% mention "software" while 26% mention manufacturing/hardware. Many of these are plant reliability, manufacturing reliability, or hardware reliability roles. This title alone could account for ~500+ false positives.

2. **Management exclusion is working correctly but creates a policy question.** 605 "Manager/Director of Software Engineering" roles are excluded from SWE. This is correct for studying IC requirements, but these roles are relevant for RQ1's "senior archetype shift" analysis. Consider a separate "SWE management" flag for these.

3. **Scraped data has higher regex purity (94.2% vs 87.9-89.2%).** This suggests 2026 LinkedIn postings use more standardized SWE title conventions, or that the scraper's search queries naturally select for explicit SWE titles.

4. **The embedding_high tier's titles are mostly malformed.** Top titles include "engineer software", "engineer systems", "engineer, software" -- these look like parsing artifacts where first/last name fields were swapped with title fields. The 794 embedding_high SWE rows should be audited for data quality issues, not just classification accuracy.

5. **title_lookup_llm contains valid SWE titles that regex missed.** "Firmware Engineer" (212), "MLOps Engineer" (155), "DevSecOps Engineer" (106), "AI Engineer" (83) are genuine SWE roles that regex did not catch. The title_lookup_llm tier adds real value despite its false positive problem. A targeted regex expansion for these specific titles would improve tier 1 precision.

6. **"Test Engineer" overlap with manufacturing is massive.** 56% of SWE-classified test engineers mention manufacturing/hardware in their descriptions. This is the single highest-risk boundary for false positives and deserves targeted exclusion rules.

---

## 11. Recommendations

1. **For primary SWE analyses:** Use `is_swe = true` with the understanding that 5-7% false positives are present. This is acceptable for most analyses.

2. **For precision-sensitive analyses:** Add filter `swe_classification_tier = 'regex'` to restrict to high-precision titles (90.9% of sample). This drops title_lookup_llm and embedding_high rows.

3. **For robustness checks:** Report results with and without:
   - QA/test engineer titles
   - Automation/reliability engineer titles
   - title_lookup_llm tier rows

4. **Pipeline improvement:** Consider expanding regex patterns to include "firmware engineer", "MLOps engineer", "DevSecOps engineer", and "AI engineer" to reduce reliance on the noisier title_lookup_llm tier.

5. **Pipeline improvement:** Add manufacturing/hardware exclusion rules for "test engineer", "reliability engineer", and "automation engineer" -- e.g., exclude when title lacks software/code qualifiers and description contains manufacturing/hardware signals.
