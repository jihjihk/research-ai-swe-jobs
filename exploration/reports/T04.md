## 5. False Positive Estimation

### Method: Software keyword scan in description

Checked whether SWE-classified descriptions mention software/code/programming-related terms (software, programming, coding, developer, python, java, javascript, typescript, c++, api, frontend, backend, full-stack, devops, deploy, repository, git, codebase):

| Tier | Total | Has SW Signal | No SW Signal | No-SW % |
|---|---|---|---|---|
| regex | 47,585 | 45,367 | 2,218 | 4.7% |
| title_lookup_llm | 3,948 | 3,482 | 466 | 11.8% |
| embedding_high | 794 | 741 | 53 | 6.7% |
| **Total** | **52,327** | **49,590** | **2,737** | **5.2%** |

Not all rows without software keywords are false positives -- some postings use domain-specific language. But this provides a reasonable upper bound on false-positive risk.

### Suspicious title categories within SWE:

| Title Pattern | SWE Count | Mentions "software" | Mentions mfg/hardware |
|---|---|---|---|
| %automation engineer% | 545 | 56% | 44% |
| %reliability engineer% | 989 | 56% | 32% |
| %test engineer% | 699 | 73% | 68% |
| %quality assurance% | 278 | 60% | 41% |
| %qa engineer% | 162 | 77% | 22% |

**Key findings:**
- **Test engineer** has the highest manufacturing keyword co-occurrence (68%), meaning nearly 2/3 of SWE-classified test engineers mention manufacturing or hardware in their descriptions. Many of these are likely product/hardware test roles, not software testers.
- **Automation engineer** has 44% manufacturing overlap, indicating a substantial fraction are industrial automation roles.
- **Reliability engineer** has 32% manufacturing overlap, lower than test/automation but still indicating meaningful contamination.
- **QA engineer** is the cleanest of the suspicious categories at 22% manufacturing overlap.

### Estimated false positive rate:
- **Regex tier (90.9% of SWE):** ~3-5% false positives. Regex patterns like "software engineer" are highly precise.
- **Title_lookup_llm tier (7.5%):** ~15-25% false positives. This tier includes "automation engineer", "reliability engineer", "test engineer" which conflate software and hardware/manufacturing roles.
- **Embedding_high tier (1.5%):** ~7-10% false positives. Slightly elevated but small absolute count.
- **Overall weighted estimate:** ~4-6% false positives (~2,100-3,100 rows out of 52,327).

## 6. False Negative Estimation

893 non-SWE rows match strong SWE title patterns (software engineer, software developer, full-stack, backend engineer, frontend engineer, web developer).

### Breakdown by category:

| Category | Count | Notes |
|---|---|---|
| Management | 610 | Director/Manager of SW Engineering -- intentional exclusion |
| Other | 159 | Mixed -- includes teaching, editor, client technology roles |
| Intern | 61 | Internship roles |
| Senior IC | 58 | Senior/Staff/Principal-prefixed roles caught by edge cases |
| Advisory | 5 | Consultants |

**Key insight:** 610/893 (68.3%) of potential false negatives are **management roles** (e.g., "Director, Software Engineering - Card Core", "Manager, Software Engineering, Back End"). These are intentionally excluded from SWE because they are not IC development roles. This is a correct design decision for a study of SWE posting requirements.

Top management false-negative titles are dominated by Capital One's naming convention ("Director, Software Engineering - Card Core", "Senior Director, Software Engineering - Card Authorizations", etc.), which inflates the count due to multi-posting.

The remaining ~283 potential false negatives include:
- **Interns (61):** Software engineering internships -- excluded by level, arguably correct
- **Teaching (21):** "College of IT: FT Instructor - Software Engineering" -- correctly excluded
- **Senior IC edge cases (58):** Roles with unusual titling that the regex doesn't capture
- **Other (159):** Titles like "Client Technology - Software Engineering" where the role may or may not be IC development

**Estimated false negative rate:** ~283 genuine misses / ~53,000 target = **~0.5%**, which is negligible.

## 8. Boundary Role Classification Across Periods

This is the analytically critical part of the audit: do boundary roles shift classification across periods?

| Role | Arshkon SWE% | Asaniczka SWE% | Scraped SWE% | Stable? |
|---|---|---|---|---|
| Data Engineer | 91.6% (359/392) | 85.5% (1,259/1,473) | 92.8% (1,888/2,034) | Yes |
| ML Engineer | 90.9% (10/11) | 96.7% (29/30) | 93.3% (249/267) | Yes (small n) |
| Machine Learning Eng. | 100% (87/87) | 91.5% (375/410) | 98.3% (738/751) | Yes |
| DevOps | 97.7% (250/256) | 99.3% (754/759) | 99.1% (994/1,003) | Yes |
| Site Reliability | 100% (91/91) | 99.3% (277/279) | 99.8% (412/413) | Yes |
| Platform Engineer | 90.7% (49/54) | 86.3% (314/364) | 85.0% (340/400) | Mostly |
| Cloud Engineer | 98.1% (105/107) | 96.2% (205/213) | 90.7% (353/389) | Mild shift |
| Security Engineer | 1.6% (4/249) | 0.4% (3/677) | 3.1% (40/1,272) | Stable (non-SWE) |
| QA Engineer | 32.3% (10/31) | 59.5% (72/121) | 42.8% (80/187) | **Unstable** |
| Test Engineer | 32.3% (54/167) | 33.0% (414/1,256) | 24.5% (231/942) | Mostly |
| Solutions Architect | 2.3% (3/133) | 0.5% (4/754) | 0.1% (1/688) | Stable (non-SWE) |

### Key findings:

1. **Core SWE roles are stably classified.** Data Engineer, ML Engineer, DevOps, SRE all classify as SWE at >85% across all periods. These are safe to include in SWE analyses.

2. **Security Engineer is consistently non-SWE.** 0.4-3.1% SWE classification. The boundary is stable and correctly placed on the SWE-adjacent side. The small uptick in scraped (3.1%) is worth monitoring but not concerning.

3. **QA Engineer is the most unstable boundary.** Classification as SWE ranges from 32.3% to 59.5% across sources. This means QA engineers are being classified inconsistently, and any temporal analysis comparing QA inclusion rates could create spurious trends. The low base count (31-187 per source) amplifies the instability.

4. **Cloud Engineer shows mild reclassification drift.** From 98.1% SWE (arshkon) to 90.7% SWE (scraped). This 7.4pp shift could reflect real changes in how cloud engineer roles are described (more infrastructure-focused in 2026), or could be a classifier artifact.

5. **Platform Engineer is borderline stable** with 85-91% SWE classification. The boundary is placed correctly (mostly SWE), but ~10-15% of platform engineers are classified as adjacent.

6. **Test Engineer classification is declining.** From 32-33% SWE (2024) to 24.5% (2026). This 8pp drop could reflect real role evolution (more specialized testing, less code) or title convention changes.

### Risk assessment for temporal analysis:

The unstable QA and drifting test engineer boundaries together affect:
- QA Engineer: 10-80 SWE rows per source (very small)
- Test Engineer: 54-414 SWE rows per source (moderate)

Combined, these represent ~1,000-1,500 rows within SWE, or roughly 2-3% of the 52K SWE sample. A 20-30pp swing in their classification rate could shift SWE counts by ~200-400 rows, which is small relative to total SWE but could matter for compositional analyses within specific role subcategories.

**Recommendation:** For sensitivity analysis, report results with and without QA/test engineer titles.

## 11. Surprises and Unexpected Patterns

1. **Test engineer has the highest manufacturing co-occurrence (68%).** Nearly 2/3 of SWE-classified test engineers mention manufacturing or hardware keywords. This is the single highest-risk boundary for false positives and the most compelling case for targeted exclusion rules.

2. **"Ruby on Rails Developer" is classified as non-SWE.** This is a clear false negative (conf=0.637, unresolved tier). A dedicated regex pattern for "Rails Developer" would fix this and similar framework-named-developer titles.

3. **"Senior Deep Learning Engineer" is SWE-adjacent, not SWE.** Classified via embedding_adjacent (conf=0.814). Given that deep learning engineering is heavily code-based, this seems like a misclassification that an expanded regex could fix.

4. **Capital One dominates false negatives.** The management false-negative list is heavily skewed by Capital One's titling convention ("Director, Software Engineering - Card Core", "Manager, Software Engineering, Android"). This is an artifact of Capital One posting many distinctly-titled management roles, not a systemic classifier problem.

5. **Embedding_high contains reversed title artifacts.** "ENGINEER SYSTEMS 3" and "ENGINEER SOFTWARE 3" appear to be cases where name/title fields were swapped during ingestion. These 20+ rows have correct classification despite garbled titles, suggesting the embedding classifier is robust to title noise.

6. **Scraped data has 94.2% regex purity vs 87.9% for asaniczka.** The scraper's search queries naturally select for postings with explicit SWE title keywords, reducing reliance on noisier classification tiers. This is a beneficial side effect that improves classification quality for the scraped period.

## 12. Recommendations

1. **For primary SWE analyses:** Use `is_swe = true` with the understanding that 4-6% false positives are present. This is acceptable for most analyses.

2. **For precision-sensitive analyses:** Add filter `swe_classification_tier = 'regex'` to restrict to high-precision titles (90.9% of sample). This drops title_lookup_llm and embedding_high rows.

3. **For robustness checks:** Report results with and without:
   - QA/test engineer titles (unstable boundary)
   - Automation/reliability engineer titles (manufacturing contamination)
   - title_lookup_llm tier rows

4. **Pipeline improvement (regex expansion):** Add regex patterns for "firmware engineer", "MLOps engineer", "DevSecOps engineer", "AI engineer", "Rails developer", "deep learning engineer" to promote these from title_lookup_llm/embedding to the more precise regex tier.

5. **Pipeline improvement (manufacturing exclusion):** Add manufacturing/hardware exclusion rules for "test engineer", "reliability engineer", and "automation engineer" -- exclude when title lacks software/code qualifiers and description contains manufacturing/hardware signals.

6. **Data quality:** Investigate reversed-title artifacts in embedding_high tier (e.g., "ENGINEER SYSTEMS 3"). These likely reflect upstream ingestion issues.
