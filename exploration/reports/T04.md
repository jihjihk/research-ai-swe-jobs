# T04: SWE Classification Audit

## Finding

The SWE classification is reliable for the regex tier (84% of SWE rows, ~2% estimated FP rate) but has a **26% false positive rate in the embedding_llm tier** (14% of SWE rows). The primary FP pattern is hardware/physical test engineers and manufacturing automation engineers being classified as software engineers. Overall estimated FP rate is ~5.5% (~1,677 of 30,283 SWE rows). False negative rate among postings with SWE-like titles is ~12%, primarily missing roles like "Elixir Engineer", "GUI Developer", and "Python Developer" that lack standard SWE title keywords.

## Borderline SWE audit (embedding_llm tier, n=50)

From 50 randomly sampled embedding_llm tier SWE postings:
- **33 (66%)** are true SWE (Java developers, .NET developers, full-stack developers, etc.)
- **4 (8%)** are borderline (MLOps engineers, QA leads in non-software contexts)
- **13 (26%)** are false positives (not primarily software development)

Common false positive patterns in embedding_llm tier:
1. **Hardware test engineers** (e.g., "Test Engineer" for opto-electronics, LabVIEW hardware test, semiconductor test)
2. **Manufacturing reliability/automation** (e.g., "Reliability Engineer" for fixed equipment, "Automation Engineer" for PLC controls)
3. **Quality assurance in manufacturing** (e.g., "Quality Assurance Engineer" for production line)
4. **Firmware engineers in hardware-primary contexts** (e.g., well services equipment)

## Borderline non-SWE audit (titles with engineer/developer/software, n=50)

From 50 randomly sampled non-SWE postings with SWE-like title keywords:
- **34 (68%)** are true negatives (civil engineers, electrical engineers, manufacturing engineers, etc.)
- **10 (20%)** are borderline (cybersecurity, network engineering, robotics)
- **6 (12%)** are false negatives (should have been classified as SWE)

False negative examples:
- "Elixir Engineer, Global Cart Team" -- clearly a programming role
- "GUI Developer" -- software GUI development
- "Python Developer" -- standard software development
- "Manager of Software Engineering" -- software management role
- "React Native developer" -- mobile development (classified as adjacent, arguable)
- "Software Engineering Team Lead" -- SE leadership (classified as adjacent, arguable)

## Regex tier quality

The regex tier (84% of SWE rows) is high-quality. Top titles include: Software Engineer (2,373), Data Engineer (861), Systems Engineer (720), Software Developer (349), DevOps Engineer (335). However, "Systems Engineer" (720 rows) includes some ambiguous cases (e.g., "Senior Electrical Systems Engineer", "Sr. AV Systems Engineer") that are hardware-focused. Estimated FP rate in regex tier is ~2%, concentrated in the "Systems Engineer" and "Product Engineer" titles.

## SWE-adjacent profile (8,486 rows)

Top titles: Network Engineer (937), Quality Engineer (863), Data Analyst (595), Architect (535), Solutions Architect (197). This group contains technical roles that involve some code or technical design but are not primarily software development. The classification is reasonable -- these roles are appropriate for RQ2 cross-occupation comparisons.

All SWE-adjacent rows come from embedding-based tiers (5,189 embedding_adjacent, 3,297 embedding_llm), which is expected since regex patterns target SWE-specific titles.

## Control group profile (125,995 rows)

Top titles: Accountant (5,132), Registered Nurse (3,207), Electrical Engineer (1,928), Accounting Manager (1,634), Mechanical Engineer (1,429). The control group is a mix of non-technical (accounting, nursing) and technical-but-not-software (electrical, mechanical, civil engineering) occupations. This provides a good baseline for cross-occupation comparisons in RQ1.

Note: 6 rows are flagged as both is_swe_adjacent and is_control (likely a minor classification edge case). 2 rows are flagged as both is_swe and is_control.

## Implication for analysis

**SWE classification is adequate for RQ1-RQ3 analysis** with the following caveats:
1. The 5.5% FP rate introduces noise but should not systematically bias period comparisons (FP rate is similar across sources/periods).
2. For sensitivity analysis, restricting to regex tier only (84% of SWE rows) provides a cleaner sample at the cost of smaller n.
3. The embedding_llm tier should be used with caution -- its 26% FP rate means ~1 in 4 rows in this tier is not actually a SWE role.
4. False negatives (missed SWE postings) are concentrated among roles with unconventional titles ("Elixir Engineer", "GUI Developer") and are likely a small fraction of the total SWE population.

## Data quality note

- SWE confidence scores are not granular within tiers: regex = 1.0, embedding_llm = 0.9, embedding_high = 0.85-0.99. There is no useful confidence gradient for within-tier filtering.
- The 0.3-0.7 confidence range specified in the task contains zero SWE rows (all non-SWE). The confidence score is binary-like rather than continuous.
- The "Systems Engineer" title (720 SWE rows) is the largest source of ambiguity in the regex tier, containing a mix of software systems engineering and hardware/electrical systems engineering.
- SWE-adjacent classification boundary is reasonable but includes roles like "Data Analyst" that some researchers would include in a broader SWE definition.
