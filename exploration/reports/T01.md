# T01: Column Coverage Audit

## Finding

Of the 80 columns in `stage8_final.parquet` (1,395,790 rows across 4 source-platform combinations), **19 columns are >50% null in at least one cross-period LinkedIn source** for the SWE subset. Core analysis columns (title, description, seniority_final, yoe_extracted, metro_area) have adequate coverage across all three periods, but **company_industry is 0% for asaniczka** (the largest source), **skills_raw is 0% for scraped LinkedIn and 1% for arshkon**, and **LLM columns are entirely absent from the stage8 file** (they exist only in stage10_llm_integrated.parquet, where LLM extraction has 0 labeled SWE rows for scraped data and 0 labeled classification rows for any source).

The data grew since the schema doc was written: 1,395,790 total rows (vs documented 1,217,299), with scraped LinkedIn at 145,583 (vs 26,853) and scraped Indeed at 73,328 (vs 13,567). SWE counts also increased: arshkon 5,024, asaniczka 24,117, scraped LinkedIn 24,101, scraped Indeed 6,137.

## Surprises & unexpected patterns

1. **LLM classification has zero labeled rows anywhere.** In stage10, all SWE rows are either `deferred` or `rule_sufficient` for classification. `seniority_llm`, `swe_classification_llm`, `ghost_assessment_llm`, and `yoe_min_years_llm` are all null for SWE. This means LLM seniority cannot be used as an analysis variable yet.

2. **LLM extraction coverage is asymmetric.** Arshkon has 99.8% labeled (5,015/5,024 SWE), asaniczka 78.4% (18,895/24,117), but scraped is 0% labeled / 100% deferred for SWE. The `description_core_llm` column is entirely unavailable for scraped data.

3. **Scraped data is 5.4x larger than documented.** The scraper accumulated considerably more data than the schema doc reflects, likely from continued daily scraping through March 31.

4. **`company_industry` has different semantics across sources.** Arshkon uses single-label industries ("IT Services and IT Consulting"). Scraped LinkedIn has compound multi-industry labels joined by "and"/"," in 49.3% of rows (e.g., "Accounting, IT Services and IT Consulting, and Business Consulting and Services"). Arshkon has 41.6% "and" labels too, but these are mostly natural compound names rather than multi-tag joins. Cross-source industry analysis requires decomposition of compound labels.

5. **`skills_raw` is unusable for cross-period comparison.** It's 96% populated for asaniczka, 1.3% for arshkon, and 0% for scraped LinkedIn. The only cross-period skills analysis must come from text mining of descriptions.

6. **`work_type` is 0% for asaniczka** (all null), limiting cross-period employment-type analysis.

7. **Scraped LinkedIn `date_posted` is only 0.7% populated for SWE** (179/24,101). Temporal analysis within the scraped period must rely on `scrape_date`.

## Implication for research

**For RQ1-RQ3, the binding data constraints are:**

| Analysis need | Binding constraint |
|---|---|
| Seniority-stratified analysis | Asaniczka has 0 native entry-level labels. 71.8% unknown overall. |
| Company industry controls | Asaniczka = 0% industry. Only arshkon-to-scraped comparisons can use industry. |
| Skills/requirements analysis | Must use description text mining, not `skills_raw`. |
| LLM-cleaned text | Unavailable for scraped data. Rule-based `description_core` is the fallback. |
| LLM seniority | Not available for any source. `seniority_final` is the only option. |
| Company size controls | Only arshkon (99%) has company_size for LinkedIn. Scraped LinkedIn = 0%. |

**New questions raised:**
- Why does LLM classification have 0 labeled rows? Is this a budget issue or a routing logic gap?
- The compound industry labels in scraped data need a decomposition strategy before any industry analysis.
- With description_core_llm unavailable for scraped, how much does the 44% boilerplate removal accuracy in `description_core` bias text analyses?

## Methodology

Computed non-null rate and distinct count for all 80 columns, stratified by source (arshkon, asaniczka, scraped_linkedin, scraped_indeed) and subset (all rows, SWE only). Used DuckDB on `stage8_final.parquet`. Also checked `stage10_llm_integrated.parquet` for LLM column presence and coverage. Examined top-value distributions for 30 key analysis columns and tested for semantic differences (compound labels, delimiter patterns) across sources.

All coverage computations are based on SQL `COUNT(column)` which counts non-null values. Boolean columns where false is a valid value are counted as non-null.

## Data quality note

- **Row counts have changed.** The stage8 file contains 1,395,790 rows, significantly more than the 1,217,299 documented in `docs/preprocessing-schema.md`. Scraped data grew from ~40K to ~219K. The schema doc needs updating.
- **`seniority_raw` = "not applicable"** in scraped data (20,962 rows, including 4,200 SWE) maps to `seniority_native = NULL`. These rows get `seniority_final` from rule-based imputation only.
- **`is_remote` and `is_remote_inferred`** are both 100% non-null everywhere but the underlying values are all false/0 for 2024 sources. This is a data artifact, not real coverage.
- **Scrape date range** for scraped data: 2026-03-20 to 2026-03-31 (12 dates for both LinkedIn and Indeed).
