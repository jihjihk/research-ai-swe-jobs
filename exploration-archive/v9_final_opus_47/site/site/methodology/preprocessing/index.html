    Preprocessing pipeline¶

    Why preprocessing exists¶

    The raw input to this exploration is three heterogeneous LinkedIn panels: two Kaggle snapshots from 2024 (arshkon and asaniczka) and an ongoing scraped corpus from 2026-03 onward. Each source arrives with different schemas, different coverage of fields like industry and company size, different seniority-label conventions, and different text conventions (scraped markdown vs arshkon fused vs asaniczka pipe formats). None of the three is directly queryable against the others.

    Preprocessing turns this into a single analysis-ready dataset with:

    • A unified 39-column schema with a stable uid primary key.
    • Occupation, seniority, YOE, and location classifications that are source-independent.
    • LLM-cleaned descriptions with boilerplate removed for text-sensitive analyses.
    • Provenance flags so every row can be audited back to its source.

    Without this, cross-period comparison is statistically meaningless — every apparent "shift" could be platform-schema drift or classifier drift.

    Pipeline overview¶

    Ten stages organized in two layers:

    Raw Data (3 sources)
  |
  +-- Stage 1: Ingest & Schema Unification
  |     +-- 1a: Arshkon ingest + companion joins
  |     +-- 1b: Asaniczka ingest + description/skills joins
  |     +-- 1c: Scraped ingest (LinkedIn + Indeed)
  |     +-- 1d: Concatenation to canonical 39-column format
  +-- Stage 2: Aggregator / Staffing Handling
  +-- Stage 4: Company Canonicalization + Dedup
  +-- Stage 5: Occupation + Seniority Classification
  +-- Stage 6: Location Normalization       --+
  +-- Stage 7: Temporal Alignment             | (single script)
  +-- Stage 8: Quality Flags & Provenance   --+
  |
  V
  stage8_final.parquet   <-- rule-based baseline
  |
  +-- Stage 9: LLM Boilerplate Removal
  +-- Stage 10: LLM Classification + Final Integration
  |
  V
  data/unified.parquet            (one row per unique posting; full schema)
  data/unified_observations.parquet  (daily panel: posting x scrape_date)
  data/unified_core.parquet       (analysis-ready subset: selected_for_llm_frame = TRUE)

    Stage 3 is intentionally absent — the original Stage 3 was removed because it did not add useful signal.

    Two-layer design rationale¶

    1. Rule-based baseline (Stages 1-8) — deterministic, fast (~30 minutes end-to-end), reproducible. Produces a usable corpus with regex/lookup-based labels. Can be run independently of LLM stages.

    2. LLM augmentation (Stages 9-10) — Codex (GPT-5.4-mini) by default, with Claude and OpenAI optionally enabled. Takes hours to days depending on API quotas. Rule-based columns are preserved as both classification fallbacks and cache keys.

    Only Stage 4 reduces row count (deduplication). Every other stage is row-preserving.

    Stage-by-stage summary¶

    Stage What it does Why
    1 — Ingest Loads 3 sources, applies source-specific joins, concatenates to canonical 39-column format. Different sources have different schemas; downstream needs a single surface.
    2 — Aggregators Identifies staffing agencies (Dice, Lensa, Robert Half) via exact-name + regex; extracts the real employer from description text; derives company_name_effective. Aggregators distort company-level analyses; need to resolve them to real employers.
    4 — Dedup The only row-reducing stage. Canonicalizes company_name_effective into company_name_canonical; removes exact duplicates, fuzzy near-duplicates (token_set_ratio >= 85%), and multi-location collapses. Same posting can appear 2+ times across different scrape runs or across different city/state variants; need one row per unique posting.
    5 — Classification Writes is_swe, is_swe_adjacent, is_control via a 3-tier system (regex, curated title lookup, embedding). Extracts rule-based YOE with a clause-aware parser. Writes immutable seniority_rule + seniority_rule_source. First analytical classification boundary; downstream stratifications depend on it.
    6-8 — Normalize Single script covering location parsing (city/state/country/remote), temporal alignment (period, posting_age_days), and quality flags (language detection, date validation, ghost-job heuristics). Need comparable geography, time, and quality fields across sources.
    9 — LLM extraction Selects the deterministic "sticky core" LLM frame over source x analysis_group x date_bin. Segments descriptions into sentence units; LLM identifies boilerplate units for removal. Produces description_core_llm — the only cleaned-text column. Hard-skips descriptions under 15 words. Boilerplate (benefits, EEO language, company overview) inflates length and noise; text-sensitive analyses need cleaned text.
    10 — LLM classification Reuses the Stage 9 core frame. Routes every eligible row to LLM for 4 tasks: SWE classification, seniority, ghost assessment, and YOE extraction. The LLM seniority result overwrites seniority_final when non-null. Rule-based seniority misses ~34 to 53% of rows (the LLM abstains on unclear cases too, per design). LLM YOE is the primary signal within the frame.

    Output data structure¶

    The final outputs are four parquet files:

    File Unit of observation Purpose
    unified.parquet one row per unique posting Full schema; use for audits and out-of-LLM-frame rows
    unified_observations.parquet one row per posting x scrape_date Full-schema daily panel
    unified_core.parquet one row per unique posting Default analysis dataset. Strict projection filtered to selected_for_llm_frame = TRUE; curated column set.
    unified_core_observations.parquet one row per posting x scrape_date Daily panel for the core subset

    Column families in unified_core.parquet¶

    Group Columns
    Identity uid
    Source & time source, source_platform, period, date_posted, scrape_date
    Job content title, description, description_core_llm, description_length
    Company company_name, company_name_effective, company_name_canonical, is_aggregator, company_industry, company_size
    Occupation is_swe, is_swe_adjacent, is_control, analysis_group, swe_classification_tier, swe_classification_llm
    Seniority seniority_final, seniority_final_source, seniority_3level, seniority_rule, seniority_rule_source, seniority_native
    YOE yoe_extracted, yoe_min_years_llm
    Geography location, city_extracted, state_normalized, metro_area, is_remote_inferred, is_multi_location
    Quality is_english, date_flag, ghost_job_risk, ghost_assessment_llm
    LLM coverage llm_extraction_coverage, llm_classification_coverage

    Source-specific gaps (why pooled-vs-arshkon-only matters)¶

    Field arshkon asaniczka scraped (LinkedIn)
    description 100% 96.2% (joined) 100%
    seniority_native 66.5% 100% but no 'entry' level 100%
    date_posted 100% 100% 2.8%
    company_industry 99.3% (joined) 0% 100%
    company_size joined 0% 0%
    Entry-level native labels yes none yes

    asaniczka has no native entry labels at all — a key reason the exploration uses YOE-based J3/S4 as primary and runs T30's 13-definition panel to check direction-consistency.

    LLM stages — see the dedicated page¶

    The LLM prompts used in Stages 9 and 10 (extraction and classification) are shown in full on the LLM prompts page. Prompts are versioned by content hash; any modification forces cache re-resolution.

    Budget and coverage caveats¶

    Stages 9 and 10 take an explicit --llm-budget (no default); the budget splits across three categories at 40/30/30 (SWE / SWE-adjacent / control). Coverage columns track per-row outcomes:

    • labeled — has LLM results
    • deferred — eligible but budget-capped this run
    • not_selected — not in the Stage 9 core frame
    • skipped_short — description under 15 words

    Scraped 2026 coverage: 56.9% labeled, 43.0% not_selected, 0.02% deferred. Text-sensitive analyses must restrict to labeled or report a labeled-vs-not split.

    Cache design: all LLM responses are cached in SQLite (preprocessing/cache/llm_responses.db) keyed by (input_hash, task_name, prompt_version). Re-running with a higher budget adds fresh calls within the deterministic core frame without re-costing cache hits.

    Memory constraint¶

    The pipeline runs on a 31 GB RAM machine. Every stage uses pyarrow chunked reads with 200K-row batches (50K for LLM stages) to avoid loading full parquet files into pandas. See docs/preprocessing-guide.md for the full development practices document.
