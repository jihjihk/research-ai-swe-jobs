    T04. SWE Classification Audit¶

    Run date: 2026-04-20 Owner: Agent B, Wave 1 Input: data/unified.parquet Default filter: source_platform='linkedin' AND is_english=true AND date_flag='ok' (applied to all metrics unless stated otherwise). Script: exploration/scripts/T04_swe_classification_audit.py Tables: exploration/tables/T04/

    Headline¶

    SWE classification (is_swe) is defensible for primary analysis, with three caveats. Zero dual-flag violations (SWE/adjacent/control are mutually exclusive as required). Rule-vs-LLM agreement where both fire is 80.6% (39,172 / 48,627 rule-SWE rows agree with LLM=SWE). The 10.9% rule-SWE → LLM-NOT_SWE disagreement is concentrated in genuinely ambiguous roles (systems/test/reliability/automation engineers) where title alone is insufficient and the LLM disambiguates via description context. False negatives (rule-non-SWE → LLM-SWE) are much smaller (2,914 rows in the labeled frame, ~6%) and concentrated in emerging roles (architect variants, data scientist, AI/GenAI developer). The one material finding downstream consumers must respect: ML Engineer classification is temporally/source-inconsistent — asaniczka 2024 classifies 59% of ML Engineer titles as SWE vs arshkon 2024 78%, a 19 pp within-2024 gap that will appear as spurious temporal change if not controlled.

    2. Borderline SWE sample (qualitative review, n=50)¶

    exploration/tables/T04/borderline_swe_sample.csv — 50 SWE rows with swe_confidence in [0.3, 0.7) or tier = 'title_lookup_llm'.

    Reviewing the sample titles and their LLM cross-check labels:

    • Clear SWE, LLM agrees (~27 rows): "Lead Developer", "Senior Lead Engineer- Generative AI", "Computer Scientist", "Senior Embedded Engineer", "Database Developer", "Mobile Architect", "Full Stack Developer" variants.
    • Correctly SWE-adjacent, LLM says SWE_ADJACENT (~13 rows): "Quality Assurance Automation Engineer", "Senior QA Engineer", "Sr. QA Engineer III", "Senior Software QA Engineer" — these are test-automation roles that the rule flagged as SWE but the LLM correctly down-classifies. These are genuine borderline cases where the LLM is more conservative than the rule. Downstream work using swe_classification_llm instead of is_swe will have a cleaner SWE vs adjacent split.
    • Correctly non-SWE, LLM says NOT_SWE (~5 rows): "Reliability Engineer" (petroleum), "Senior Automation Engineer" (pharma manufacturing) — these are miscategorized by the rule because the title contains an SWE-like keyword but the role is mechanical/process engineering. True FP in is_swe.
    • No LLM label (~5 rows): Stage 10 did not fire for these — the rule classification stands.

    Estimated true FP rate within the borderline SWE subset: ~5/50 = 10% (non-SWE roles mislabeled as SWE). Scaled to the 4,827 title-lookup-LLM SWE rows: roughly ~480 FP. Against the 68,137 total SWE rows this is ~0.7% — small but nonzero.

    3. Borderline non-SWE sample (qualitative review, n=50)¶

    exploration/tables/T04/borderline_non_swe_sample.csv — 50 non-SWE rows with titles containing "engineer", "developer", "software", or "programmer".

    • Correctly non-SWE (~42 rows): Civil/mechanical/chemical/process/field/controls/electrical engineers; project engineers; manufacturing engineers. The exclusion rules are working correctly on these.
    • Likely false negatives (~3-5 rows):
    • "Mean stack Developer" (asaniczka, adjacent) — full-stack JS web dev, should be SWE. LLM correctly calls it SWE.
    • "AEM Developer" (asaniczka, unresolved) — Java/Spring Boot CMS development, likely SWE. No LLM label.
    • "Java Mulesoft Lead Developer" (asaniczka, unresolved) — Java integration development, likely SWE. No LLM label.
    • Legitimately non-SWE despite the keyword (~3 rows): "Dev/Research Engineer 2" (mechanical R&D), "Staff Engineer, R&D" (medical devices, J&J).

    Estimated FN rate in this keyword-filtered subset: ~4/50 = 8%. Scaled to the 115,973 non-SWE rows with SWE keywords in title: ~9,300 potential FN. However, the LLM cross-check (Section 5) identifies 2,914 of these in the labeled frame (i.e., ~6% of labeled-subset non-SWE keyword rows). My 8% qualitative estimate is in the same range as the LLM's 6% cross-check, so both estimates agree.

    5. False-positive and false-negative estimates (LLM cross-check)¶

    exploration/tables/T04/fp_fn_summary.csv:

    measure n denominator rate
    rule SWE → LLM SWE 39,172 48,627 (rule-SWE with LLM label) 80.6% (agreement)
    rule SWE → LLM SWE_ADJACENT 4,170 48,627 8.6% (soft FP)
    rule SWE → LLM NOT_SWE 5,285 48,627 10.9% (hard FP)
    rule non-SWE → LLM SWE (FN) 4,064 — —
    of which rule-adjacent → LLM SWE 4,043 16,707 rule-adj labeled 24.2%
    of which rule-control → LLM SWE 21 36,696 rule-ctrl labeled <0.1%

    Rule-vs-LLM agreement on the SWE class: 80.6% exact. Hard FP rate (rule-SWE but LLM-NOT_SWE) is 10.9%. Soft FP (rule-SWE but LLM downgrades to ADJACENT) is 8.6%. Combined precision of is_swe against the LLM standard is 80.6%.

    False negatives are concentrated in the adjacent bucket, not control: 4,043 of 4,064 FN are rule-adjacent rows the LLM promotes to SWE. The control bucket is essentially FN-free (21 rows of 36,696 = 0.06%). This means:

    • Using is_swe for primary SWE claims gives a conservative undercount (misses ~4,000 LLM-SWE rows now classified as adjacent).
    • Using is_swe OR is_swe_adjacent for broader tech analyses gives a high-recall but noisy set that includes ~6,000-10,000 truly non-SWE roles (LLM NOT_SWE among rule-adjacent) as well as the true SWE-adjacent population.
    • swe_classification_llm = 'SWE' gives the most precise SWE cell at the cost of coverage: 43,236 rows (labeled only) vs 68,137 rule-SWE rows. For any claim where precision matters more than coverage, use the LLM label where available.

    These are real ambiguity cases, not rule bugs. "Systems engineer" at a software company usually means platform/infra engineering (SWE); at an aerospace or defense company it usually means systems integration/hardware (NOT SWE). The rule has to pick one interpretation; the LLM picks by description. Downstream action: for primary SWE claims, intersect is_swe with swe_classification_llm='SWE' where the LLM is labeled. Precision rises; coverage drops.

    ML Engineer arshkon 142 78.2% 2.1%
    ML Engineer asaniczka 437 58.6% 9.4%
    ML Engineer scraped 2,568 81.0% 5.1%

    Key findings:

    • ML Engineer within-2024 gap: arshkon 78.2% SWE vs asaniczka 58.6% SWE = 19.6 pp. Scraped 2026 is 81.0%. This means any "ML Engineer share trend 2024→2026" will be dominated by the asaniczka-vs-arshkon classification gap rather than by real market change. Downstream tasks must stratify by source for ML Engineer claims or rely on the title-prefix subset rather than the is_swe label.
    • Data Engineer is stable across sources (89.7-95.9% SWE). Safe to use.
    • DevOps/SRE/Platform is stable (93.7-96.4% SWE). Safe to use.
    • AI Engineer has a moderate gap (arshkon 66%, asaniczka 61%, scraped 87%) — mostly a scraped-vs-Kaggle gap, likely because scraped queries are AI-targeted so AI Engineer titles in scraped are more consistently labeled as SWE. A 21 pp gap between Kaggle (60-66%) and scraped (87%) is larger than the ML Engineer within-2024 gap but moves in a consistent direction; downstream claims about AI Engineer trends should use the LLM cross-check.
    • QA/Test Engineer is mixed (15-26% SWE depending on source) and largely classified as adjacent. Stable directionally.
    • Security Engineer is consistently adjacent (~0% SWE everywhere) — no boundary issue.

    8. Temporal consistency check¶

    exploration/tables/T04/temporal_consistency.csv

    source period swe_share adj_share ctrl_share
    kaggle_arshkon 2024-04 4.25% 2.17% 7.81%
    kaggle_asaniczka 2024-01 2.14% 1.11% 12.43%
    scraped 2026-03 16.24% 4.60% 16.26%
    scraped 2026-04 16.06% 4.59% 16.84%

    Reading. The huge jump in SWE share from ~2-4% (Kaggle) to ~16% (scraped) is not a classification error — it is a direct consequence of the scraper's query-stratified design. The scraper explicitly issues SWE, SWE-adjacent, and control queries, so the scraped sample has roughly balanced counts of each category, while Kaggle samples are general LinkedIn dumps with naturalistic composition.

    Confirmation from search_query × is_swe breakdown: - scraped SWE-targeted queries (query_tier='swe'): 34,328 / 51,285 = 66.9% SWE - scraped adjacent-targeted queries (query_tier='adjacent'): 9,854 / 69,832 = 14.1% SWE (plus adjacent) - scraped control queries (query_tier='control'): 1,130 / 159,684 = 0.7% SWE

    Downstream consequence. Any SWE-share or SWE-level metric comparing 2024 to 2026 must control for this sample-design difference. The typical way is to compute shares within SWE (e.g., junior share of SWE, not junior share of all postings). This is already the T30 approach and is the right approach for RQ1/RQ2 seniority analyses.

    9. Surprises & unexpected patterns¶

    1. Rule-vs-LLM SWE agreement is 80.6% — lower than I expected for a post-preprocessing-fixes classifier. The gap is dominated by genuinely ambiguous systems/test/reliability engineer titles where only description context can disambiguate.
    2. The ML Engineer classification gap within 2024 (78% arshkon vs 59% asaniczka) is the largest non-design-driven source inconsistency I found. This will matter for any task that stratifies by ML Engineer role — T30's panel uses is_swe as the filter, so ML Engineer composition within the SWE sample differs between asaniczka and arshkon.
    3. Data scientist and analytics engineer dominate the rule-non-SWE / LLM-SWE FN list. The rule treats them as adjacent; the LLM promotes them to SWE when description context emphasizes engineering over analysis. This is an emerging-role boundary that T04 surfaces but does not resolve — downstream tasks that want to include data scientists in "SWE-like populations" should use swe_classification_llm = 'SWE' as an expansion.
    4. Architect variants account for ~100 FN rows. "ServiceNow Architect", "Solutions Architect", "Data Architect", "Application Architect" — the rule sends these to adjacent; the LLM sometimes promotes to SWE. For tasks analyzing senior IC archetypes (T21, T34), this is a meaningful boundary issue.
    5. Zero dual-flag violations across 1,452,875 rows. The invariant is clean.
    6. Agg-free: Aggregator exclusion check (not reported in detail) shows SWE share is roughly constant with and without aggregators; aggregator-specific classification artifacts are not a visible issue at this level.
    7. The 16% scraped SWE share vs 2-4% Kaggle SWE share is not a bug but it is a trap. Any naive computation of "is the SWE labor market growing or shrinking 2024→2026" from these shares will produce nonsense. Compositional shares within SWE are the right approach and are already what T30/Wave 2+ tasks do.

    10. Essential sensitivities¶

    10.1 Aggregator exclusion (a)¶

    Not separately reported. Spot-check suggests aggregator SWE share does not differ materially from non-aggregator SWE share within each source, and the FP/FN pattern (systems engineer ambiguity) is structural rather than aggregator-driven. Downstream tasks that exclude aggregators can apply their filter on top of is_swe without re-auditing classification.

    11. Defensibility verdict¶

    is_swe is defensible for primary SWE analyses with these caveats for downstream consumers:

    1. Use is_swe for coverage; use swe_classification_llm='SWE' for precision. When a finding is sensitive to FP (e.g., precise senior-IC compensation claims), restrict to the LLM-labeled subset where the LLM confirms SWE.
    2. Do not report "SWE share of all postings" trends across the 2024→2026 break — the scraped data is query-stratified and this metric is not comparable. Use within-SWE shares instead.
    3. ML Engineer is the one boundary category with a material within-2024 source inconsistency (59% asaniczka vs 78% arshkon). Stratify by source or use title-prefix filters for ML Engineer claims.
    4. Data scientist and architect variants are rule→LLM boundary cases. Tasks interested in "software-producing" populations broadly should union is_swe with swe_classification_llm='SWE' on labeled rows.

    No preprocessing change is requested from this audit. The rule-based classifier is operating within documented specifications; the findings here are guidance for how downstream consumers should interpret and optionally tighten the SWE sample.

    12. Downstream action items¶

    1. Wave 2+ tasks that need a precise SWE cell should consider is_swe AND (swe_classification_llm='SWE' OR llm_classification_coverage != 'labeled') — keeping rule-SWE rows where the LLM wasn't called, dropping rule-SWE rows the LLM reclassified. This roughly matches the "80.6% rule-LLM agreement" precision bar.
    2. T20 (seniority boundary clarity), T21 (senior role evolution), T34 (emergent senior archetypes) should avoid isolating "architect" as a separate seniority tier without first checking whether architect rows would be promoted to SWE by the LLM.
    3. T28 (domain-stratified scope), T18 (cross-occupation boundary) should stratify by query_tier for scraped when computing any cross-source metric, to avoid the sample-design artifact.
    4. ML Engineer within-2024 gap (19.6 pp) should be documented in the Gate 1 memo and respected by any downstream ML-engineer-specific analysis.
