    T01 — Data Profile & Column Coverage¶

    Agent A, Wave 1 Date: 2026-04-15 Input: data/unified.parquet (1,397,647 rows, 94 columns) Script: exploration/scripts/T01_profile.py Tables: exploration/tables/T01/ Figures: exploration/figures/T01/

    Default filter applied unless noted: source_platform = 'linkedin' AND is_english AND date_flag = 'ok'.

    Headline finding¶

    Three data sources with very uneven LLM text coverage are the binding constraint for every text-dependent Wave 2 analysis. After the default LinkedIn filter, we have 1.09M rows across three sources and 63,701 SWE rows; but description_core_llm is labeled for only 30.7% of scraped SWE rows, versus 99.9% for arshkon and 94.0% for asaniczka. The 2026 current window is where text coverage is thinnest — exactly the period whose content changes we most want to characterize. Any text-based 2024-vs-2026 comparison that restricts to labeled rows is operating on a 2026 SWE sample of ~12.5K, not 40.9K. No backfilling with raw text is allowed for boilerplate-sensitive work, so this cap is real.

    Secondary findings: - company_industry is 0% in asaniczka, making any industry-stratified 2024 baseline arshkon-only. Arshkon has 374 distinct industries (42.3% compound), scraped LinkedIn has 6,569 (49.8% compound) — different taxonomies and different granularity. - yoe_extracted covers only 44.9% (arshkon) / 46.4% (asaniczka) / ~59% (scraped) — the YOE-based seniority proxy has ~55% missingness in 2024 sources. - 65 source x column cells are >50% null — most of them expected (platform-exclusive metadata), but several bite: description_core_llm in scraped, asaniczka_skills being asaniczka-only, company_industry being zero in asaniczka, etc.

    "Columns with different semantics across sources" (important for Wave 2+):

    1. company_industry — arshkon has 374 distinct values from a LinkedIn companion table; scraped LinkedIn has 6,569 distinct values from the live scraper page scrape; asaniczka has zero. The compound-label rates differ too (arshkon 43%, scraped 50%). A cross-period industry comparison must either (a) restrict to a harmonized shared vocabulary or (b) be reported as arshkon-vs-scraped only, explicitly.
    2. seniority_native — arshkon has all 5 levels (including entry and intern); asaniczka has only associate and mid-senior (no entry labels at all by construction); scraped LinkedIn has all levels; Indeed has none. Asaniczka cannot produce a native entry-level signal.
    3. date_posted vs scrape_date — arshkon/asaniczka carry date_posted only; scraped LinkedIn carries scrape_date with only 2.8% date_posted, so posting age and any temporal dynamics finer than scrape week require Indeed (or scraped Indeed for within-window age work).
    4. search_query / search_metro_* — scraped-only query provenance. Do not mistake search_metro_name for posting location; use metro_area.
    5. skills_raw vs asaniczka_skills — two completely different structured-skills columns. Only asaniczka has structured skills; arshkon and scraped have none (or 1.9% in arshkon, mostly empty). Any "skills field" analysis is an asaniczka-only exercise.
    6. company_size — arshkon via companion, Indeed direct, LinkedIn scraped/asaniczka have none. Not usable for cross-source company-size analysis.

    Methodology¶

    1. Loaded data/unified.parquet as a DuckDB view — no full-file pandas load.
    2. For row counts: authoritative aggregates by source × source_platform × period × is_swe/is_swe_adjacent/is_control, both unfiltered and under default LinkedIn filter.
    3. For LLM coverage: grouped SWE rows by llm_extraction_coverage and llm_classification_coverage per source.
    4. For column coverage: per column, computed nonnull_rate = count(CASE WHEN col IS NOT NULL AND CAST(col AS VARCHAR) <> '' THEN 1 END) / count(*) per source, under the default filter. description_core_llm uses the empty-string guard because Stage 9 writes empty strings for short-description skips.
    5. Coverage heatmap: 94 columns × 4 source slices. Values are non-null rates on the default-filtered set (SWE-only for the focused heatmap, all rows for the full one). Indeed column is computed on the raw Indeed slice without the default filter (Indeed is excluded from it by construction).

    Surprises and unexpected patterns¶

    1. The 2026 scraped window has WORSE LLM coverage than the 2024 historical snapshots for SWE rows, despite the scraper being the live instrument. 30.7% labeled vs 94-100% for Kaggle is the reverse of the naive expectation. Mechanism: Stage 9 selection frame was sized for the total historical dataset and is now being outgrown by the scraped window; not_selected is 69% of scraped SWE. This has immediate downstream implications for every text-based 2024→2026 comparison — the 2026 side is the bottleneck, not the 2024 one.
    2. Scraped LinkedIn date_posted is 1.4% populated. The 2.8% figure in the schema doc is for the full scraped LinkedIn slice — on the post-filter frame it drops further. This means within-scrape-window posting dynamics (how stale are postings?) are essentially not measurable on LinkedIn, only on Indeed.
    3. company_industry has 6,569 distinct values in scraped vs 374 in arshkon. The scraped taxonomy is live LinkedIn page-scrape and is an order of magnitude finer; a naive cross-source industry rollup will fragment the 2026 side and look like "industry diversity growth" when it is instrument difference.
    4. Indeed SWE volume (11,863) is larger than arshkon SWE volume (4,691). Indeed is doc-marked "sensitivity only," but for non-seniority-stratified sanity checks it is actually the larger 2026 SWE frame and worth leaning on where seniority stratification is not required.
    5. real_employer is populated on only 0.3% of arshkon rows despite the aggregator-resolution machinery. Most arshkon aggregator rows did not get an employer extracted, so deduplication-by-canonical-employer is weaker on arshkon than on scraped (3.5%).
    6. Scraped SWE n_control ≈ n_swe (41,790 vs 40,881). The control cohort is on par with the SWE cohort in the current window, which makes cross-occupation DiD (T18) feasibly powered on the 2026 side.

    Data quality caveats¶

    • Default filter loses essentially nothing. LinkedIn slices lose <0.3% to is_english or date_flag = ok per default_filter_cost.csv. Indeed loses 15 rows to is_english and zero to date_flag. The default filter is a near-free screen.
    • seniority_native in arshkon has a 1,408-row null block on SWE rows (30% of arshkon SWE) — LinkedIn did not label those at the source. Plan around this in T02.
    • yoe_extracted is missing in ~55% of 2024 SWE rows and ~41% of scraped SWE rows. When the YOE proxy is used for label-independent validation, the sample halves. This is a separate and compounding sample-size issue from the LLM text coverage.
    • description_core_llm empty string ≠ null. Stage 9 writes empty strings for short-description hard skips. The coverage computation here uses an empty-string guard; downstream scripts must do the same or they will overcount usable rows.
    • Supplemental cache vs selected core. selected_for_llm_frame = sticky balanced core only. Rows with llm_extraction_sample_tier = supplemental_cache have LLM output but are NOT part of the balanced frame — any balanced-sample claim must filter to core.
    • company_industry distinct count in asaniczka is literally zero (0 non-null rows). Any script that computes "top industries per source" must handle this gracefully or it will silently drop asaniczka.

    Action items for downstream tasks¶

    1. For every text-based Wave 2 task (T09, T11, T12, T13, T14, T15, T21): filter to llm_extraction_coverage = 'labeled' and report the post-filter N per period up front. The effective 2026 SWE sample is ~12.5K, not 40.9K. If a task needs more, it must explicitly opt in to raw-text fallback and run the boilerplate sensitivity.
    2. T03 (Agent B seniority audit): use seniority_final_source composition table seniority_final_source_full.csv from T02 as input — it partitions SWE rows into title_keyword/llm/unknown buckets per source.
    3. T04 (Agent B SWE audit): swe_classification_llm labeled share on scraped SWE is 17%. Spot-check accuracy must be weighted toward scraped tier composition, not documented numbers.
    4. T06/T17 (Agent C/J): industry analysis is arshkon-vs-scraped only. Asaniczka drops out entirely. Do not pool.
    5. Pool-vs-arshkon decision: for 2024 baselines the cost of pooling arshkon+asaniczka is that you lose company_industry, gain asaniczka_skills, and change the native-seniority instrument. For text analyses where the LLM coverage is what matters, pooling helps (arshkon 4.7K + asaniczka 17K labeled vs scraped 12.5K labeled); for industry analyses pooling is impossible.
    6. Indeed is a larger 2026 SWE frame than arshkon is a 2024 SWE frame (11.9K vs 4.7K). For non-seniority-stratified sensitivity checks, Indeed has real statistical weight — use it more aggressively than "sensitivity only" framing might suggest.
    7. Scraper growth will worsen, not improve, text coverage until the Stage 9 selection target is raised. Flag this to partner: the balanced core frame needs to grow with the scraped window.
