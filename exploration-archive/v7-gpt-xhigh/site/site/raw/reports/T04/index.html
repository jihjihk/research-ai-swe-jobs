    T04. SWE Classification Audit¶

    Headline¶

    The SWE flag is usable for Wave 1/Gate 1, with one important sensitivity: title_lookup_llm is the fragile tier. The full SWE sample is dominated by regex matches (91.2%), while the title lookup tier is only 7.2% overall but carries most of the visible false-positive risk. False negatives remain visible among unresolved titles like Ruby on Rails Developer, ETL Developer, and Software Integration Engineer.

    Evidence base: existing T04 tables in exploration/tables/T04/. Manual assessments below use the two existing 50-row sample CSVs only. No full data rerun was performed.

    Borderline SWE Manual-Sample Assessment¶

    The borderline SWE sample is 50 rows from is_swe = true where either swe_confidence is 0.3-0.7 or swe_classification_tier = 'title_lookup_llm'. In the current artifact, all sampled rows are title_lookup_llm with confidence 0.9.

    Manual assessment from title plus excerpt:

    • Likely primary SWE or software-test/development roles: about 38/50.
    • Likely false positives or SWE-adjacent/non-SWE roles: about 12/50.
    • Main false-positive families: generic Test Engineer, Reliability Engineer, Automation Engineer, Quality Assurance Engineer, and Quality Engineer in aerospace, electronics, manufacturing, medical-device, carwash/equipment, or field reliability contexts.
    • Clear true positives in the same tier: ReactJS Developer, Angular developer, AI Developer, DevSecOps Engineer, Firmware Engineer, Java Architect, UI Developer, Software Engineering Manager, and software QA/test automation roles.

    Estimated precision for this deliberately fragile tier is roughly 76% in the sample. Since title_lookup_llm is 7.2% of all SWE rows, the rough full-SWE false-positive contribution is about 1.7 percentage points if regex and embedding-high tiers are assumed high precision. This is not a full-population false-positive rate.

    Borderline Non-SWE Manual-Sample Assessment¶

    The borderline non-SWE population is is_swe = false with title-normalized matches on engineer, developer, or software. It is large: 97,786 rows across the four source-periods in the existing population table.

    Manual assessment of the existing 50-row sample:

    • Likely missed primary SWE: about 6/50.
    • Additional SWE-adjacent or ambiguous technical misses: about 6/50.
    • Clear false negatives include Ruby on Rails Developer, Principal Engineer, YouTube ML Recommendations Systems, ETL Developer, Software Integration Engineer, and probably some BI/data/reporting developer roles.
    • Correct non-SWE exclusions dominate the sample: civil, structural, mechanical, electrical, industrial, geotechnical, project, process, field service, data-center facilities, and manufacturing engineering roles.

    This sample is enriched for hard cases and should not be converted into a dataset-wide false-negative rate. It does show that unresolved developer/software titles need a sensitivity check when downstream analyses depend on high recall.

    Group n

    Top SWE-adjacent titles are conceptually close but distinct from primary SWE: quality engineer, data scientist, network engineer, data analyst, architect, solutions architect, security engineer, systems administrator, data architect, and database administrator.

    Top control titles look distinct from SWE: accountant, registered nurse, electrical engineer, financial analyst, mechanical engineer, accounting manager, civil engineer, and nurse variants. The control group is broad and should not be interpreted as a single occupation family.

    ML engineer, data engineer, and DevOps/SRE/platform roles are classified consistently as SWE. Sales/solutions engineer and support engineer are mostly excluded, which is desirable.

    The unstable boundary is QA/test/SDET: most rows are neither SWE nor adjacent, but a meaningful subset is SWE or adjacent. This boundary can vary by source and employer wording. Generic software engineer also has a surprisingly large no-flag remainder, which should be checked before any high-recall SWE sensitivity.

    False-Positive and False-Negative Estimates¶

    Best estimate from existing samples:

    • False positives: approximately 24% within the title-lookup borderline SWE sample, concentrated in test/reliability/automation/quality roles outside software. Because title lookup is only 7.2% of SWE, this implies a rough full-SWE FP contribution around 1.5-2.0 percentage points under high-precision assumptions for other tiers.
    • False negatives: approximately 12% primary-SWE misses in the enriched borderline non-SWE sample, or up to 24% if SWE-adjacent/ambiguous technical roles are included. This is not a dataset-wide FNR because the sample intentionally selects titles with engineer, developer, or software.

    Recommended sensitivity for downstream tasks: rerun key findings excluding swe_classification_tier = 'title_lookup_llm'. For high-recall checks, separately profile unresolved developer, software, ETL, BI, integration, and ML recommendations titles.

    Surprises & Unexpected Patterns¶

    • The most fragile positive tier is smaller in 2026 than in 2024, so false positives may bias historical/current comparisons if ignored.
    • QA/test roles are the hardest occupational boundary. Some are software automation roles; others are hardware, manufacturing, or compliance test roles.
    • Several obvious software titles appear in the borderline non-SWE sample. This suggests recall risk in the unresolved pool, not just precision risk in the SWE pool.
    • Data engineer and ML engineer classification is very stable, which is useful because those roles often sit near the SWE/SWE-adjacent boundary.
