    T01 Data Profile & Column Coverage¶

    Headline finding¶

    The primary LinkedIn analysis frame is large enough for the project, but the usable surface is uneven. In the current data/unified.parquet, the primary frame contains 1,186,281 LinkedIn rows, including 59,972 SWE rows. The main binding constraints for later waves are text coverage in the scraped SWE slice, the source-specific nature of several company fields, and the fact that seniority_native is not a reliable entry-level cross-source baseline.

    Methodology¶

    I profiled the primary analysis frame (source_platform = 'linkedin' AND is_english = true AND date_flag = 'ok') and also checked the full source/platform mix to keep the scraped Indeed tail visible. For each column, I computed non-null rate and distinct count by source, with a second pass restricted to SWE rows. I also checked llm_extraction_coverage for SWE rows, because description_core_llm is the only cleaned-text column.

    Coverage that matters¶

    analysis family binding constraint severity
    text description_core_llm coverage is very thin in scraped SWE (16.1% labeled) and only moderate in asaniczka SWE (85.8% labeled) severe for scraped, moderate overall
    seniority seniority_final is fully populated, but seniority_native is diagnostic only and asaniczka has zero native entry labels severe for native-label validation, low for seniority_final use
    geography state_normalized is reasonably strong; metro_area is only moderate moderate for metro work, mild for state-level work
    company company_size and company_size_category are arshkon-only; company_industry is missing in asaniczka severe for pooled company analysis
    requirements yoe_extracted is present in 62%-71% of SWE rows, depending on source moderate

    Source-semantic differences¶

    column what changes across sources
    company_industry present in arshkon and scraped, absent in asaniczka; scraped values are much more compound-like (14.3% contain separators vs 3.4% in arshkon)
    company_size / company_size_category arshkon-only companion fields; absent in asaniczka and scraped
    company_id_kaggle arshkon-only identifier from the companion file
    seniority_native asaniczka has no native entry labels, so it cannot support entry-level validation there
    metro_area / state_normalized same field names, but the scraped source has different location coverage and only 26 distinct metro values because the scrape frame is narrower geographically

    Surprises and unexpected patterns¶

    • metro_area is more usable than I expected: 59.2% non-null in arshkon SWE, 71.1% in asaniczka SWE, and 74.7% in scraped SWE.
    • company_industry is almost complete in scraped LinkedIn, but completely missing in asaniczka. That makes industry analysis asymmetric across the 2024 baselines.
    • description_core_llm is excellent in arshkon SWE and still decent in asaniczka SWE, but scraped SWE text is thin enough that later text work will need explicit coverage thresholds.
    • seniority_final is the stable seniority column, but the diagnostic seniority_native frame is already visibly lopsided across sources.

    Data quality caveats¶

    • The scraped Indeed tail is real and large, but it is outside the primary LinkedIn frame and should stay out of cross-period comparisons unless the analysis is explicitly about platform mix.
    • Many zero-coverage columns in arshkon are source-specific fields from Kaggle companions or scrape metadata. They are not missing by accident; they are absent by design.
    • seniority_final is usable directly. seniority_native is not a pooled cross-source entry baseline because asaniczka has zero native entry labels.
    • description_core_llm is the only cleaned text column. The retired rule-based cleaned-text fields should not be revived downstream.

    Downstream action items¶

    1. Use description_core_llm only on rows with llm_extraction_coverage = 'labeled', and treat scraped text work as coverage-limited.
    2. Use seniority_final for the main analyses, then validate entry-level claims with the YOE proxy and arshkon-only seniority_native.
    3. Treat company-size work as arshkon-only and company-industry work as source-asymmetric.
    4. Prefer state_normalized over metro_area when a geography analysis needs breadth rather than fine location detail.
    5. Keep the scraped Indeed tail out of the core narrative unless a later task shows platform composition is itself a result.
