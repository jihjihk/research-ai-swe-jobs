    T03. Seniority Label Audit (seniority_final)¶

    Run date: 2026-04-20 Owner: Agent B, Wave 1 Input: data/unified.parquet Default filter: source_platform='linkedin' AND is_english=true AND date_flag='ok' Sample: is_swe = True SWE rows under the default filter. Script: exploration/scripts/T03_seniority_label_audit.py Tables: exploration/tables/T03/

    Headline¶

    seniority_final is defensible as the primary seniority column, with three caveats. The LLM half is well-calibrated where it commits (97.9% of rows the LLM labels mid-senior agree with the LinkedIn native label in scraped; 97.6% in arshkon), and where the rule fires the LLM almost never overrides it. The dominant failure mode is abstention, not mislabeling: the LLM returns 'unknown' on 34-53% of the rows it was called on, producing a very large residual unknown pool — especially in arshkon (52.6%) and scraped (47.2%). Kappa against native drops to 0.12-0.18 when unknown is included, but rises to 0.66 (scraped) and 0.45 (arshkon) on the conditional distribution where the LLM did assign a label. Downstream consumers should always report what fraction of their sample falls into unknown, and T30 should prefer YOE-based (J3/S4) operationalizations for any finding that depends on entry-level or senior shares.

    1. seniority_final_source profile¶

    1.1 By source (SWE, default filter)¶

    exploration/tables/T03/source_profile_by_source.csv

    source llm title_keyword title_manager unknown total
    kaggle_arshkon 4,687 1 0 3 4,691
    kaggle_asaniczka 18,124 1 0 4 18,129
    scraped 25,816 8,174 1 11,326 45,317

    Interpretation. Kaggle sources (arshkon, asaniczka) are overwhelmingly LLM-resolved for the SWE subset because every selected row was routed to the LLM and the LLM subsequently overwrites title_keyword values it is equally confident about. Scraped LinkedIn has a very different composition: only 57% are LLM-resolved; 18% retain the Stage 5 title_keyword value (i.e., not_selected from the LLM frame); 25% are genuinely unknown. The distribution asymmetry is a direct consequence of Stage 9/10 frame selection (selected_for_llm_frame) — essentially the whole SWE Kaggle sample is in the frame, but scraped has a large not_selected SWE pool.

    1.3 By aggregator¶

    exploration/tables/T03/source_profile_by_source_aggregator.csv

    Aggregator rows (is_aggregator=True) behave like non-aggregator rows within each source. Arshkon aggregator SWE (n=430) is 100% LLM; scraped aggregator SWE (7,708) is 57% LLM / 12% title_keyword / 30% unknown — very close to the non-aggregator share (56% / 19% / 24%). Aggregator-specific label quality does not appear to be a failure mode of seniority_final_source itself (agreement numbers below show the finer story).

    2. Rule vs LLM internal agreement¶

    Scope: rows where Stage 5 rule fired (seniority_rule ∈ {entry, associate, mid-senior, director}) AND the LLM was called (seniority_final_source = 'llm'). These are exactly the rows where both signals are available, so we can audit how often the LLM overrides a high-confidence rule answer.

    2.1 Kappa by source¶

    exploration/tables/T03/rule_vs_llm_kappa_by_source.csv

    source n kappa_rule_vs_llm observed_agreement expected_agreement
    kaggle_arshkon 1,777 0.805 0.972 0.856
    kaggle_asaniczka 10,670 0.506 0.969 0.938
    scraped 11,367 0.810 0.969 0.836

    High kappa in arshkon and scraped; kappa drops in asaniczka because the rule-fires are almost all mid-senior (imbalance inflates expected agreement). Observed agreement is ~97% everywhere.

    2.2 Per-class accuracy (rule as reference)¶

    exploration/tables/T03/rule_vs_llm_per_class_accuracy.csv

    source seniority_rule n llm_matches_rule share
    kaggle_arshkon director 12 12 100.0%
    kaggle_arshkon entry 99 99 100.0%
    kaggle_arshkon mid-senior 1,666 1,616 97.0%
    kaggle_asaniczka director 78 44 56.4%
    kaggle_asaniczka entry 129 129 100.0%
    kaggle_asaniczka mid-senior 10,463 10,170 97.2%
    scraped director 173 155 89.6%
    scraped entry 662 661 99.8%
    scraped mid-senior 10,532 10,196 96.8%

    Notable discrepancy: in asaniczka, 34/78 rule-labeled director rows get reassigned by the LLM to mid-senior. Inspecting the crosstab, the LLM drops ~44% of asaniczka director titles down to mid-senior. In scraped, the same director downgrade is 7% (12/173). I attribute this to asaniczka descriptions being short/harmonized via the companion job_summary.csv: the description may not carry explicit director-level language even when the title does. In arshkon, every Stage 5 director rule is preserved. This is not a seniority_final defect per se (the LLM has more evidence than the title alone), but it means senior/director shares may be mildly deflated in asaniczka relative to arshkon. T30 picks this up as the asaniczka senior-asymmetry test.

    Entry-rule preservation: 99.8-100% of Stage 5 entry-level rule fires survive the LLM pass. So any downstream junior-side analysis on rows where Stage 5 fired an entry rule is essentially rule-validated.

    3. seniority_final vs seniority_native¶

    Only arshkon and scraped have populated seniority_native. Asaniczka cannot be used for this comparison (zero native entry labels, mostly mid-senior/associate only).

    Native labels include intern and executive (absent from the canonical enum). I compute kappa three ways:

    • (a) Raw: native labels kept as-is. Mismatches on intern/executive are unavoidable.
    • (b) Harmonized: intern → entry, executive → director.
    • © Harmonized, excluding seniority_final = 'unknown': agreement conditional on the LLM (or rule) committing.

    3.1 Kappa¶

    exploration/tables/T03/native_vs_final_kappa.csv

    source n_raw kappa_raw n_harm kappa_harm n_harm_excl_unk kappa_harm_excl_unk observed_harm_excl_unk
    kaggle_arshkon 3,283 0.171 3,283 0.180 1,643 0.453 0.884
    scraped 38,659 0.120 38,659 0.145 18,308 0.658 0.929

    Reading. seniority_final agrees poorly with native when we include the large unknown pool (which by construction cannot match any native label except unknown, which native never uses). When we condition on the LLM committing, agreement is 88-93% observed and kappa in "moderate" (arshkon) to "substantial" (scraped) territory. Scraped kappa is higher than arshkon kappa largely because scraped has a much cleaner mid-senior dominance pattern (native mid-senior → final mid-senior = 15,641/15,975 = 97.9% conditional).

    3.2 Per-class accuracy excluding unknown (harmonized)¶

    exploration/tables/T03/native_vs_final_per_class_accuracy_excl_unknown.csv

    source native (harmonized) n_native_labeled n_match share_match
    kaggle_arshkon associate 79 4 5.1%
    kaggle_arshkon director 22 6 27.3%
    kaggle_arshkon entry (+intern) 140 73 52.1%
    kaggle_arshkon mid-senior 1,402 1,369 97.6%
    scraped associate 465 23 4.9%
    scraped director (+executive) 285 65 22.8%
    scraped entry (+intern) 1,583 1,285 81.2%
    scraped mid-senior 15,975 15,641 97.9%

    Per-class interpretation. (Note: the reference is LinkedIn's own seniority_native label; "match" here means seniority_final equals harmonized native.)

    • Mid-senior is essentially a solved cell — 98% cell accuracy in both sources.
    • Entry has moderate agreement — 81% in scraped, 52% in arshkon. Most of the arshkon entry disagreement flows to mid-senior (65/140) or unknown elsewhere; the scraped dominant disagreement is entry → mid-senior (256 rows). The scraped-vs-arshkon gap suggests the LinkedIn "entry" native label got stricter (or seniority_final got looser on entry) between 2024-04 and 2026-03. This is a material finding for RQ1 and is picked up again in T30's within-2024 calibration.
    • Associate is mostly reclassified as mid-senior by the LLM (5% cell accuracy either way). This is partly because "associate" LinkedIn natively covers a very wide band (≈ mid-level IC) that the canonical 5-level enum splits differently. If any analysis stratifies on associate-only, expect large labeler disagreement.
    • Director has ~23-27% cell accuracy — the LLM+rule output frequently collapses director down to mid-senior. Combined with the asaniczka director-downgrade finding in Section 2, the director slice is the single weakest cell of seniority_final. S2 (director-only) from T30 should be treated as a low-confidence sensitivity.

    3.3 Temporal instability signal¶

    Kappa is 0.66 (scraped, 2026) vs 0.45 (arshkon, 2024) harmonized-ex-unknown. Per-class entry accuracy is 81% vs 52%. This is consistent with two non-exclusive stories: (i) LinkedIn tightened its native entry definition between 2024 and 2026, or (ii) seniority_final got looser on entry (LLM trained to be more permissive on entry labels). Either way, native-label drift is a real risk — per the preamble, any entry-level finding must be cross-checked with the YOE-based proxy (T30 J3).

Arshkon aggregator has lower kappa (0.30 vs 0.48), but the sample is small (n=165) and the dominant driver is associate-labeled aggregator postings. Recommendation: aggregator exclusion should not be mandated as a sensitivity specifically for seniority label quality in scraped; it is worth checking in arshkon-only analyses.

    5. Weak-marker title diagnostic (routing-error estimate)¶

    Sample: 100 LLM-labeled SWE rows where the title contains a roman-numeral level marker (II, III, IV, V) or "level/lvl N".

    exploration/tables/T03/weak_marker_llm_sample.csv

    Summary (weak_marker_llm_summary.csv):

    source final='unknown' final='mid-senior' final=other total
    kaggle_arshkon 143 (83%) 27 2 172
    kaggle_asaniczka 841 (79%) 226 5 1,072
    scraped 1,252 (77%) 330 37 1,619

    In the 100-row qualitative sample, the LLM committed to a label on 19 rows — all mid-senior. Cross-referencing with yoe_min_years_llm:

    • "Software Engineer II" (n=3 committed): YOE 3, 5, and one null → labeled mid-senior. The "II" suffix typically indicates associate-tier (≈ 2-3 YOE) in practice, so labeling as mid-senior is arguably upward-biased on these rows. However, the mid-senior label is reasonable when YOE is ≥ 3.
    • "Software Engineer III" / "SDE III" / etc. (n=8 committed): YOE 5-9 → mid-senior. Correct.
    • "Software Engineer IV" (n=3 committed): YOE 4-12 → mid-senior. Correct.
    • "Senior II" / "Tech Lead - Level 6" / "Staff - Level 4" — all correct mid-senior.

    Qualitative routing-error estimate. When the LLM commits on weak-marker titles (19/100 = 19% of the time), I see no obvious downward errors (never labels a higher level as entry) but a small upward tilt on II-suffix titles (plausibly 2-3 of the 4 "II" commitments should have been associate). Extrapolating to the ~2,860 total weak-marker LLM-labeled rows: ~540 commitments of which perhaps ~100-200 II-titled rows are labeled one tier high. This would mildly inflate the mid-senior share relative to a title-rule baseline, but the overall effect size is small (~0.5-1 pp of the mid-senior share in each source).

    However, the bigger effect is the 77-83% abstention rate. A weak-marker title that doesn't carry through to description-level explicit seniority phrasing gets dropped to unknown — losing an entire population of correctly-leveled engineering roles from seniority-stratified analyses. This is the primary rationale for T30's YOE-based primary (J3/S4): yoe_min_years_llm is populated on the unknown-seniority subset too, and recovers that signal.

    6. LLM unknown share¶

    exploration/tables/T03/llm_unknown_share_by_source.csv

    source total LLM-called n_unknown share_unknown
    kaggle_arshkon 4,687 2,465 52.6%
    kaggle_asaniczka 18,124 6,214 34.3%
    scraped 25,816 12,192 47.2%

    The LLM declines to assign a seniority on 34-53% of rows it was called on. Per the schema design rationale, this is by design — the LLM only commits on explicit signals. But the aggregate effect is large: in the default-filtered SWE sample, seniority_final = 'unknown' is 26% of arshkon, 34% of asaniczka, 52% of scraped (which has the additional not_selected pool on top).

    Implication for all downstream seniority-stratified tasks: the "unknown" slice is not small and is not balanced across sources. Any denominator that uses "all SWE" (rather than "known-seniority SWE") will shift its share estimate simply because the unknown share differs between 2024 and 2026. T30 handles this by reporting both "of all SWE" and "of known-seniority" denominators for every definition.

    7. Defensibility verdict¶

    seniority_final is defensible as the production seniority column, subject to three explicit caveats that downstream consumers must respect:

    1. Abstention, not mislabeling, is the dominant failure. The unknown share is 26-52% across sources and shifts across periods. Never report entry-level or senior shares without documenting which denominator is used and what the corresponding unknown share is.
    2. Director slice is weak. Only 22-27% of native director rows get a director label in seniority_final, and the LLM specifically downgrades 44% of asaniczka rule-labeled directors to mid-senior. Any analysis that isolates directors (e.g., T30 S2) should be treated as a low-confidence sensitivity, not a primary claim.
    3. Associate slice is essentially unusable at present. Native associate → final associate cell accuracy is 5% in both sources; the LLM routes most of this pool to mid-senior. Do not stratify on associate as a standalone cell. The T30 J2 definition (entry ∪ associate) is the only safe way to use associate.

    No remediation is proposed for the core pipeline — the conservative-abstention behavior is working as intended. The remediation for analysis is to (a) report the abstention share alongside every seniority-stratified claim, and (b) prefer YOE-based primaries (T30 J3/S4) for claims that need label-independent validation. T30 encodes this recommendation in the shared panel.

    Essential sensitivities¶

    • Aggregator exclusion (a). Scraped aggregator kappa is within 3 pp of non-aggregator; arshkon aggregator kappa is weaker but the sample is small. Report separately where downstream consumers need it, but no mandatory aggregator-only sensitivity for seniority label quality.
    • Within-2024 calibration. Arshkon vs asaniczka comparison is complicated by asaniczka's lack of entry labels; both sources behave similarly on mid-senior but diverge on director (asaniczka downgrades ~44% of director titles).

    Surprises & unexpected patterns¶

    1. The LLM unknown rate is much higher than I'd expect for a modern LLM classifier — 34-53% on SWE rows it was called on. The LLM design rationale (explicit-signals-only) produces this directly. Downstream tasks need to internalize that seniority_final is ⅔ a crisp labeler and ⅓ an explicit abstainer.
    2. Director accuracy is dramatically worse than mid-senior accuracy (22-27% vs 97-98% conditional cell accuracy). The director failure mode is one-way: the LLM drops director titles to mid-senior when the description lacks director-level management language. This is plausibly correct on the underlying role composition (many "Director" titles in software are really IC/TL roles), but it means S2 (director-only from T30) cannot be trusted as a primary measure.
    3. Weak-marker titles (II/III/IV) almost never get labeled by the LLM (77-83% abstention). The implicit policy is "if the description doesn't restate the level, don't infer from the title number." This preserves YOE-based analyses but collapses a large, clean class-informative signal into unknown.
    4. Asaniczka director-downgrade is larger than arshkon's (44% vs 0% LLM downgrade of rule-labeled directors). This is a direct compositional difference between the two 2024 sources that will surface in T30's asaniczka senior-asymmetry test.
    5. Scraped-vs-arshkon entry accuracy gap (81% vs 52% conditional on native=entry). Either LinkedIn tightened its entry label between 2024-04 and 2026-03, or the LLM got looser on entry. Either way, the native label cannot be a stable cross-period reference — yet another reason to prefer YOE-based primaries in T30.

    Data caveats¶

    • Asaniczka cannot be audited via seniority_native (no native entry labels, only mid-senior/associate). All native-vs-final analyses in this report are arshkon-only or scraped-only.
    • Indeed has no seniority_native (0% populated), so seniority_final cannot be validated on the scraped Indeed subset. Indeed is out of scope for the primary LinkedIn analysis anyway.
    • seniority_rule is written by Stage 5 and is preserved through Stage 10. Rows where seniority_rule = 'unknown' AND seniority_final_source = 'llm' are the normal majority case (the LLM adds value where the rule is silent). The schema-guaranteed impossible case (rule non-unknown AND final_source='llm' that contradicts rule strongly) is rare in practice (e.g., 13 rows in scraped where rule=mid-senior but final=entry).

    Downstream action items¶

    1. T30 must prefer J3 (YOE ≤ 2) and S4 (YOE ≥ 5) as primaries for entry and senior claims. Label-based definitions (J1, S1, S2) should be sensitivities.
    2. T30 must run the asaniczka senior-asymmetry test (S4 and S1 shares in asaniczka vs arshkon 2024). Section 2.2 provides the prior expectation: asaniczka directors get downgraded; check whether this propagates to S4/S1 senior shares.
    3. T30 must report known-seniority denominators alongside all-SWE denominators for every label-based definition, because the unknown share differs across sources (26% arshkon, 34% asaniczka, 52% scraped).
    4. Any Wave 2+ task that isolates directors (T21, T34) must note this audit — director precision is too low to support director-only primary claims. Use mid-senior+director as the safe senior aggregate.
    5. Entry-level findings (T20, Wave 3 RQ1 tasks) must be validated against yoe_min_years_llm ≤ 2 even after consulting seniority_final. The 2024→2026 entry native-label drift is a real risk.
