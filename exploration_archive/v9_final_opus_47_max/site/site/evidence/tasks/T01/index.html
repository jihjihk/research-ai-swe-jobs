    T01. Data profile & column coverage¶

    Agent: A, Wave 1 Date: 2026-04-20 Input: data/unified.parquet (96 columns, 1,452,875 rows) Outputs: exploration/tables/T01/*.csv, exploration/figures/T01/*.png, scripts under exploration/scripts/T01_*.py

    Headline finding¶

    The unified file contains 1,452,875 rows across three sources. Within the default SWE analysis frame (LinkedIn + English + date_flag='ok' + is_swe=True), we have 68,137 SWE postings: arshkon 4,691 (2024-04) + asaniczka 18,129 (2024-01) + scraped 45,317 (2026-03 & 2026-04). The scraped source keeps growing (2026-04 already has more SWE than 2026-03 and the window ends 2026-04-17).

    The dominant binding data constraint for Wave 2+ is not row count — it is (a) the gap in description_core_llm / LLM classification coverage on scraped rows outside the selected core frame (only ~57% of scraped SWE rows have labeled LLM text), and (b) source-specific field gaps that force asymmetric sourcing: company_industry and company_size are zero for asaniczka, skills_raw is zero for scraped, date_posted is only 1% populated for scraped LinkedIn (fall back to scrape_date), and is_remote source flag is 0% for arshkon (known data artifact).

    Most consequential finding for downstream work: seniority_final for arshkon SWE rows is almost entirely derived from the LLM (4,687 of 4,691 = 99.9%), and the LLM is substantially stricter than seniority_native. Only 53 of arshkon's 738 seniority_native='entry' SWE rows end up as seniority_final='entry'; 618 go to 'unknown'. This is correct behavior per the documented design (LLM uses explicit seniority signals only), but it means any entry-level claim based on seniority_final alone will have vastly different effective counts than one based on seniority_native. The T30 panel and T02 decision rule must carry this.

    Methodology¶

    Computed via DuckDB aggregations over data/unified.parquet, never loading the full parquet into pandas. Empty strings are treated as null for VARCHAR columns when computing the non-null rate. Three subsets were profiled: all, swe (is_swe=True), and swe_linkedin_default (is_swe AND source_platform='linkedin' AND is_english AND date_flag='ok'). Aggregator sensitivity tables repeat the key columns split on is_aggregator.

    Script: exploration/scripts/T01_data_profile_coverage.py.

    5. seniority_final_source composition¶

    This is the most consequential compositional finding.

    SWE LinkedIn default — where seniority_final came from¶

    Source llm title_keyword title_manager unknown
    kaggle_arshkon 4,687 1 0 3
    kaggle_asaniczka 18,124 1 0 4
    scraped 25,816 8,174 1 11,326
    • Kaggle seniority is essentially LLM-only (99.97%+) — because Kaggle SWE rows almost all route to the LLM, and the LLM overrides Stage 5.
    • Scraped seniority is a 3-way mix: LLM (57%), rule-based title_keyword (18%), unknown (25%). The 25% unknown share on scraped means one in four scraped SWE rows has no seniority label at all — they are not_selected rows whose titles lacked strong keywords.

    Consequence for cross-source entry-level comparisons¶

    Arshkon SWE entry-level rates under different operationalizations:

    Operationalization Entry count Entry share
    seniority_native = 'entry' (arshkon SWE LinkedIn default) 738 / 4,691 15.7%
    seniority_final = 'entry' 177 / 4,691 3.8%
    seniority_final = 'entry' OR 'associate' 192 / 4,691 4.1%

    The LLM collapses 618 of arshkon's 738 seniority_native='entry' SWE rows to 'unknown'. This is not a bug — the LLM looks for explicit seniority signals in the description text while LinkedIn's own "Entry level" platform label often reflects YOE thresholds that the LLM is told to ignore. But it means the choice between seniority_native and seniority_final will determine whether 2024 "entry" on arshkon is 3.8% or 15.7% of SWE. T30 will enumerate this; the fact that the two columns disagree by a factor of 4× is itself a critical research finding.

    7. Aggregator sensitivity (exploration/tables/T01/aggregator_sensitivity.csv)¶

    Key cross-stratifications from the SWE LinkedIn default subset:

    • Scraped aggregator rows: 7,708 total. description_core_llm labeled on 4,383 (56.9%). yoe_min_years_llm labeled on 3,037 (39.4%). LLM text coverage is similar between aggregator and non-aggregator.
    • Arshkon aggregator rows: 430 total, all fully LLM-covered. seniority_final='llm' source for 100% — strong confidence but very small N.
    • Asaniczka aggregator rows: 2,943 total, company_industry still 0% (asaniczka-wide hole), otherwise strong LLM coverage.

    Aggregator composition is not an LLM-coverage differential, so the recommendation is to keep aggregators in the main analyses and report is_aggregator=False-only sensitivities. Beware source-level aggregator share differences when pooling (9% arshkon vs 17–18% asaniczka/scraped).

    8. Surprises & unexpected patterns¶

    1. seniority_final = 'entry' count on arshkon SWE is only 177 (3.8% of arshkon SWE), vs 738 (15.7%) under seniority_native = 'entry'. The LLM actively removes LinkedIn's own entry labels that are not backed by explicit description-text seniority cues. Whether this is "real" or "noise depends on your operationalization" is itself the key question for the project. The T30 panel is load-bearing.

    2. Asaniczka seniority_native has only mid-senior and associate — not even nulls. This is stricter than the documented "no entry labels" — there's literally no intern, director, executive, or unlabeled row in asaniczka's source data. Either asaniczka's scraper filtered these out upfront or LinkedIn's 2024 feed returned only these two. Implication for T02: since asaniczka has no null seniority either, the associate label is not "the entry-level bucket dumped into associate" but rather a substantive LinkedIn label.

    3. Scraped SWE has 19,500 rows not_selected for LLM — about 43% of scraped SWE cannot use description_core_llm or any LLM-derived column. This is ~40% of the growing-period data, significant for Wave 2+ text analyses. Expanding the LLM budget or admitting supplemental cache is the only way to raise this.

    4. search_metro_name populated for asaniczka (95%) is a mild surprise — asaniczka is documented as "historical snapshot" but the pipeline reconstructed approximate metros. Note asaniczka's top search metros are dominated by ambiguous small cities (Alexandria, Santa Clara) whose actual metros overlap DC, Bay Area — treat with care; the normalized metro_area field (LinkedIn 26-metro frame) is the correct one for geographic analyses.

    5. Scraped Indeed has date_posted from 2016-10-17 to 2026-04-18. This is a 9.5-year span. Indeed is sensitivity-only per project rules but still: the raw date_posted on scraped Indeed is not "time of posting from an active 2026 feed" — these are older postings that the aggregator surfaces with their original date. Treat all temporal claims on scraped Indeed with extreme caution; the scrape_date is the only anchor of current presence.

    6. work_type has different semantic fill pattern than company_industry — work_type is 100%/100% on arshkon/scraped but 0% on asaniczka, while company_industry is 99%/99.96% arshkon/scraped but 0% on asaniczka. Two different asaniczka gaps, same pattern. Asaniczka is systematically sparse in employer-metadata side-tables despite 892K rows — an architectural data shape, not random missingness.

    7. LLM-rule SWE classification disagreement is substantial. On rows where both are populated, 1,472 scraped rows with is_swe=True are NOT_SWE per LLM; 2,604 with is_swe=False are SWE per LLM. ~4K disagreements out of ~52K LLM-labeled LinkedIn rows (7.8% gross disagreement). This is relevant for T04 (SWE classification audit) — the is_swe rule and the LLM classification are not interchangeable, and the LLM tends to be more restrictive on some tiers. Worth including an is_swe AND swe_classification_llm='SWE' intersected sensitivity in key RQ1 findings.

    9. Data quality caveats (for downstream agents)¶

    • Do not combine seniority_native and seniority_final results within a single claim. They are different operationalizations with different effective samples.
    • For text-sensitive analyses, restrict to llm_extraction_coverage='labeled' and carry this filter through the report. Denominators differ between text and non-text analyses.
    • selected_for_llm_frame=True marks the sticky balanced core, which is the only subset where cross-source balance was enforced. Supplemental cache rows exist but should not be pooled with core rows for balanced claims.
    • Scraped LinkedIn temporal unit is scrape_date / scrape_week, not date_posted. Do not try to use date_posted for scraped-LinkedIn cohort building.
    • Source-level aggregator share varies from 9% to 18%. Include is_aggregator=False-only sensitivities for any employer/company-composition claim.
    • Effective entry-level sample sizes are small: 177 arshkon / 238 asaniczka / 1,650 scraped-2026-03 / 803 + 847 scraped-2026-04 (wait: scraped 2026-04 has 803 — let me re-check — actually scraped has 847 in 2026-03 and 803 in 2026-04, both under seniority_final='entry'). With only ~1,650 entry-level rows across the two scraped months and 177 + 238 = 415 across Kaggle, power for fine-grained entry-level cuts (by metro, by company, by archetype) will be tight.

    10. Action items for downstream tasks¶

    1. T02 (this agent, next): Apply the associate-as-junior test. Given finding 1 above, the T02 test needs to compare asaniczka associate against arshkon seniority_final='entry' (not just seniority_native='entry') — because that's what downstream Wave 2 work will use.

    2. T03 / T04 (Agent B): The 4× discrepancy between seniority_native and seniority_final on arshkon entry rows is the central audit finding — T03 should quantify where the LLM re-routes entry labels and whether that re-routing is stable across title types. T04 should report is_swe/LLM-SWE disagreement by tier.

    3. T06 / T16 (Agents C, J): Employer/company analyses cannot use company_industry for asaniczka; must rely on title-derived proxies or limit to arshkon-vs-scraped. company_size is arshkon-only — no 2024→2026 size comparison exists.

    4. T11 / T14 / T35 (Agents G, I, S): Skills analyses must use text-extracted tech matrices; skills_raw is not a cross-source feature. Asaniczka's structured skills should be a separate sensitivity, not a primary.

    5. T17 (Agent J): Remote work trend cannot be computed — annotate as RQ-ineligible.

    6. T19 (Agent K): Cannot do sub-week temporal slicing on asaniczka (6-day window) or arshkon (16-day). Cross-period rate-of-change should use source-period buckets, not fine time bins.

    7. T30 (Agent B): The T30 panel is critical. T01's finding that arshkon SWE entry drops from 15.7% → 3.8% under the switch from seniority_native to seniority_final means every entry-level claim must be reported in the 4-row T30 panel. The primary should be J3 (yoe_min_years_llm ≤ 2) because it's genuinely label-independent and robust to the seniority_native/seniority_final disagreement.

    11. Essential sensitivities¶

    • Aggregator-excluded re-run of key Wave 2 claims, given source-level aggregator share varies 2× between arshkon and asaniczka/scraped.
    • LLM-labeled-only subset for any density/frequency metric using text.
    • Core-frame only (selected_for_llm_frame=True) for balanced cross-source comparison; supplemental cache is a robustness check only.
    • Arshkon-only baseline for any seniority_native-dependent analysis (asaniczka has no entry labels).
