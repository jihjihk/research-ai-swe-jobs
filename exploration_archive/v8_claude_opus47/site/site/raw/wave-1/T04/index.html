    T04 — SWE classification audit¶

    Agent: B (Wave 1) · Date: 2026-04-17 · Input: data/unified.parquet, LinkedIn, English, date_flag='ok'.

    Headline¶

    The is_swe flag is defensible as the primary SWE classifier with an estimated false-positive rate on borderline cells of ~20-25% (concentrated in QA/Test/Systems/Reliability engineer titles) and a false-negative rate on strong-SWE-keyword titles of under 1% (fewer than 50 rows across 990K). The dual-flag invariant holds: zero violations of (is_swe + is_swe_adjacent + is_control) > 1 across all sources. Boundary classification (ML/Data/DevOps/AI Engineer) is highly consistent across periods — the rule puts 98-100% of these titles in SWE regardless of source. The main issue is not misclassification but how the rule defines the SWE boundary: QA/Test engineering is currently classified as SWE by the rule, but the LLM disagrees on 30-40% of those rows. Any analysis that wants a "pure coder" population should filter on swe_classification_llm = 'SWE' (where labeled) rather than rely on is_swe alone.

    Methodology¶

    Base filter: source_platform = 'linkedin' AND is_english = true AND date_flag = 'ok'. All SWE subset adds AND is_swe = true. SWE rows after filter: arshkon 4,691 · asaniczka 18,129 · scraped 40,881.

    Step 2. Borderline SWE sample (50 rows)¶

    Sample conditions: SWE and either swe_confidence ∈ [0.3, 0.7] OR tier = title_lookup_llm. All 50 sampled rows have swe_confidence = 0.9 and tier = title_lookup_llm — meaning the swe_confidence range [0.3, 0.7] is empty, so the query selected from the title_lookup_llm pool exclusively.

    Content assessment: The sample is dominated by QA/Test engineering titles ("Test Engineer", "Automation Engineer", "Quality Assurance Engineer", "Software Test Engineer"), SRE ("Reliability Engineer"), security engineering ("DevSecOps Principal Engineer"), and AI engineering ("Artificial Intelligence Engineer", "MLOps Engineer", "Database Engineer"). All are reasonable SWE classifications but live near the SWE/SWE-adjacent boundary.

    LLM cross-check disagreement within the sample: Of the 50 borderline rows, the LLM was called on 7 — of those, 3 returned SWE, 2 returned SWE_ADJACENT, 2 returned NOT_SWE. This is a 4/7 = 57% disagreement rate on this hardest slice, but the sample is small. See the full-corpus LLM disagreement numbers below for the robust picture.

    Likely false-positive rate for borderline SWE ≈ 20-30%. Titles like "Automation Engineer" and "Quality Assurance Engineer" are SWE-adjacent by most definitions — they involve coding but their primary function is QA/testing infrastructure, not product software development.

    Sample: tables/T04/step2_borderline_swe_sample.csv.

    Step 3. Borderline non-SWE sample (50 rows, titles w/ engineer/developer/software but is_swe=false)¶

    Content assessment: Overwhelmingly non-SWE engineering roles — Process, Civil, Electrical, Chemical, Mechanical, Manufacturing, Industrial, Network, Field, Design Engineer. Most sit in tier = 'unresolved' (meaning no SWE classifier fired — correct). A minority with "Developer" in the title (Appian, Power BI, Pega, SharePoint, SQL, Hadoop, etc.) are borderline — these are configuration/development roles in vendor platforms.

    Corpus-wide FN estimate. Rows with strong SWE-indicator title patterns (software engineer, backend/frontend/full-stack engineer/developer, web/mobile/app developer, or language-specific developer like python developer) that are flagged is_swe = false:

    Source Strong-FN candidates SWE true positives Estimated FN rate
    arshkon 6 4,691 0.13%
    asaniczka 16 18,129 0.09%
    scraped 25 40,881 0.06%

    Manual inspection of the 47 candidates shows that ~15-20 are genuinely mobile/web app developer roles that should have been SWE (examples: "Android App Developer", "Shopify App Developer", "Senior Mobile App Developer", "Web App Developer"). The rest are compound titles where the rule punted (e.g., "Senior Software Engineer - Solution Architect" — two function labels create ambiguity).

    Estimated corpus FN rate: ≤ 0.15%. Negligible for aggregate statistics but worth documenting for any analysis that specifically targets mobile app development subset.

    Sample: tables/T04/step3_borderline_non_swe_sample.csv.

    Step 4. is_swe_adjacent and is_control profiles¶

    Top adjacent titles¶

    Adjacent rows cluster on: Data Scientist, Data Analyst, Quality Engineer, Network Engineer, Architect (Solutions/Enterprise/Data), Security Engineer, Systems Administrator. These are reasonable SWE-adjacent definitions — the roles involve technical work and sometimes code but are not primarily software development.

    Adjacent tiers split: regex (~45%), embedding_adjacent (~30%), title_lookup_llm (~25%). The embedding_adjacent tier is the largest indicator of rule vs embedding vs LLM composition — it fires where titles are semantically near-SWE but not regex-matchable. Roughly similar distribution across sources.

    Top control titles¶

    Control is dominated by: Registered Nurse (+variants), Accountant (Senior/Staff/Project), Financial Analyst, Electrical Engineer, Mechanical Engineer, Civil Engineer. These are the study's non-tech occupation group for RQ comparisons.

    Adjacency vs SWE distinction looks clean in spot check. The adjacent sample has titles like "Data Scientist", "Data Analyst", "Network Engineer", "Quality Engineer" — which the research design explicitly treats as SWE-adjacent rather than SWE. Adjacent is distinct from SWE and distinct from Control.

    Tables: tables/T04/step4_*.csv.

    Step 5. Estimated FP / FN rates¶

    False-positive estimates¶

    Two approaches:

    1. LLM cross-check disagreement — a direct quality signal. For rows where the LLM was called (llm_classification_coverage = 'labeled'), what fraction disagree with is_swe = true?
    Source LLM-called is_swe=true rows LLM says NOT_SWE LLM says SWE_ADJACENT Disagreement rate
    arshkon 2,917 365 269 21.7%
    asaniczka 5,190 1,074 548 31.3%
    scraped 6,918 497 609 16.0%

    The LLM-called subset is non-random — it over-represents title_lookup_llm and embedding_high tiers (the harder cases). If we assume regex-tier rows are ~99% correct SWE by construction (they matched explicit SWE-title patterns), the estimated corpus-wide FP rate under the LLM's definition is:

    arshkon FP ≈ (2,917 × 21.7% / 4,691) × (share of arshkon that's non-regex tier) ≈ 3-5%. scraped FP ≈ 16% × (40,881 − 38,131) / 40,881 ≈ 1.1% corpus-wide (rough).

    1. Intuitive classification of borderline titles (QA, Test, Systems, Reliability — all tipped "adjacent" by LLM). These represent 15-20% of the borderline subset but only 2-5% of total SWE rows.

    Combined FP estimate: 3-5% corpus-wide, concentrated in QA/Test/Systems-engineering cells. Analyses that need a strict "software developer" population should filter on swe_classification_llm = 'SWE' (where labeled) or exclude titles matching quality|test|assurance|systems|reliability|application engineer.

    False-negative estimates¶

    From Step 3: strong-SWE-title FN rate is ≤ 0.15% corpus-wide. Most "should have been SWE" mis-classifications are mobile/web "App Developer" titles. Not a meaningful concern for aggregate analysis.

    Step 6. Dual-flag invariant¶

    (is_swe + is_swe_adjacent + is_control) > 1 count: zero violations across all three sources (arshkon 110,360 rows, asaniczka 847,526, scraped 253,626). Mutual exclusivity holds.

    Data quality caveats¶

    1. swe_confidence is effectively tier-bound (regex=1.0, title_lookup_llm=0.9, embedding=0.88-0.90). The range [0.3, 0.7] is empty in the SWE population — the borderline sample definition given in the task was substantively a title_lookup_llm-only sample. This is worth noting but not a defect.
    2. FP estimates use LLM as a reference. The LLM is itself imperfect; its definition of SWE may be tighter (excludes QA/Test) than the rule's. The 16-31% disagreement represents a definitional difference more than a labeling error. Pick the definition intentional for the analysis question.
    3. The rule labels QA/Test engineering roles as SWE. This is a defensible choice (they write automation code) but it inflates the SWE share for aggregate counts. If the analysis wants "product software developers only," add an exclusion rule for titles matching qa|test|assurance|automation (?=engineer).
    4. Adjacent-tier allocation uses embeddings, which are black-box. Spot-check precision appeared high, but downstream analysis may want to re-validate if adjacent rows are used as a distinct group.

    Surprises & unexpected patterns¶

    1. AI Engineer is a 45× growth story from 2024 to 2026. Arshkon (April 2024) has 27 matches, scraped 2026-03/04 has 1,227. Corpus size explains ~9×; the remaining ~5× is genuine title proliferation. This is a directly RQ-relevant finding and was not specifically part of T04's mandate — worth flagging to Wave 2 skill-trend agents as a high-signal target.
    2. LLM-vs-rule disagreement is 16% in scraped but 31% in asaniczka, almost 2×. Mechanism: asaniczka's harder/weirder titles are more often routed to title_lookup_llm tier and the LLM is more willing to disagree with the title-lookup-based classification. Alternative mechanism: asaniczka has a noisier title column (more job-board aggregator spam), and the LLM correctly identifies many as non-SWE while the rule-lookup erred. Either way, asaniczka SWE rows should carry a larger uncertainty band than scraped SWE rows.
    3. The title_lookup_llm tier is the FP-concentration zone. If a single action were to be taken to tighten SWE classification, it would be to re-evaluate the title_lookup_llm tier's title allowlist — specifically QA/Test engineering titles. However this must be done carefully because QA/Test titles are a non-trivial share (~2-5% of the current SWE corpus) and changing their classification will shift time-series.
    4. SWE-to-control ratio varies 6× across sources (arshkon 0.54, asaniczka 0.17, scraped 0.98). This is purely compositional — the scraper targets SWE-heavy searches, while asaniczka is a broad job-board scrape. Any RQ comparing SWE to control must normalize for this compositional difference.
    5. Data Scientist is unambiguously SWE-adjacent in all three sources (>97% adjacent-rate). For content analyses using the adjacent cohort, Data Scientist is the single largest contributing title. Analyses that want to exclude Data Scientist can do so cleanly via swe_classification_tier != 'title_lookup_llm' OR swe_confidence >= 0.95 on the adjacent subset (approximate exclusion).
    6. Zero dual-flag violations across 1.2M rows is itself notable — indicates the preprocessing invariant is rigorously enforced.

    Action items for downstream tasks¶

    • Content analyses targeting "pure software developers" should filter is_swe = true AND (swe_classification_llm = 'SWE' OR llm_classification_coverage != 'labeled') — i.e. include rule-confident rows by default but defer to LLM for routed rows. This drops QA/Test/Systems engineer rows that the LLM flags adjacent.
    • Cross-period rate claims on SWE share (RQ1 volume) should be caveat-marked with the 3-5% FP tail. The FP is temporally stable, so first-differences are not biased by it, but absolute shares carry the tail.
    • AI Engineer title growth (27 → 1,227) should be a featured headline in RQ4 skill-requirement analyses. Wave 2 agents doing skill-density should explicitly report AI-Engineer-title prevalence as a separate line item.
    • DevOps Engineer's 2% adjacent tail is small enough to ignore; do not add sensitivity tests for it.
    • Do not re-tune swe_classification_tier without a dedicated labeling effort. The LLM disagreement rate is a definitional divergence, not a labeling defect. Changing the rule mid-study would introduce a time-series artifact.
