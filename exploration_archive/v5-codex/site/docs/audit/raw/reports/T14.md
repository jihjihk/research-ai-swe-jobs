# T14 Technology Ecosystem Mapping

## Headline finding
The technology surface is not just getting broader. It is reorganizing into a stable cloud/frontend backbone plus a much more coherent AI/LLM cluster in the scraped window. The old 2024 data cluster fragments, while `llm`, `generative_ai`, `agent`, `claude`, and related terms gain dense co-occurrence ties that were barely present in the historical snapshots.

## What we learned
The broad stack backbone is surprisingly persistent. `aws`, `ci_cd`, `kubernetes`, `javascript`, `typescript`, `sql`, and the frontend/backend families remain central across all three sources. What changes is the texture around that backbone: the scraped source adds a visibly stronger AI tool / LLM community and pulls in adjacent app and ML terms such as `fastapi`, `pandas`, `scikit_learn`, `mlflow`, `kubeflow`, and `nextjs`.

AI-tool mention growth is real, but it is not just a length artifact. In the primary spec, AI-tool / LLM mentions rise from 2.05% of arshkon SWE rows to 17.19% of scraped SWE rows. Excluding aggregators barely moves that share (2.14% to 17.61%). The postings that mention AI tools are also denser in technology mentions, so the signal is both broader and more concentrated, not just longer.

The co-occurrence network supports the same story. With company capping at 30 postings per company per source-period, the largest stable communities are cloud/devops, frontend/web, testing/methodology, and an AI/ML cluster. Under no-cap sensitivity, the same communities remain, which means the result is not an employer-concentration artifact.

The structured skills file is useful, but it is not the same measurement surface as the extracted tech matrix. After aliasing, 167 tech columns overlap and rank correlation is only 0.69. The structured field captures more generic and process-oriented skill language; the regex-extracted matrix is more sensitive to explicit technology tokens. They are complementary, not interchangeable.

## What surprised us
The biggest surprise is how little the seniority signal survives in the structured skill baseline. On asaniczka, the label-based `entry` vs `mid-senior` comparison leaves only one FDR-significant skill (`pandas`), while the YOE-based junior proxy yields a different set with zero overlap. That is a measurement warning, not just a sparsity issue.

Another surprise is that the AI co-occurrence neighborhood is broader than a pure model-stack story. The top associations include `generative_ai`, `nlp`, `pytorch`, but also `fastapi`, `nextjs`, `typescript`, and `opentelemetry`. The AI tool layer is being wrapped into application delivery, not only into traditional ML work.

## Evidence assessment
The tech-shift result is strong. The within-2024 calibration table shows that several 2026 gains are far above the arshkon-vs-asaniczka baseline noise: `llm`, `generative_ai`, `agent`, `claude`, `typescript`, `gcp`, and `kubernetes` all have signal-to-noise ratios comfortably above 2. One important counterexample is `c_plus_plus`: its 2026 decline is smaller than the 2024 source-to-source difference, so it should not be treated as a robust temporal result.

The network result is also strong because it survives the required sensitivity checks. The company-capped and uncapped networks both preserve the same high-level cluster structure. The only thing that changes materially is the size of the AI and cloud communities, which is itself part of the substantive story.

The structured-vs-extracted validation is moderate evidence. The correlation is decent, but the disagreement is informative rather than negligible. It tells us the structured field is richer on generic skill language, while the regex matrix is better at explicit stack terms. For the paper, that means the structured skills field is a baseline, not a replacement.

The seniority-by-skill result is weak if you insist on `seniority_final = entry` alone. It is too sparse. The YOE proxy is the more stable view and should be treated as the primary junior comparator for skill-level claims. Material disagreement between the two is itself the result.

## Sensitivity checks
Aggregator exclusion: the AI-tool share moves only slightly from 2.05% to 2.14% in arshkon and from 17.19% to 17.61% in scraped. The direction does not change. See [T14_sensitivity_no_aggregators.csv](../../../assets/tables/T14/T14_sensitivity_no_aggregators.csv).

Company capping: the co-occurrence network keeps the same cloud/frontend/testing/agile backbone with and without capping. Without the cap, the cloud community grows a bit more and the AI community becomes larger, but the cluster identities do not change. See [T14_community_overlap_arshkon_vs_scraped.csv](../../../assets/tables/T14/T14_community_overlap_arshkon_vs_scraped.csv).

Within-2024 calibration: the AI and LLM shifts are well above the arshkon-vs-asaniczka baseline. `c_plus_plus` and a few generic language changes are not. The right reading is not “everything changed,” but “the new AI stack changed in a way the historical source gap does not explain.”

## Emerging narrative
The strongest narrative is not generic scope inflation. It is ecosystem reconfiguration: cloud/devops and frontend remain the core, but the scraped window has a more explicit AI/LLM layer that is tied into application and ML tooling. That layer is visible in the network, in the top change table, and in the AI-tool co-occurrence associations.

The structured skills baseline adds a second point: the seniority story is not cleanly legible in skill inventories alone. The label-based junior bucket is too narrow to carry the analysis. For downstream work, the YOE proxy should remain the junior validator whenever skill differences are involved.

## Data quality caveats
The technology matrix is binary and description-driven, so mention rate does not equal required depth. That matters especially for AI terms, where aspirational mentions can rise faster than actual workflow change.

The structured skills file contains a much broader and noisier vocabulary than the regex tech matrix. That breadth is useful, but it is also why the overlap correlation is only moderate.

The seniority skill analysis is especially fragile under `seniority_final = entry`. The table is still worth keeping, but only as a high-precision slice, not as the main junior surface.

## Direction for downstream tasks
Use the tech network result as the paper’s main technology-side structural evidence. The cloud/frontend backbone plus the new AI community is strong enough to matter.

Treat the structured skills file as a validation surface, not a ground truth. It is most useful for checking whether extracted tech trends are missing broader process language or vice versa.

Do not let the sparse explicit-entry bucket define junior skill conclusions. Any junior skill or convergence analysis should keep the YOE proxy in view.
