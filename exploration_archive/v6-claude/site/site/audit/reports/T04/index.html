    T04 — SWE Classification Audit¶

    Agent B (Wave 1). Input: data/unified.parquet. Default filter: is_english = TRUE AND date_flag = 'ok' AND source_platform = 'linkedin'.

    Headline finding¶

    SWE classification is reliable enough to use as primary. The regex tier (88-93% of is_swe=TRUE rows) is essentially noise-free, the dual-flag invariant holds perfectly, and the boundary categories (ML/Data/DevOps/Security) partition cleanly with only modest period drift. The main risk is the title_lookup_llm tier (5-11% of SWE rows): manual labeling of 50 borderline rows from this tier estimates 24% hard false-positive and 28% borderline-ambiguous, weighted to an aggregate FP contribution of roughly 1.3-2.7% of the full SWE sample. False negatives in the "contains engineer/developer/software" non-SWE pool are rare (~6% hard FN in 50 samples from a much larger pool), with most missed cases being DevSecOps / low-code platform developers / web-performance engineers.

    Recommendation: Use is_swe as-is for primary analyses. For any headline finding whose magnitude could plausibly be driven by a 2-3% classification FP, rerun with tier title_lookup_llm excluded and report the delta (sensitivity dimension g in the preamble).

    3. Borderline SWE sample (50 rows, tier=title_lookup_llm)¶

    Sampled 50 rows from the title_lookup_llm tier (since there is no confidence-in-[0.3,0.7] subpopulation). Manually labeled each as SWE / BORDERLINE / NOT_SWE based on title + first 250 chars of description. Full labels: exploration/tables/T04/04_borderline_swe_sample_labeled.csv.

    Label Count
    SWE (true positive) 24
    BORDERLINE (QA / test / firmware / cybersecurity) 14
    NOT_SWE (false positive) 12

    Clear false positive examples in the sample: - Computer Engineer IV — Combat vehicle systems engineer, hardware-adjacent (not SWE) - Lead Test Engineer — Aerospace systems engineer at Leidos (not SWE) - Quality Assurance Engineer — Magnera (materials manufacturer) QA (not SWE) - Automation Engineer — Food manufacturing industrial automation (not SWE) - Sr Automation Engineer — Electronic circuits / manufacturing (not SWE) - Senior Systems Integration Engineer — Medtronic medical device systems (not SWE) - Reliability Engineer — Bulk manufacturing (not SWE) - Principal Systems Test Engineer — DoD hardware test

    These share a pattern: "Engineer" titles in industrial/aerospace/medical-device companies where the role is not writing software. The title_lookup_llm path catches the generic "Engineer" token without sufficient description context.

    Borderline examples (ambiguous): - QA Automation / Test Engineer / Software Quality Engineer — debatable whether these are SWE or SWE_adjacent - DevSecOps Engineer II, Senior DevSecOps Engineer — I labeled these SWE (they write automation code) but reasonable people could call them adjacent - FPGA Firmware Engineer, Senior Firmware Engineer — embedded systems dev, borderline - Junior QA Engineer — pure QA, borderline

    FP contribution estimate: - Hard FP rate within title_lookup_llm tier: 24% (12/50) - Borderline rate: 28% (14/50), which if half should really be adjacent ≈ 14% additional leakage - Title_lookup_llm share of total SWE: 5-11% across sources - Aggregate hard-FP contribution to SWE sample: 24% × (5-11%) = 1.3-2.7% - Aggregate soft-FP contribution: add ~1-2% for borderline leakage → ~2.3-4.7% total

    This is within the 5% tolerance typical for production NLP classifiers. It is NOT large enough to invalidate cross-period comparisons, but it IS large enough that any headline finding whose effect size is smaller than 5% should be rerun with swe_classification_tier = 'title_lookup_llm' excluded (sensitivity dimension g).

    4. Borderline non-SWE sample (50 rows, title contains engineer/developer/software, is_swe=FALSE)¶

    Sampled 50 rows. Full labels: exploration/tables/T04/05_borderline_nonswe_sample_labeled.csv.

    Label Count
    Legitimately NOT_SWE 35
    BORDERLINE (ambiguous adjacent/SWE) 12
    FALSE_NEGATIVE (should be SWE) 3

    Clear false negatives identified: - FLEX Principal Engineer - Web Performance (Marriott) — This is clearly a front-end web performance SWE role. Classified as none (not SWE, not adjacent, not control). (swe_confidence 0.53 from the LLM backup path.) - Senior Pega Developer — Pega is a low-code platform but Pega developer is a real developer role. Classified as none. - Senior Cloud DevSecOps Engineer — Federal cloud DevSecOps (Applied Insight). Classified as none despite swe_confidence of 0.64.

    Borderline cases worth noting: - Workday Financials Developer/Analyst, Sr Workday Payroll Engineer — Workday is platform config work; debatable whether it counts as SWE - Dashboard BI Developer — BI/dashboarding is borderline - Sr. Software Cybersecurity Engineer — Abbott medical cybersecurity, could be SWE or SWE_adjacent (correctly tagged adjacent) - Senior Commissioning Engineer - Robotics — Robotics with smart control code, borderline - Quality Engineer 1 — "Software Quality Engineer 1" with automated testing, borderline adjacent vs SWE

    FN rate estimate: - Hard FN rate in this pool: 6% (3/50) - Borderline rate: 24% (12/50) - This sample is biased toward finding FNs because we filtered the non-SWE pool to rows whose title contains "engineer/developer/software" — the base rate of SWE in the remaining ~950K non-SWE postings (that do NOT contain these words) is much lower. - The title-filter subset contains roughly 150K-200K rows, so ~6% FN × 150K ≈ 9K missed SWE rows across all sources. Spread over 4 period-buckets, that's ~2K per bucket — non-trivial but small relative to the 19K-21K SWE per scraped bucket.

    The FN pattern has an interpretable structure: the missed cases are overwhelmingly in newer specialized roles (DevSecOps, Pega platform dev, web-performance engineering) or platform-config work (Workday, Salesforce, BI). The rule-based regex was designed for 2024 vocabulary and does not fully capture the 2026 taxonomy. This is a known-weakness the title_lookup_llm tier is supposed to catch; its 90% confidence threshold is conservative enough that these slip through.

    Observations:

    1. ML Engineer → SWE consistently. 96-99% of ML Engineer titles are classified as is_swe across all sources. Cross-period drift is minimal. No concern.

    2. Data Engineer → SWE with a small adjacent slice. 93-96% SWE, 2-4% adjacent. Slight drift: arshkon 93%, scraped 96% — scraped is slightly more aggressive about calling Data Engineers SWE. This is compatible with real vocabulary shift (Data Engineering is more code-heavy in 2026 job descriptions).

    3. DevOps Engineer → overwhelmingly SWE. 97-99% across all sources. Tight.

    4. Platform Engineer is a concern. Asaniczka has 81% SWE / 19% adjacent, but scraped has 89% SWE / 5% adjacent. Cross-period drift of ~8 pp on the SWE share. This is the boundary-case role most at risk of differential classification between periods. Action: downstream tasks that include Platform Engineers should run a sensitivity with platform-engineer-only tagged separately.

    5. Security Engineer → adjacent consistently. 96-97% adjacent, 2-3% SWE. The small SWE share likely reflects roles titled "Security Software Engineer" that the classifier correctly identifies as SWE. Scraped SWE share is slightly higher (2.8-3.0% vs 0.6-1.6% on 2024) — this may reflect real growth in product-security-engineering roles or minor classifier drift. Too small to move aggregate numbers.

    6. QA Engineer split is the noisiest boundary. Asaniczka classifies 9% as SWE / 4% adjacent / 86% other (i.e., most asaniczka "QA Engineer" titles don't match the SWE regex narrowly enough and fall through into the unclassified pool). Scraped is 35-37% SWE / 18-21% adjacent / 42-45% other — a major jump in the SWE share. This is significant and is almost certainly due to the 2026 scraped pool having more "QA Software Engineer" / "SDET" / "QA Automation Engineer" variants that trip regex patterns.

    7. This last pattern — QA Engineer SWE share jumping from 9% to 37% between asaniczka and scraped — is a genuine classifier drift concern. If downstream tasks compare "SWE share of QA" or related metrics, this drift should be controlled for. The aggregate impact on total is_swe counts is small (~500 extra SWE rows in scraped vs a baseline of 19K-21K), but the share of QA roles flagged as SWE is period-dependent.

    Among rows where the LLM did provide a classification, rate of LLM saying NOT_SWE (possible FP indicator):

    • arshkon: 365 / (2,283+269+365) = 12.5% LLM says NOT_SWE
    • asaniczka: 1,074 / 5,190 = 20.7%

    Asaniczka has the highest LLM-disagreement rate (20.7%), consistent with its larger title_lookup_llm share and looser title filtering. Scraped has the lowest (7%). This is a reasonable upper bound on classifier FP: 7-21% of the LLM-labeled subset. But the LLM is only called on a subset of rows, and those are rows the rule-based pipeline was uncertain about, so this FP rate does NOT apply to the full SWE sample — just to the LLM-labeled subset, which is 18-29% of total SWE.

    Combining with the tier breakdown: regex-tier FP ~0%, title_lookup_llm-tier FP ~24%, weighted by population share → aggregate FP ~1.3-2.7% (consistent with the borderline sample analysis in Section 3).

    Surprises and unexpected patterns¶

    1. The swe_confidence column is not a continuous score. It takes exactly three values (1.0 for regex, 0.9 for title_lookup_llm, ~0.89 for embedding_high). The task spec said to sample "confidence 0.3-0.7 or tier=title_lookup_llm" but the former is empty. This is not a bug — it reflects the tier-based design — but it means the borderline sample had to be drawn purely from title_lookup_llm.

    2. QA Engineer SWE-share jumps from 9% (asaniczka) to 37% (scraped). This is the only boundary case that shows substantial cross-period drift. Probably reflects real 2026 vocabulary shift (more "Software QA" / "SDET" postings) but worth flagging for downstream work.

    3. Security Engineer is almost never SWE (2-3%), even though "Security Software Engineer" titles exist. The adjacent classification is correct, but it means any task looking at "security-related SWE roles" cannot just filter is_swe — it needs to explicitly include is_swe_adjacent for security-engineer capture.

    4. Control category is cleaner than expected. Accountants, nurses, EEs, MEs — no contamination with SWE-adjacent roles in the top 50.

    5. False negatives in the non-SWE pool are concentrated in 2026 newer roles (DevSecOps, Pega, web-performance engineers). The classifier was tuned on 2024 vocabulary and misses some 2026-native role names. Aggregate impact is small (~6% of the filtered subset, ~1-2% of the true non-SWE pool).

    6. 3 false negatives in 50 non-SWE samples is higher than the ~1% I expected from a "carefully designed" classifier. But all 3 are edge cases (Pega, web perf, DevSecOps) that legitimate SWE rosters might disagree on — not egregious misses.

    Data quality caveats¶

    1. swe_confidence is effectively a tier code. Sampling "confidence in [0.3, 0.7]" returns 0 rows. Document this for future agents so nobody writes a filter that expects continuous confidence values.

    2. The title_lookup_llm tier has a 0.9 blanket confidence. The LLM was called only when the rule-based stage could not decide; it doesn't produce a granular confidence. The tier is the signal.

    3. Non-English and non-OK rows were excluded by the default filter. Classification quality on non-English or date-flag-bad rows is not assessed here and is irrelevant to the primary analysis.

    4. Borderline sample size is 50 per direction (100 total). The 95% Wilson CI on a 24% proportion with n=50 is roughly [14%, 37%]. Don't over-interpret the exact number; the takeaway is "order of magnitude 20-30% within title_lookup_llm tier."

    5. Boundary case counts use simple LIKE matching on title. Titles with unusual phrasings (e.g., "Machine Learning Platform") may not match. The boundary analysis is indicative, not exhaustive.

    Action items for downstream tasks¶

    1. Use is_swe as primary. FP rate is ~2-3% aggregate, well within acceptable bounds.

    2. For effect sizes <5%, rerun with swe_classification_tier != 'title_lookup_llm' (sensitivity dimension g). Report the delta. If the finding survives, it's robust to SWE FP.

    3. Treat the QA boundary as period-drift-risky. When a task compares a metric across periods that includes QA Engineers, rerun once excluding rows whose title normalizes to *quality* or *qa* and report the delta.

    4. Treat Platform Engineer similarly. The 81% → 89% SWE-share drift between asaniczka and scraped is small but non-zero.

    5. Consider a Stage 5 rule refresh to catch 2026 vocabulary (DevSecOps, Pega Developer, Web Performance Engineer, Cloud Platform Engineer, Forward Deployed Engineer). This is the main FN pattern. Not required for Wave 1 analysis, but a high-value preprocessing improvement for a future pipeline run.

    6. Do NOT apply a confidence threshold filter — confidence is constant within tier so any such filter reduces to a tier filter. Use the tier column directly.
