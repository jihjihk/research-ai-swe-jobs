    T03 — Seniority Label Audit¶

    Agent B (Wave 1). Input: data/unified.parquet. Default filter: is_english = TRUE AND date_flag = 'ok' AND source_platform = 'linkedin' AND is_swe = TRUE.

    This is the canonical seniority-quality reference for the rest of the exploration. Downstream tasks should cite the verdict at the bottom rather than re-running these checks.

    Headline finding¶

    seniority_final is defensible as the primary seniority column, but with one specific caveat: arshkon seniority_native = 'entry' is substantially less trustworthy than its scraped-period counterpart, so any sanity-check that uses seniority_native as a reference must be restricted to arshkon AND interpreted with the caveat that ~26% of arshkon native-entry rows have YOE >= 5. Strict kappa between seniority_final and seniority_native is 0.45 on arshkon vs 0.65 on scraped — a material temporal-instability gap in the native classifier.

    The three-way junior-share comparison (2024 pooled LinkedIn SWE → 2026):

    Operationalization 2024 2026 Direction Magnitude
    (a) seniority_final = 'entry' (of all rows) 1.66% 3.12% ↑ ~2x
    (a) seniority_final = 'entry' (of known-seniority rows) 2.70% 6.70% ↑ ~2.5x
    (b) seniority_native = 'entry' (arshkon-only, all rows) 15.73% (n/a) — —
    (c) YOE <= 2 (label-independent, of all rows) 7.74% 11.60% ↑ ~1.5x
    (c) YOE <= 2 (of YOE-known rows) 11.22% 17.19% ↑ ~1.5x

    All three production-capable operationalizations agree on direction (junior share rose 2024→2026). They disagree substantially on level (seniority_final is anchored at a much lower base because of the large unknown bucket on scraped) and on magnitude (seniority_final ≈ 2x, YOE proxy ≈ 1.5x). The discrepancy is driven by differential LLM-routing: on scraped, ~40% of SWE rows have seniority_final_source = 'unknown' (no rule fired and the row was not routed to the LLM, or the LLM returned unknown), so seniority_final-based denominators are dominated by a very different mix of "unknowns" than on 2024 sources. See the disagreement-mechanism section below.

    Methodology¶

    Scripts: - exploration/scripts/T03_seniority_audit.py — main profile and cross-tab generator - exploration/scripts/T03_kappa_and_diagnostics.py — Cohen's kappa on the cross-tabs - exploration/scripts/T03_native_quality.py — YOE profile of native='entry', path × final mapping - exploration/scripts/T03_weak_marker_sample.py — 128-row sample of LLM-routed SWE rows with weak seniority markers

    Data: data/unified.parquet. Filter applied throughout: is_english = TRUE AND date_flag = 'ok' AND source_platform = 'linkedin' AND is_swe = TRUE (defaults from the preamble).

    Seniority columns per docs/preprocessing-schema.md Section 4: - seniority_native — platform-provided LinkedIn label (arshkon 100%, asaniczka mid-senior/associate only, scraped 100%) - seniority_final — combined Stage-5 rule + Stage-10 LLM column (production) - seniority_final_source — title_keyword | title_manager | llm | unknown

    1. seniority_final_source profile (SWE LinkedIn)¶

    Source table: exploration/tables/T03/01_final_source_profile_with_pct.csv

    source period n (SWE) title_keyword title_manager llm unknown
    kaggle_arshkon 2024-04 4,691 37.8% 0% 62.2% 0.06%
    kaggle_asaniczka 2024-01 18,129 58.8% 0% 28.6% 12.6%
    scraped 2026-03 19,777 44.1% 0% 17.6% 38.4%
    scraped 2026-04 21,104 43.0% 0.005% 16.3% 40.7%

    Key takeaways: - The LLM-routed share differs massively by source: arshkon 62%, asaniczka 29%, scraped 17%. This reflects LLM budget allocation — arshkon gets the highest LLM coverage per row because it's the smallest source, while scraped gets the least. - The unknown share on scraped is 39-41% — nearly 4 in 10 scraped SWE rows have NO strong title rule AND were not routed to the LLM (or the LLM returned unknown). These rows carry seniority_final = 'unknown' and drop out of any seniority-stratified analysis. - Arshkon has only 0.06% unknown, because the small row count means everything got rule-matched or routed.

    Implication for cross-period comparisons: the seniority_final = 'entry' rate on scraped is computed against a denominator where ~40% of the rows are unknown by construction. Percent-of-all metrics therefore understate the entry level share on scraped relative to arshkon. Percent-of-known-seniority is the better denominator, but still depends on the unknown rows being unknown "at random," which we cannot verify.

    2. Rule-vs-LLM internal agreement on weak-marker titles¶

    Source: exploration/tables/T03/05_llm_routed_marker_summary.csv, 05_llm_routed_weak_marker_sample_v2.csv

    We sampled the full population (1,836 rows) of LLM-routed SWE rows whose raw title contains a weak seniority marker (Junior/Jr./Intern, Senior/Staff/Principal/Lead/Sr., or I/II/III/IV) and looked at what the LLM returned. These are rows where a strong title rule did NOT fire (so normalization stripped the marker, or the keyword list is narrower than ours), meaning the LLM is operating on description context.

    Bucket sizes and LLM verdicts:

    source marker_kind n LLM → entry LLM → mid-senior LLM → associate LLM → unknown
    kaggle_arshkon junior_marker 8 7 0 0 1
    kaggle_arshkon senior_marker 120 0 118 0 2
    kaggle_arshkon roman_marker 123 1 14 1 107
    kaggle_asaniczka junior_marker 16 15 0 0 1
    kaggle_asaniczka senior_marker 399 1 385 0 13
    kaggle_asaniczka roman_marker 504 1 63 0 440
    scraped junior_marker 21 14 3 0 4
    scraped senior_marker 181 1 175 0 5
    scraped roman_marker 464 2 56 8 398

    Findings: - Junior markers (Junior/Jr./Intern/New Grad): LLM says entry 36/45 times (80%). Disagreements are typically legitimate — e.g., titles like "Junior/Mid-Level Software Engineer" or edge cases where the description contradicts the marker. - Senior markers (Senior/Staff/Principal/Lead/Sr.): LLM says mid-senior 678/700 times (97%). Very consistent — essentially zero routing errors on this marker class. - Roman-numeral markers (I/II/III/IV): LLM correctly refuses to guess. 945/1,091 (87%) return unknown, and only 0.4% return entry. This is DESIRED behavior — a level-II engineer at one company is a junior SWE while at another company it's a mid-level, and the LLM appropriately signals uncertainty. This is the "LLM seniority design rationale" described in preprocessing-schema.md (seniority_final_source = 'llm' AND seniority_final = 'unknown' is valid output, not a defect).

    Routing-error rate estimate: <5%. The LLM's routing is consistent with a strong rule everywhere a rule would have produced a confident answer. The only concern is the Junior marker disagreements (~20%), but manual inspection of a handful shows they are genuinely ambiguous titles.

    3. Cross-tabs and Cohen's kappa — seniority_final vs seniority_native¶

    Sources: exploration/tables/T03/03_crosstab_arshkon.csv, 03_crosstab_scraped_linkedin.csv, 03_kappa_summary.csv

    Two regimes: - Strict — restricts to rows where native is known AND seniority_final != unknown. Measures the agreement on the subset the production pipeline confidently labeled. - Lenient — counts seniority_final = unknown as a disagreement. Measures the full agreement including coverage loss.

    Native labels intern and executive were mapped to entry and director respectively for the comparison.

    Tag N strict κ strict κ lenient Entry recall (strict) Entry recall (lenient) Mid-senior recall (strict)
    arshkon 1,667 0.454 0.184 52.1% 9.6% 98.0%
    scraped_linkedin 15,982 0.646 0.135 81.3% 18.8% 98.2%

    Two critical findings:

    1. Temporal instability in LinkedIn's native classifier. Strict kappa is 0.65 on scraped but 0.45 on arshkon — a 0.20 gap. Entry-level recall (strict) is 81% on scraped but only 52% on arshkon. This is not a production pipeline defect; the production pipeline agrees with native labels well on scraped but struggles with arshkon because arshkon's native entry labels are themselves noisy. See section 4.

    2. Coverage collapse on scraped (lenient kappa). Lenient kappa is 0.13 on scraped — worse than arshkon's 0.18 — because ~47% of native-labeled scraped SWE rows fall into seniority_final = unknown. This is the ~40% "scraped unknown" bucket showing up in kappa terms.

    Associate recall is near-zero (1.5% strict arshkon, 1.8% strict scraped). This reflects a design decision: the LLM prompt does not routinely output associate as a category, so the rare cases where native says associate almost always map to something else in final.

    Director recall is 22-26% strict — the rule-based pipeline does have director detection but it's narrower than LinkedIn's director definition (which seems to include a lot of VP/Director-titled engineering leaders that our pipeline calls mid-senior).

    4. Native-label quality diagnostic — the arshkon entry-label problem¶

    Source: exploration/tables/T03/07_native_entry_yoe_profile.csv

    For rows where seniority_native = 'entry', what does yoe_extracted look like?

    source period n_native_entry mean YOE median % with YOE % YOE ≤ 2 % YOE ≥ 5 % YOE ≥ 7
    kaggle_arshkon 2024-04 738 4.18 3 62.1% 17.8% 26.4% 9.5%
    scraped 2026-03 2,264 2.34 2 39.6% 32.3% 4.2% 1.6%
    scraped 2026-04 2,425 2.37 2 38.0% 30.7% 4.0% 1.7%

    This is the most important data-quality finding of T03. Arshkon's native-entry rows have mean YOE of 4.18 and 26% of them have YOE >= 5. These are not entry-level roles by any reasonable definition — they are a mix of actual juniors, associate/mid-level, and even some senior roles that happen to be tagged "entry" by LinkedIn's 2024-era labeler.

    Scraped 2026 native-entry rows look much cleaner — mean YOE 2.34, median 2, only 4% ≥ 5 YOE. This is consistent with what one would expect from a native "entry" label.

    Mechanism hypothesis: LinkedIn appears to have changed how it assigns the 'entry' label between early 2024 and 2026. In the 2024 arshkon snapshot, 'entry' captured a much broader pool that included a lot of mid-level roles. By 2026, 'entry' tracks YOE much more tightly. This is not something our preprocessing pipeline can fix — it is a property of the upstream label.

    Implication: Any analysis that uses seniority_native = 'entry' as a ground-truth reference on arshkon will OVERESTIMATE the 2024 entry share (because the label is polluted with mid-level rows). This partially explains why the seniority_native arshkon-only 2024 entry share (15.7%) is so much higher than both seniority_final 2024 (2.7% of known) and the YOE proxy 2024 (11.2% of YOE-known) — the 2024 native label simply does not mean what it says.

    Implication for downstream tasks: Do NOT treat arshkon seniority_native = 'entry' as ground truth. Use it only as a diagnostic to flag where the production pipeline disagrees. The YOE proxy is a better label-independent anchor for cross-period entry-share comparisons.

    Do they agree?¶

    Direction: YES — seniority_final (all three denominators) and the YOE proxy all show a rise from 2024 to 2026. Magnitude differs but direction is unanimous.

    Magnitude: - seniority_final (of all rows): 1.66% → 3.12% (×1.9) - seniority_final (of known seniority): 2.70% → 6.70% (×2.5) - YOE ≤ 2 (of all rows): 7.74% → 11.60% (×1.5) - YOE ≤ 2 (of YOE-known rows): 11.22% → 17.19% (×1.5)

    Levels differ by ~5x between seniority_final and the YOE proxy. The YOE proxy is anchored at a much higher rate because "YOE ≤ 2" includes a lot of roles that require 1-2 YOE minimum but are not labeled entry-level by title or description. These are mid-career roles with flexible YOE floors, not junior roles. The YOE proxy is therefore a loose upper bound on juniors, while seniority_final is a tight lower bound.

    The seniority_native arshkon-only figure (15.7%) is entirely inconsistent with the other two and — per section 4 — is best explained by the arshkon native classifier being too generous. Downstream tasks should not rely on arshkon seniority_native='entry' as a 2024 baseline.

    6. Mechanism investigation — why do the levels differ?¶

    The seniority_final vs YOE-proxy level gap (~5x) is driven by:

    1. The scraped "unknown" bucket. 39-41% of scraped SWE rows have seniority_final = 'unknown'. These rows DO have YOE extracted in some cases, so they show up as numerator in the YOE proxy but not in the seniority_final='entry' numerator. This drives the YOE proxy higher.

    2. YOE "at most 2" captures broad mid-level roles. Many mid-level postings specify a YOE floor of 2 or 3. The yoe_extracted <= 2 filter does not distinguish "1-2 YOE requirement for an entry role" from "2+ YOE minimum for a mid role." The LLM pipeline correctly distinguishes them (so seniority_final is more conservative) while the YOE proxy does not.

    3. The arshkon native label is polluted but the production pipeline handles it gracefully. Of 738 arshkon native-entry rows, 618 (84%) end up as seniority_final = 'unknown' — the LLM correctly refuses to call them entry when the description evidence is ambiguous. A further 53 map to entry, 65 to mid-senior, and 2 to associate. The production pipeline is more conservative than the native label in 2024, which is the right behavior given the native label's noise.

    Bottom line on the magnitude disagreement: both operationalizations are measuring different things. seniority_final measures "postings with explicit entry-level signals"; the YOE proxy measures "postings with low experience requirements." Both rose from 2024 to 2026. The direction is robust.

    7. Recommendation — is seniority_final defensible as primary?¶

    YES. seniority_final is defensible as the primary seniority column for the rest of the exploration. Key reasons:

    1. It is conservative where the data is noisy (arshkon native 'entry' → most become 'unknown' because the LLM cannot confirm, which is the right behavior).
    2. It agrees with the label-independent YOE proxy on direction in cross-period analyses.
    3. LLM routing errors are <5% on weak-marker titles.
    4. Rule and LLM halves are internally consistent where both fire.

    Required caveats for downstream use:

    1. Report results both "of all rows" and "of known-seniority rows." The scraped ~40% unknown bucket means the "of all rows" denominator is not comparable across sources. Always report both denominators.

    2. Always pair with the YOE proxy. Any seniority-stratified finding must show seniority_final primary + YOE proxy as label-independence check. If they disagree on direction, DO NOT pick a side without investigating (as per the core preamble).

    3. Do not use seniority_native='entry' as a 2024 ground-truth baseline on arshkon. The arshkon entry label has mean YOE 4.18 and 26% of rows have YOE ≥ 5; it is too noisy to anchor a cross-period comparison. Use arshkon seniority_native only as a diagnostic, not a reference.

    4. Interpret cross-period entry-share rises conservatively. The rise is real (direction agrees under all three ops) but part of it is mechanically driven by LLM coverage differences between sources (arshkon LLM coverage 62% vs scraped 17%). The LLM can only re-label rows it sees, so arshkon's lower unknown rate may mean it has more entry rows surfaced via the LLM than scraped does, inflating the 2024 denominator structure.

    No remediation needed. The pipeline behavior is correct given the upstream data quality. The downstream protocol is the mitigation, not a code fix.

    Surprises and unexpected patterns¶

    1. Arshkon's native entry label is the noisy one, not scraped's. Going in, I assumed 2024 native labels would be cleaner (older data, more curation) and 2026 scraped labels would be the unreliable ones. The reverse is true: arshkon entry has mean YOE 4.18, scraped entry has mean YOE 2.34. This is a big update for how I think about the 2024 baseline.

    2. The LLM returning 'unknown' on Roman-numeral titles is a feature, not a bug. 87% of Engineer II/III/IV titles get final='unknown'. This is the design working correctly — company-internal level numbers don't map cleanly to junior/mid/senior without context. Downstream tasks must not interpret this as a data-quality defect.

    3. The LLM was called but Stage 5 title-keyword rules still dominate asaniczka (58.8% keyword vs 28.6% LLM). This is higher than I expected given asaniczka's large size. The title-keyword classifier is catching a lot of Senior/Lead/Principal/Staff roles in asaniczka.

    4. Zero title_manager rows almost everywhere — only 1 in scraped 2026-04. This may suggest the manager-title classifier is under-utilized; worth a one-line follow-up.

    Data quality caveats¶

    1. title column in unified.parquet contains some rows with invalid UTF-8 byte sequences — DuckDB's regex engine raises InvalidInputException when combined with window functions on these rows. Work-around: use LIKE patterns instead of regexp_matches, or normalize via Python after fetching with encode(). This is a preprocessing issue worth fixing in a future pipeline run (a one-line replace_invalid_utf8 during ingestion would be sufficient).

    2. title_normalized strips seniority markers (Junior/Senior/Jr/Sr), so marker-based sampling must use the raw title column. This is by design but worth documenting for other agents.

    3. Stage 9/10 LLM budgets differ across sources by a factor of ~4x in terms of coverage (arshkon 62% LLM-routed vs scraped 17%). Any cross-source comparison of seniority classification quality has to account for this.

    4. The Indeed subset (not under the default filter) has seniority_native = NULL entirely, so cross-validation on Indeed is not possible for the native-label audit.

    Action items for downstream tasks¶

    1. Use seniority_final as the primary seniority column. Do not re-derive seniority or apply alternative imputation schemes unless the task explicitly argues for one.
    2. Always pair seniority_final-based results with the YOE ≤ 2 label-independent check. Report both. Disagreement on direction is a finding; disagreement on magnitude usually is not.
    3. Never report seniority_native='entry' rates on arshkon as a cross-period baseline without the caveat that arshkon's 2024 entry label has mean YOE 4.18 and is demonstrably noisy.
    4. For percent-of-known calculations, always report the unknown fraction alongside. ~40% of scraped SWE rows are unknown — downstream tasks need to know how much of the denominator is being dropped.
    5. When computing the 2024→2026 entry-share change, report it under both seniority_final and the YOE proxy. Both show the rise, but magnitudes differ by ~60%. Treat the YOE proxy as a loose upper bound and seniority_final as a tight lower bound.
    6. Consider filing a preprocessing follow-up for (a) invalid UTF-8 in raw title and (b) the very low title_manager hit rate (effectively unused).
