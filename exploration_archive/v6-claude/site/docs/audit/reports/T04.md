# T04 — SWE Classification Audit

Agent B (Wave 1). Input: `data/unified.parquet`. Default filter: `is_english = TRUE AND date_flag = 'ok' AND source_platform = 'linkedin'`.

## Headline finding

**SWE classification is reliable enough to use as primary.** The regex tier (88-93% of `is_swe=TRUE` rows) is essentially noise-free, the dual-flag invariant holds perfectly, and the boundary categories (ML/Data/DevOps/Security) partition cleanly with only modest period drift. The main risk is the `title_lookup_llm` tier (5-11% of SWE rows): manual labeling of 50 borderline rows from this tier estimates **24% hard false-positive and 28% borderline-ambiguous**, weighted to an aggregate FP contribution of roughly **1.3-2.7%** of the full SWE sample. False negatives in the "contains engineer/developer/software" non-SWE pool are rare (~6% hard FN in 50 samples from a much larger pool), with most missed cases being DevSecOps / low-code platform developers / web-performance engineers.

**Recommendation:** Use `is_swe` as-is for primary analyses. For any headline finding whose magnitude could plausibly be driven by a 2-3% classification FP, rerun with tier `title_lookup_llm` excluded and report the delta (sensitivity dimension g in the preamble).

## 3. Borderline SWE sample (50 rows, tier=title_lookup_llm)

Sampled 50 rows from the `title_lookup_llm` tier (since there is no confidence-in-[0.3,0.7] subpopulation). Manually labeled each as SWE / BORDERLINE / NOT_SWE based on title + first 250 chars of description. Full labels: `exploration/tables/T04/04_borderline_swe_sample_labeled.csv`.

**Clear false positive examples in the sample:**
- *Computer Engineer IV* — Combat vehicle systems engineer, hardware-adjacent (not SWE)
- *Lead Test Engineer* — Aerospace systems engineer at Leidos (not SWE)
- *Quality Assurance Engineer* — Magnera (materials manufacturer) QA (not SWE)
- *Automation Engineer* — Food manufacturing industrial automation (not SWE)
- *Sr Automation Engineer* — Electronic circuits / manufacturing (not SWE)
- *Senior Systems Integration Engineer* — Medtronic medical device systems (not SWE)
- *Reliability Engineer* — Bulk manufacturing (not SWE)
- *Principal Systems Test Engineer* — DoD hardware test

These share a pattern: "Engineer" titles in industrial/aerospace/medical-device companies where the role is not writing software. The `title_lookup_llm` path catches the generic "Engineer" token without sufficient description context.

**Borderline examples (ambiguous):**
- QA Automation / Test Engineer / Software Quality Engineer — debatable whether these are SWE or SWE_adjacent
- DevSecOps Engineer II, Senior DevSecOps Engineer — I labeled these SWE (they write automation code) but reasonable people could call them adjacent
- FPGA Firmware Engineer, Senior Firmware Engineer — embedded systems dev, borderline
- Junior QA Engineer — pure QA, borderline

**FP contribution estimate:**
- Hard FP rate within title_lookup_llm tier: **24%** (12/50)
- Borderline rate: **28%** (14/50), which if half should really be adjacent ≈ 14% additional leakage
- Title_lookup_llm share of total SWE: **5-11%** across sources
- Aggregate hard-FP contribution to SWE sample: **24% × (5-11%) = 1.3-2.7%**
- Aggregate soft-FP contribution: add ~1-2% for borderline leakage → ~2.3-4.7% total

**This is within the 5% tolerance typical for production NLP classifiers.** It is NOT large enough to invalidate cross-period comparisons, but it IS large enough that any headline finding whose effect size is smaller than 5% should be rerun with `swe_classification_tier = 'title_lookup_llm'` excluded (sensitivity dimension g).

- *Senior Pega Developer* — Pega is a low-code platform but Pega developer is a real developer role. Classified as none.

**Borderline cases worth noting:**
- *Workday Financials Developer/Analyst*, *Sr Workday Payroll Engineer* — Workday is platform config work; debatable whether it counts as SWE
- *Dashboard BI Developer* — BI/dashboarding is borderline
- *Sr. Software Cybersecurity Engineer* — Abbott medical cybersecurity, could be SWE or SWE_adjacent (correctly tagged adjacent)
- *Senior Commissioning Engineer - Robotics* — Robotics with smart control code, borderline
- *Quality Engineer 1* — "Software Quality Engineer 1" with automated testing, borderline adjacent vs SWE

**FN rate estimate:**
- Hard FN rate in this pool: **6%** (3/50)
- Borderline rate: **24%** (12/50)
- This sample is biased *toward* finding FNs because we filtered the non-SWE pool to rows whose title contains "engineer/developer/software" — the base rate of SWE in the remaining ~950K non-SWE postings (that do NOT contain these words) is much lower.
- The title-filter subset contains roughly 150K-200K rows, so ~6% FN × 150K ≈ 9K missed SWE rows across all sources. Spread over 4 period-buckets, that's ~2K per bucket — non-trivial but small relative to the 19K-21K SWE per scraped bucket.

**The FN pattern has an interpretable structure:** the missed cases are overwhelmingly in newer specialized roles (DevSecOps, Pega platform dev, web-performance engineering) or platform-config work (Workday, Salesforce, BI). The rule-based regex was designed for 2024 vocabulary and does not fully capture the 2026 taxonomy. This is a known-weakness the `title_lookup_llm` tier is supposed to catch; its 90% confidence threshold is conservative enough that these slip through.

## Surprises and unexpected patterns

1. **The `swe_confidence` column is not a continuous score.** It takes exactly three values (1.0 for regex, 0.9 for title_lookup_llm, ~0.89 for embedding_high). The task spec said to sample "confidence 0.3-0.7 or tier=title_lookup_llm" but the former is empty. This is not a bug — it reflects the tier-based design — but it means the borderline sample had to be drawn purely from title_lookup_llm.

2. **QA Engineer SWE-share jumps from 9% (asaniczka) to 37% (scraped).** This is the only boundary case that shows substantial cross-period drift. Probably reflects real 2026 vocabulary shift (more "Software QA" / "SDET" postings) but worth flagging for downstream work.

3. **Security Engineer is almost never SWE (2-3%), even though "Security Software Engineer" titles exist.** The adjacent classification is correct, but it means any task looking at "security-related SWE roles" cannot just filter `is_swe` — it needs to explicitly include `is_swe_adjacent` for security-engineer capture.

4. **Control category is cleaner than expected.** Accountants, nurses, EEs, MEs — no contamination with SWE-adjacent roles in the top 50.

5. **False negatives in the non-SWE pool are concentrated in 2026 newer roles** (DevSecOps, Pega, web-performance engineers). The classifier was tuned on 2024 vocabulary and misses some 2026-native role names. Aggregate impact is small (~6% of the filtered subset, ~1-2% of the true non-SWE pool).

6. **3 false negatives in 50 non-SWE samples is higher than the ~1% I expected from a "carefully designed" classifier.** But all 3 are edge cases (Pega, web perf, DevSecOps) that legitimate SWE rosters might disagree on — not egregious misses.

## Data quality caveats

1. **swe_confidence is effectively a tier code.** Sampling "confidence in [0.3, 0.7]" returns 0 rows. Document this for future agents so nobody writes a filter that expects continuous confidence values.

2. **The `title_lookup_llm` tier has a 0.9 blanket confidence.** The LLM was called only when the rule-based stage could not decide; it doesn't produce a granular confidence. The tier is the signal.

3. **Non-English and non-OK rows were excluded by the default filter.** Classification quality on non-English or date-flag-bad rows is not assessed here and is irrelevant to the primary analysis.

4. **Borderline sample size is 50 per direction (100 total).** The 95% Wilson CI on a 24% proportion with n=50 is roughly [14%, 37%]. Don't over-interpret the exact number; the takeaway is "order of magnitude 20-30% within title_lookup_llm tier."

5. **Boundary case counts use simple LIKE matching on `title`.** Titles with unusual phrasings (e.g., "Machine Learning Platform") may not match. The boundary analysis is indicative, not exhaustive.

## Action items for downstream tasks

1. **Use `is_swe` as primary.** FP rate is ~2-3% aggregate, well within acceptable bounds.

2. **For effect sizes <5%, rerun with `swe_classification_tier != 'title_lookup_llm'`** (sensitivity dimension g). Report the delta. If the finding survives, it's robust to SWE FP.

3. **Treat the QA boundary as period-drift-risky.** When a task compares a metric across periods that includes QA Engineers, rerun once excluding rows whose title normalizes to `*quality*` or `*qa*` and report the delta.

4. **Treat Platform Engineer similarly.** The 81% → 89% SWE-share drift between asaniczka and scraped is small but non-zero.

5. **Consider a Stage 5 rule refresh** to catch 2026 vocabulary (DevSecOps, Pega Developer, Web Performance Engineer, Cloud Platform Engineer, Forward Deployed Engineer). This is the main FN pattern. Not required for Wave 1 analysis, but a high-value preprocessing improvement for a future pipeline run.

6. **Do NOT apply a confidence threshold filter** — confidence is constant within tier so any such filter reduces to a tier filter. Use the tier column directly.
