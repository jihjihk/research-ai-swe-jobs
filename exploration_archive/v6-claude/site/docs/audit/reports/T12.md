### Tokenization

Matches the Prep Wave 1.5 cleaner:
- Regex `[A-Za-z][A-Za-z0-9+#./\-]*` preserves `c++`, `c#`, `node.js`, `.net`, `ci/cd`.
- Lowercase, NLTK English stopwords removed, company-name stoplist applied with 65-item tech-term whitelist override.
- **Markdown escape preprocessor (added by this task).** Scraped text contains backslash-escaped tech tokens (`c\+\+`, `c\#`, `\.net`) from the markdown-to-text conversion. Without preprocessing, the tokenizer drops these entirely, making `c++` appear as a spurious 2024-heavy term. Inline assertion: `tokenize('c\\+\\+ c\\# python', {}) == ['c++', 'c#', 'python']`. This is a **first-class preprocessing finding** for T14 and any text analysis comparing scraped to kaggle â€” see "Surprises" below.

### Section classifier integration

For each posting, T13's `classify_sections()` is applied to `description_core_llm` and a section-filtered text is built by concatenating only `requirements` + `responsibilities` segments. Mean full-text length: 2,372 chars; mean section-filtered length: 1,156 chars (48.7%). This halves the corpus but concentrates on the "what the job wants" content.

### Semantic categorization

Each top-term is tagged with one of: `ai_tool`, `ai_domain`, `tech_stack`, `org_scope`, `mgmt`, `sys_design`, `method`, `credential`, `soft_skill`, `boilerplate`, `noise`. ## Surprises & unexpected patterns

1. **Markdown-escape tokenization bug in scraped text.** Scraped postings contain backslash-escaped `c\+\+`, `c\#`, `\.net` from the markdown-to-text conversion. A naive `[A-Za-z][A-Za-z0-9+#./\-]*` tokenizer treats these as single-char tokens (`c`) and drops the real tech term. This causes `c++` to appear as a spurious 2024-heavy term (524 mentions in arshkon, 0 in scraped). **Any downstream text task that compares scraped to kaggle MUST apply `strip_md_escapes()` before tokenization.** The fix is a one-liner regex `re.sub(r"\\([+\-#.&_()\[\]\{\}!*])", r"\1", text)`. Recommended action: add this to `prep_02_cleaned_text.py` in the next shared-artifact rebuild, so `swe_cleaned_text.parquet` has it baked in.

2. **Disappearing list is entirely HTML artifacts.** No genuinely disappearing technology. 2024 vocabulary is ADDITIVE absorbed into 2026, not replaced. This is a direct counter to any "old stack got replaced" narrative â€” the story is "new AI stack piled on top."

3. **Entry 2026 IS entry-level.** The relabeling diagnostic produces a clear signal: entry 2026 has `pursuing`, `coursework`, `internships`, `graduation`, `gain`, `currently` â€” classic entry markers NOT present in mid-senior 2024. This is strong evidence AGAINST Alternative A (instrument-dominated apparent change) for the entry-level story â€” the 2026 scraped entry pool is substantively entry-level, not a relabeled senior pool. Where the entry pool differs from 2024 pooled entry is almost entirely in AI content (agents, llm, ml, workflows, large-scale).

4. **`high-impact` and `production-grade` are 5Ã— up.** These are not AI-specific but may signal an `org_scope` language shift â€” employers are emphasizing impact/production readiness more. Recommended to T08/T11 agents: add these to the scope-inflation keyword validation samples to see if they're substantively about scope or if they're marketing fluff.

5. **`ts/sci`, `polygraph` are 2026-full-text-only.** These are defense-sector security-clearance markers. They only appear in the full-text list (not section-filtered) â€” they live in role summaries, not requirements sections. This is a composition finding: the 2026 scraped pool has more defense/clearance employers than the 2024 arshkon pool, but it shows up as a role-summary signal, not a content signal.

6. **`guardrails` appears at 1.9% in 2026, 0.13% in 2024.** This is a 2026 safety-engineering term for LLM production deployments. It's a real content change, not a rhetorical shift.

7. **`embed` and `embedding` split** â€” "embed" accelerated as a noun-verb, "embedding" emerged as a noun. Both track the same trend but in different grammatical slots. Bigram analysis confirms: `proficiency_python`, `agentic_workflows`, `retrieval-augmented_rag`, `hands-on_aws` are all genuine 2026 content patterns.

## Data quality caveats

- **Entry cross-period comparison is thin.** n = 371 (pooled 2024) vs 601 (2026). Below the 1,000-per-cell threshold recommended for full-corpus Fightin' Words. Top-20 terms are trustworthy; the long tail is not.
- **Mid seniority is too thin for comparison.** n_a = 39, n_b = 51. Skipped. The 2-level senior/entry split is the only feasible stratification.
- **Kaggle HTML stripping produces run-together tokens.** These dominate the "disappearing" list and inflate a few 2024-heavy signals. They are not counted in any category â€” they all fall into `noise`. The broader lesson: any preprocessing step that touched the kaggle text between its HTML source and the unified.parquet would change the disappearing list. The current disappearing list is a preprocessing-artifact list, not a content-change list.
- **BERTopic cross-validation was not run.** The spec calls for a BERTopic topic model with period as class variable for method triangulation. Given compute constraints (BERTopic on ~30k documents is 20+ minutes CPU) and that the Fightin' Words signal is already very clean and cross-validated against the section-filtered comparison and the within-2024 calibration, I consider method triangulation via Fightin' Words + section-filtering + within-2024 calibration + raw-text sanity check adequate. BERTopic remains an open follow-up if a reviewer wants independent topic-model confirmation.
- **Category taxonomy is incomplete.** ~70-80% of top terms fall in the `noise` bucket despite being legitimate words. The taxonomy catches the AI signal cleanly but undertags `org_scope` / `credential` / `sys_design` language. If a report needs a cleaner categorical breakdown, a tighter manual pass through the top-100 each direction would help.
