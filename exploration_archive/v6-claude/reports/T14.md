## Headline

1. **The Gate 1 "AI mention SNR 925" signal is dominated by a small AI/LLM cluster that did not exist in 2024 as a distinct tech ecosystem.** The three biggest contributors (absolute rise, 2024→2026) are `agents_framework` +12.1 pp, `llm` +10.2 pp, and `machine_learning` +8.0 pp. A first-class **AI-tool sub-community** (`copilot`, `cursor_tool`, `claude_tool`) and a much larger **LLM/RAG application community** (17 techs including `langchain`, `langgraph`, `rag`, `openai_api`, `claude_api`, `fine_tuning`, `mcp`, `agents_framework`, `pytorch`, `scikit_learn`) appear as discrete modules in the 2026 co-occurrence graph.
2. **Total distinct tech mentions per posting rose from median 4.0 → 6.0 across all SWE postings**, and the rise is concentrated in AI-mentioning postings (2024 AI-mentioning median 5.0 → 2026 AI-mentioning median 8.0). Density per 1K chars rose from 1.85 → 2.20 among AI postings while falling slightly (1.84 → 1.60) among non-AI postings, so the AI-posting expansion is **not** fully explained by description length inflation.
3. **Structured asaniczka skills and the regex tech matrix agree at Spearman ρ=0.947 on asaniczka SWE**, which is a clean validation of both the shared tech matrix and the structured-skills field.
4. **The `seniority_3level` breakdown is the weakest surface here:** asaniczka has only ~204 junior-labeled uids after Stage-10 LLM enrichment, so the structured chi-squared test flags only 5 skills at FDR 0.05 and every flagged skill appears in both "top junior" and "top mid+senior" lists because the test surfaces them at similar base rates from opposite sides. Structured-data seniority-level skill differences are effectively underpowered within asaniczka and should not be treated as a primary source.
5. **Tech breadth is rising, not specializing.** At seniority_3level=senior the median grew 4.0 → 6.0 (+2). For junior rows the rise is 4.0 → 5.0 (+1). Unknown rows rose 3.0 → 5.0. This looks like a broad stack-expansion, not a role-specialization.

## Methodology

- **Inputs.** Shared tech matrix (`exploration/artifacts/shared/swe_tech_matrix.parquet`, 63,701 rows × 123 boolean tech columns). Metadata from shared `swe_cleaned_text.parquet` joined by `uid`. Structured skills from `asaniczka_structured_skills.parquet` (478,638 skill rows, 18,114 uids).
- **Tech matrix is NOT rebuilt.** Tests in `prep_04` already validated tricky tokens (`c++`, `c#`, `.net`, `node.js`, `ci/cd`, `k8s`, `golang`, `java`-vs-`javascript`, `scala`-vs-`scalable`).
- **Company capping.** Per T06 concentration prediction table, co-occurrence network is run on cap-50-per-company-per-period frames. Mention-rate tables are reported uncapped as the primary, with a company-capping sensitivity table saved alongside.
- **Within-2024 calibration.** For every tech, the (arshkon_rate − asaniczka_rate) absolute difference is the instrument-noise baseline. The cross-period signal is (scraped_2026_rate − pooled_2024_rate) absolute. The SNR = cross / within-2024. The shared calibration table was consulted for metrics it already covers; only new metrics were computed.
- **Community detection.** NetworkX `greedy_modularity_communities` (weighted by phi coefficient), phi threshold 0.15, prevalence filter ≥1% in at least one period.

### 8. Structured vs extracted validation

Table: `structured_vs_extracted.csv`. Summary: `structured_vs_extracted_summary.txt`.

- **Spearman ρ = 0.947** (p ≈ 2.5e-29)
- **Pearson r = 0.924** (p ≈ 5.6e-25)
- **n_techs = 58** overlapping tokens

**Structured catches more than regex** (top diffs):
- `go_lang`: structured 6.95% vs extracted 2.15% (+4.80 pp). Likely because the regex only matches `go` or `golang` as bounded tokens, while the structured field captures "Go programming language".
- `machine_learning`: 6.91% vs 2.42% (+4.49 pp). Structured captures "machine learning" as a first-class skill while description regex only hits it when written verbatim.
- `computer_vision`: 1.31% vs 0.005% (+1.30 pp). Structured captures this; the regex barely fires.
- `deep_learning`, `kubernetes`, `python`, `git`, `gcp`, `docker`, `typescript` — small positive gaps.

**Regex catches more than structured** (top diffs):
- `agile`: extracted 30.5% vs structured 13.4% (−17.1 pp) — description says "agile" casually in far more postings than recruiters list it as a skill.
- `rest_api`: −8.3 pp
- `azure`: −6.3 pp
- `cicd`: −6.0 pp
- `sql_lang`: −5.4 pp
- `aws`: −4.4 pp
- `dotnet`: −3.6 pp

**Interpretation:** structured skills are stricter and closer to the **recruiter's curated skill list**; description regex is looser and captures mentions in prose ("we use AWS for our CI/CD pipeline"). The ρ=0.947 agreement on rank ordering is high enough to treat both as valid instruments of tech presence, with structured skills serving as a better **asaniczka-specific validation** of the regex-based tech matrix. No tech in the top 58 has the two instruments disagreeing on direction; the largest discrepancies are prose-usage amplification (agile, rest_api).

### 9. Seniority-level skill differences from structured data (asaniczka only)

Table: `asaniczka_skill_seniority_chi2.csv`. Summary: `asaniczka_skill_seniority_summary.txt`.

**Constraint:** asaniczka has no native entry-level labels. The seniority_3level on asaniczka rows comes from Stage-10 LLM. Only **204 asaniczka uids are junior-labeled** vs 11,532 mid+senior. The junior sample is thin for chi-squared.

750 skills tested (frequency ≥50). **5 skills significant at FDR<0.05**: iot, polygraph, excel, svn, nginx — each has 6-7 junior hits and 44-91 mid+senior hits, with junior rate 2.9-3.4% vs mid+senior 0.4-0.8% — so they're **over-represented among juniors** relative to base rate. But because the absolute counts are small, the same five skills dominate "top junior" and "top mid+senior" by construction in my report (the test surfaces them symmetrically). The finding is fragile.

**This sub-analysis is not a primary source for the report.** The structured chi-squared test on asaniczka seniority is **underpowered by roughly an order of magnitude** — there simply aren't enough asaniczka junior rows to distinguish entry vs mid+senior skill profiles. Any RQ1/RQ2 seniority-skill claim should come from `seniority_final` on the full joined frame, not from asaniczka-only structured chi-squared.

## Surprises & unexpected patterns

1. **The 2026 tech graph has HIGHER modularity than 2024 (0.66 vs 0.56).** Expectation going in was that tech bundles would blur. Instead, two crisp new modules (LLM/RAG application cluster and AI tools triad) carve themselves out of the embedding-similar tech space, increasing overall modularity.
2. **Web and Java backend merged in 2026.** In 2024 the front-end web community (React/Angular/Vue + Node/TS/JS/SQL + C#/.NET) and the Java+Spring+microservices community were separate. In 2026 they're one large 20-tech module that also absorbed all the automated-testing frameworks (jest, cypress, playwright, selenium). Hypothesis: the general-purpose full-stack role consolidated.
3. **C# + dotnet migrated from the web module to the data/analytics module in 2026.** They now co-occur with snowflake/databricks/bigquery/dbt more than with JS/React. Possible driver: enterprise .NET data pipelines are now more frequent than .NET front-end.
4. **Non-AI description density fell.** 2026 non-AI postings have MORE raw tech mentions per posting than 2024 non-AI postings, but LOWER tech-per-1K-chars density. The non-AI growth is mostly description bloat, not requirement expansion. This is **independent corroboration of the T13 "length growth is boilerplate" hypothesis**.
5. **`claude_tool` SNR 326 is the single highest-SNR tech in the corpus.** Even higher than copilot (44). The cross-period 2024→2026 shift is 0.01% → 3.37%, and the within-2024 arshkon/asaniczka delta is 0.01 pp. It's the textbook example of a "genuinely new tech that did not exist in the 2024 instrument."
6. **`agents_framework` is the largest absolute riser in pp (12.08) but has pattern noise.** Manual sampling confirms most 2026 matches are genuine AI-agent usage, but the 2024 baseline is overcounted by call-center / change-agent / security usages. The true AI-agent rise is probably 10-11 pp rather than 12 pp; directionally identical, slightly smaller.
7. **Structured asaniczka junior rows are too thin for skill chi-squared.** We have only 204 asaniczka junior uids vs 11,532 mid+senior. The seniority-skill difference test is operationally blocked by this thinness — a result worth flagging for the analysis plan.

## Data quality caveats

- Tech matrix is binary presence only; it does not measure emphasis. A posting mentioning "Python" once and one requiring five years of Python each count the same.
- `agents_framework` regex is broad — see caveat in Results §2. Any AI-narrative claim that leans hard on that single tech should be cross-checked against a tighter pattern (e.g., `ai agent|agentic`).
- Structured skills are asaniczka-only, so step-7/8/9 cannot be extended across periods. The 2024 snapshot alone can be used for asaniczka-internal diagnostics.
- Stage-9 LLM text coverage caps the 2026-side text at ~12.5k. Tech matrix, however, covers all 63,701 rows (patterns scan the cleaned text which exists for all rows), so tech analyses are not text-coverage-capped.
- Seniority-3level `mid` is tiny (39 rows 2024, 51 rows 2026) and should not be interpreted as a stand-alone bucket; it's folded into senior for most read-outs here.
