# T03: Missing Data Audit

## Finding

Missingness patterns are strongly source-driven rather than random. Asaniczka (1.01M rows, 88% of the filtered dataset) has zero coverage for company_industry, company_size, work_type, and skills_raw -- but 100% seniority_native (only mid-senior and associate). Arshkon has the richest structured metadata but the smallest SWE sample (4,438). The scraped 2026 data has good coverage on most fields but almost no date_posted (0.4%). The most consequential gap is seniority: after excluding "unknown", effective SWE sample sizes drop to 2,015 (arshkon), 15,286 (asaniczka), and 2,336 (scraped).

## Implication for analysis

1. **Cross-period seniority comparisons use 19,637 of 30,283 SWE rows (65%)**, and the composition of "known" seniority differs by source: asaniczka contributes zero native entry-level labels, while arshkon and scraped contribute the only entry-level baseline.
2. **The associate/entry overlap is extensive.** 60 title strings appear as both "associate" in asaniczka and "entry" in arshkon, covering the majority of entry-level volume. This means asaniczka's "associate" bin likely contains what arshkon would label "entry". Treating associate and entry as distinct levels across sources is unreliable without LLM re-classification.
3. **Structured skills data cannot support cross-period analysis.** skills_raw is 99.9% populated for asaniczka but 1.3% for arshkon and 0% for scraped. Description-text NLP is the only viable approach for skill/requirement analysis across periods.

## Associate/entry-level title overlap

**60 title strings** appear as both "associate" (seniority_native) in asaniczka and "entry" (seniority_native) in arshkon SWE rows. The top overlapping titles:

| Title | asaniczka associate count | arshkon entry count |
|---|---|---|
| software engineer | 105 | 52 |
| systems engineer | 48 | 19 |
| data engineer | 32 | 28 |
| automation engineer | 33 | 16 |
| test engineer | 39 | 9 |
| java developer | 22 | 18 |
| quality assurance engineer | 26 | 6 |
| software developer | 19 | 12 |
| developer | 5 | 23 |
| product engineer | 19 | 8 |

These are generic SWE titles that LinkedIn labels at different seniority levels across the two datasets. This confirms that **asaniczka's "associate" category substantially overlaps with arshkon's "entry" category**. Any analysis comparing entry-level shares across periods must account for this label inconsistency.

Additionally, within arshkon alone, many SWE titles appear at multiple seniority levels. For example, "software engineer" has 248 mid-senior, 52 entry, and 18 associate postings. "automation engineer" has 17 mid-senior, 16 entry, and 4 associate. This means the same title string is not a reliable proxy for seniority level.

## Data quality note

1. **Missingness is not random.** Asaniczka's zero coverage on company_industry, company_size, and work_type is structural (those fields were not in the original Kaggle dataset). Similarly, skills_raw being asaniczka-only reflects their companion skills file.
2. **The seniority_final "unknown" rate varies by source**, from 29% (asaniczka) to 55% (arshkon). Analyses excluding unknowns will over-represent asaniczka and under-represent arshkon in the pooled sample.
3. **Metro_area resolution varies**: 66% for arshkon, 75% for asaniczka/scraped. Geographic analyses lose 25-34% of SWE rows.
4. **yoe_extracted is sparse but comparably sparse** across sources (16-40%), so cross-period comparisons of YOE distributions are feasible, conditional on the extraction succeeding.

## Action items

1. **Do not treat asaniczka "associate" as equivalent to arshkon "associate".** Asaniczka "associate" likely includes what other sources would label "entry". Cross-period seniority analysis should either:
   - Use only arshkon + scraped (both have native entry labels), accepting the smaller sample
   - Wait for LLM seniority classification (Stage 12) to re-label all sources consistently
   - Use title-based seniority proxies as a robustness check
2. **Report entry-level counts alongside any junior-share statistics.** With n=89 (arshkon) and n=132 (scraped), confidence intervals will be wide.
3. **For cross-period analysis, the viable common field set is:** period, is_swe, seniority_final (with caveats), description, title_normalized, company_name_effective, state_normalized, metro_area (partial), is_remote_inferred, yoe_extracted (sparse).
4. **Do not use skills_raw, company_industry, company_size, or work_type for cross-period comparisons.** Extract these from description text instead.
5. **Downstream T-tasks that stratify by seniority must document their effective n** per cell and flag any cell with n < 100 as unreliable.
