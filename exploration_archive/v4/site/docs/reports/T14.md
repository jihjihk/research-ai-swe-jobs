**Caveat — c_lang extractor bug (data-quality finding).** The shared tech-matrix regex uses `\bc\+\+\b` for C++; the trailing `\b` cannot match after `+`, so C++ is under-extracted (see Step 8). ### The biggest structural change — AI cluster fusion

**In 2024, the modern-AI ecosystem was fragmented into three tiny isolated communities:**
- Comm 7: `claude_api, claude_tool, huggingface, openai_api` (4 nodes)
- Comm 9: `codex, copilot` (2 nodes)
- Plus one traditional ML community (Comm 2, 17 members: pytorch/tensorflow/sklearn/pandas/nlp/...).

**In 2026, all of these fused into a single 29-member ML+GenAI community (Comm 1).** Traditional ML tooling (pytorch, tensorflow, scikit_learn, pandas), classic ML subfields (NLP, CV, deep learning), modern-AI APIs (openai_api, claude_api, huggingface), agent/RAG infrastructure (langchain, rag, vector_databases, prompt_engineering, fine_tuning, mcp, ai_agents, transformers, sagemaker), AND AI coding tools (copilot, cursor_ide, codex, gemini, claude_tool) now all strongly co-occur.

**Interpretation:** In 2024 jobs that mentioned Copilot or Claude were distinct from jobs that mentioned TensorFlow. By 2026 they are the same jobs. Traditional ML engineers are expected to work with modern LLM APIs and agent frameworks; conversely, jobs posting about GenAI also list classical ML dependencies. This is a major consolidation of the AI-engineering identity.

### Data engineering swallows OLTP databases

2024 had a clean split: Comm 1 (enterprise Java / web / SQL / MongoDB / PostgreSQL / MySQL / Redis) vs Comm 4 (analytical — spark, kafka, snowflake, bigquery, airflow, dbt). In 2026 the OLTP data stores (mongodb, mysql, postgresql, redis, sql) **migrate into Comm 3 (data engineering)**. Their former home (Comm 2, now "Enterprise Java + web") becomes dominated by the Java + Spring + microservices + testing bundle. Interpretation: "the database layer" now belongs to data engineers, not generic web developers, in the co-mention structure.

### Test frameworks decoupled from practices

2024 Comm 3 packed jest/pytest/selenium/cypress/playwright together with agile/scrum/tdd/microservices/rest_api. In 2026 the latter migrated into the Java/enterprise cluster, leaving Comm 5 as a bare 4-node (cypress/jest/playwright/selenium) test-tools island. Testing practice vocabulary moved, testing tools didn't.

### Calibration ratio (Step 5 check against Agent Prep's 24.8x)

Recomputed within the SWE analytical frame (not the pipeline-wide frame Agent Prep used):

- Within-2024 Cohen's d (arshkon vs asaniczka): 0.165 (arshkon slightly richer)
- Cross-period Cohen's d (arshkon vs scraped): -0.355 (scraped much richer)
- **Calibration ratio: -2.15** (direction reversed)

The ratio is smaller than Agent Prep's 24.8 (they used the full pipeline frame; we use SWE-only non-aggregator). But the direction-reversal plus the magnitude (|d|=0.355 cross-period vs 0.165 within-2024) still makes tech_count one of the strongest real signals above source-variation noise. **Confirmed as a real Wave 2 finding.**

Length-normalized complement: `tables/T14/stack_diversity_density.csv`. Median density (techs per 1K chars) in 2024 ALL = 3.02, in 2026 ALL = 2.27. **After length-normalization, tech density actually DECREASED by 25%.** Postings got longer faster than they got more tech-dense. The raw +60% median-count signal is therefore heavily driven by description-length inflation (see T05: +57% median length).

**Load-bearing finding:** "Stack breadth is expanding" is real but only in raw count. Postings aren't packing more techs per sentence — they are writing more sentences, each with a moderate tech density. See Step 6 for the AI-subgroup version of the same pattern.

## 7. Structured vs extracted validation

Mapped asaniczka structured skill strings to the 153-tech taxonomy via regex rules (`structured_to_tech` in the script). 109 technologies have non-trivial prevalence in at least one representation.

**Rank correlation: Spearman rho = 0.77, Pearson r = 0.92.**

### Largest divergences (sorted by |rate_extracted - rate_structured|)

| Tech | Extracted % | Structured % | Diff | Interpretation |
|---|---|---|---|---|
| c_cpp | 0.9 | 16.2 | +15.4 | **Extractor bug — `\bc\+\+\b` regex cannot match** |
| security | 35.6 | 24.0 | -11.6 | Extractor over-fires (cybersecurity mentions, "security clearance") |
| csharp | 0.4 | 11.7 | +11.3 | **Extractor bug — C# mentioned in text as "C#" is not matching** |
| cicd | 18.5 | 10.6 | -7.9 | Description-extract catches more CI/CD variants than structured skills |
| typescript | 12.7 | 6.4 | -6.3 | Extractor over-fires or structured under-reports |
| bash_shell | 6.1 | 0.0 | -6.1 | Structured field doesn't have "Bash/Shell" as a canonical skill |
| c_lang | 5.6 | 0.0 | -5.6 | Structured field has only C++ / C#, not plain "C" |
| sass_less | 4.5 | 0.0 | -4.5 | No structured equivalent |
| sql | 26.3 | 22.6 | -3.7 | — |
| javascript | 21.3 | 17.9 | -3.4 | — |
| jquery | 3.3 | 0.0 | -3.3 | — |

**Conclusions:**
1. Strong convergent validity overall (rho=0.77).
2. **The description extractor has two concrete bugs:** `c_cpp` regex (`\bc\+\+\b`) and `csharp` regex miss most real mentions. Since C++ is structurally 16% in asaniczka and only 0.9% extracted, this is a large miss. Any "C++ is declining" claim is currently unfounded — the tool has a blind spot.
3. The structured field misses shell/bash and sass/less entirely — LinkedIn's skill vocabulary doesn't include them as canonical entries.
4. "security" runs richer in extraction because the word is abundant in compliance / clearance boilerplate that isn't really a technology skill.

**Recommendation (for the next re-run of `build_remaining.py`):** replace `\bc\+\+\b` with `(?:^|[\s,/()])c\+\+(?:$|[\s,/()])` and `c#` similarly; expand c_lang pattern; re-generate swe_tech_matrix.parquet.

## Surprises & unexpected patterns

1. **AI cluster fusion.** The 2024 "modern AI" fragmented islands (Claude API + Hugging Face, Copilot + Codex, traditional ML) fuse into a single 29-member community in 2026, roughly the same size as the cloud/DevOps community. This is the single largest structural rearrangement in the network.
2. **Data engineering annexes OLTP.** MongoDB/MySQL/PostgreSQL/Redis/SQL all moved out of the "enterprise web" community into the data-engineering community by 2026. Implies "database stuff" is increasingly signaling data-eng work, not generic backend.
3. **Stack diversity grows only in raw count.** Density (techs / 1K chars) DECREASED 25% from 2024 to 2026. The "broader stack" is a length-inflation artifact to a large extent — each individual sentence isn't more tech-dense.
4. **AI-mentioning share 9.4x.** From 2.8% of postings (2024) to 26.2% (2026). This is the headline validation of T14's AI-integration hypothesis.
5. **Data-quality finding: C++ / C# regex bugs.** The shared tech matrix's `c_cpp` pattern (`\bc\+\+\b`) cannot match because `+` is not a word character, so `\b` after `+` fails. `csharp` has the same issue. The structured-skills cross-check reveals C++ is actually mentioned in ~16% of asaniczka structured skills but 0.9% in extraction. **This should be fixed and the shared artifacts rebuilt before downstream analysis makes claims about C-family language prevalence.**
6. **"Communication" and "Troubleshooting" in structured top-20.** LinkedIn's structured skills field mixes soft and hard competencies; any structured-field-only analysis must filter the vocabulary.
7. **Test tools decoupled from practices.** In 2024 test frameworks co-occurred with Agile/Scrum/TDD; in 2026 tools form a bare 4-node island while practice vocabulary migrated to the Java/enterprise community. Testing as practice moved; testing as tooling didn't.
8. **"Security" is the one tech where entry-level underperforms in 2024 pooled data.** Security practices concentrate at mid-senior — interesting baseline for a downstream study on whether AI-era entry-level now has to "own security" earlier.
9. **Declining techs are all legacy-web.** Every declining technology fits a single semantic cluster (jquery/html/css/.net/sqlserver/c_lang), reinforcing that the shift is not random attrition but a coherent generational rotation away from classic Microsoft/legacy-web stacks.

## Sensitivities applied

- **(a) Aggregator exclusion:** applied throughout (11,957 rows removed).
- **(b) Company capping:** not applied in T14. Top-50 company concentration in scraped is moderate (HHI 0.003-0.014 from T06); unlikely to swing a 63K-row trajectory analysis but a formal cap should be run as a robustness check in the follow-up.
- **(f) Within-2024 calibration:** applied to tech_count (ratio -2.15, real). AI-mentioning prevalence growth (2.8% -> 26.2%) calibrates off Agent Prep's finding (ratio -16). Both survive.
- **(g) SWE classification tier robustness:** not split in this pass. `swe_classification_tier` is in the metadata table and can be filtered in a re-run if needed.
