# T04: SWE Classification Audit

## Headline Finding

The SWE classifier is **reliable for its core regex tier** (91.4% of SWE rows, mostly high-confidence pattern matches) but has meaningful quality issues at the margins. The LLM cross-check estimates a **13.5% false-positive rate** (rule says SWE, LLM says NOT_SWE) and an **8.5% false-negative rate** (rule says not-SWE, LLM says SWE). The title_lookup_llm tier is the weakest link, with an elevated FP rate. QA/test roles are the primary source of contamination in the SWE sample: 23 of 50 borderline SWE titles (46%) contain QA, test, automation, or firmware keywords. Zero dual-flag violations exist. Boundary roles (ML Engineer, Data Engineer, DevOps, AI Engineer) are classified **consistently across sources/periods**, indicating no temporal classification drift.

## Methodology

- **Population:** LinkedIn, English, date_flag=ok rows (1,344,441 total; 63,294 SWE; 23,366 SWE-adjacent; 152,211 control).
- **Tier breakdown:** Counted SWE and non-SWE rows by swe_classification_tier.
- **Borderline SWE sample:** 50 rows with swe_confidence 0.3-0.7 or tier=title_lookup_llm, reviewed for quality.
- **Borderline non-SWE sample:** 50 rows with "engineer", "developer", or "software" in title but is_swe=False.
- **FP/FN estimation:** Cross-tabulated rule-based is_swe vs swe_classification_llm where available (36,057 rows).
- **Dual-flag check:** Verified (is_swe + is_swe_adjacent + is_control) never exceeds 1.
- **Boundary-case analysis:** Tracked SWE classification rates for key straddler roles across sources.

## Results

### 1. SWE classification tier breakdown

**SWE rows (is_swe=True): 63,294 total**

| Tier | Count | % |
|---|---|---|
| regex | 57,879 | 91.4% |
| title_lookup_llm | 4,459 | 7.0% |
| embedding_high | 956 | 1.5% |

**Non-SWE rows with a classification tier:**

| Tier | Count | % of non-SWE |
|---|---|---|
| unresolved | 1,143,062 | 89.2% |
| regex (excluded patterns) | 125,281 | 9.8% |
| embedding_adjacent | 7,493 | 0.6% |
| title_lookup_llm | 5,311 | 0.4% |

91.4% of SWE classifications come from high-confidence regex matching. The remaining 8.6% (title_lookup_llm + embedding_high) are lower-confidence and account for most of the false-positive risk.

### 2. Borderline SWE quality assessment

Of 50 sampled borderline SWE postings (all title_lookup_llm tier, confidence 0.90):

- **23/50 (46%) have suspicious titles**: QA Engineer (6), Automation Engineer (5), Test Engineer (6), Firmware Engineer (2), Data Science Engineer (1), MLOps Engineer (2), Azure Lead Engineer (1).
- Many QA/test roles write test code but do not write production software. These are borderline by any reasonable definition.
- The title_lookup_llm tier assigns uniform 0.90 confidence, providing no gradation for borderline cases.

**Assessment:** The QA/test engineer contamination is the largest source of classification noise. Depending on the research definition of SWE (does writing test code count?), somewhere between 20-46% of borderline SWE labels could be considered false positives. For a labor market study focused on software development roles, QA roles with no software development responsibilities should arguably be SWE-adjacent rather than SWE.

### 3. Borderline non-SWE quality assessment

Of 50 sampled non-SWE rows with "engineer"/"developer"/"software" in the title:

- **Only 3/50 (6%) appeared to be genuine SWE false negatives** (e.g., roles that involve writing production software but were missed).
- Most non-SWE titles with these keywords are correctly excluded: Structural Engineer, Civil Engineer, Mechanical Engineer, Sales Engineer, Electrical Engineer, Project Engineer, Instrumentation Engineer, etc.
- Several borderline cases are correctly classified as SWE-adjacent rather than SWE: "Software Engineering MTS", "Continuous Integration Development Engineer", "Sr. Network Engineer".

**Assessment:** The false-negative rate is low for clear SWE roles. The classifier correctly excludes non-software engineering disciplines.

### 4. SWE-adjacent and control profiles

**Top SWE-adjacent titles:** Network Engineer (1,359), Quality Engineer (1,228), Data Scientist (1,199), Data Analyst (1,011), Architect (538), Security Engineer (373), Solutions Architect (351), Windows Engineer (333), Systems Administrator (256). These are clearly distinct from core SWE roles -- they involve technical work but not primarily writing/maintaining software.

**Top control titles:** Accountant (7,014), Registered Nurse (3,730), Electrical Engineer (2,997), Mechanical Engineer (2,243), Financial Analyst (2,226), Accounting Manager (2,160). The control group spans both technical (EE, ME) and non-technical (accounting, nursing) occupations for comparison.

**SWE-adjacent tier distribution:** regex (10,812), embedding_adjacent (7,493), title_lookup_llm (5,061).

### 5. False-positive and false-negative rates (LLM cross-check)

Where both rule-based and LLM labels are available (36,057 rows):

| | LLM: SWE | LLM: SWE_ADJACENT | LLM: NOT_SWE |
|---|---|---|---|
| Rule: SWE (9,659) | 7,493 (77.6%) | 861 (8.9%) | 1,305 (13.5%) |
| Rule: not-SWE (26,398) | 2,236 (8.5%) | 3,922 (14.9%) | 20,240 (76.7%) |

**Estimated FP rate: 13.5%** (rule=SWE, LLM=NOT_SWE). An additional 8.9% are reclassified to SWE_ADJACENT.
**Estimated FN rate: 8.5%** (rule=not-SWE, LLM=SWE).

**FP rate by tier:**

| Tier | SWE (agree) | SWE_ADJACENT | NOT_SWE (disagree) | Total | FP rate |
|---|---|---|---|---|---|
| regex | 7,042 | 545 | 1,095 | 8,682 | 12.6% |
| title_lookup_llm | 378 | 274 | 186 | 838 | 22.2% |
| embedding_high | 73 | 42 | 24 | 139 | 17.3% |

title_lookup_llm has the highest FP rate (22.2%), followed by embedding_high (17.3%). Even regex has 12.6% disagreement with LLM, though some of this may be LLM error rather than rule error.

### 6. Dual-flag violations

**Zero violations.** No row has more than one of (is_swe, is_swe_adjacent, is_control) set to True.

1,105,570 rows (82.2%) have none of the three flags set (general LinkedIn postings not in any study category).

### 7. Boundary-case temporal consistency

SWE classification rates for key boundary roles across sources:

| Role | Arshkon SWE% | Asaniczka SWE% | Scraped SWE% | Temporal Drift? |
|---|---|---|---|---|
| ML Engineer | 90.9% | 96.7% | 93.8% | No |
| Data Engineer | 91.6% | 85.5% | 93.2% | Minor (+7.7pp arshkon vs asaniczka) |
| DevOps | 97.7% | 99.3% | 99.3% | No |
| Platform Engineer | 90.7% | 86.3% | 85.2% | Minor drift down (-5.5pp) |
| Cloud Engineer | 98.1% | 96.2% | 91.2% | Moderate drift down (-6.9pp) |
| AI Engineer | 87.9% | 98.6% | 85.7% | Moderate (-12.9pp asaniczka to scraped) |

**Key finding:** No catastrophic temporal drift. Most roles are stable within +/-5pp. AI Engineer shows the largest instability: 98.6% SWE in asaniczka (small N=72) vs 85.7% in scraped (N=1,302). This is likely driven by the emergence of "AI Engineer" as a distinct non-SWE role category in 2026 (e.g., prompt engineering, AI evaluation roles that do not write software). Cloud Engineer similarly drops from 98.1% to 91.2%, possibly as cloud infrastructure roles diversify.

### 8. SWE confidence distribution

| Confidence bin | SWE count | Non-SWE count |
|---|---|---|
| 0.0-0.3 | 0 | 1,016,355 |
| 0.3-0.5 | 0 | 76,153 |
| 0.5-0.7 | 0 | 50,554 |
| 0.7-0.85 | 0 | 18,305 |
| 0.85-0.95 | 5,363 | 5,311 |
| 0.95-1.0 | 57,931 | 114,469 |

The classifier uses an effective 0.85 threshold: all SWE rows have confidence >= 0.85. The 0.85-0.95 bin contains 5,363 SWE and 5,311 non-SWE rows, indicating this band is essentially a coin flip. Above 0.95, the ratio shifts heavily toward correct classification.

## Surprises and Unexpected Patterns

1. **QA/test roles are the dominant source of SWE classification noise.** 46% of borderline SWE titles are QA, test, or automation roles. This is a substantive research decision (should QA engineers count as SWE?), not just a technical classification error.

2. **The LLM finds 2,236 SWE rows among rule-classified non-SWE postings.** These are not in the current SWE sample. If the FN rate of 8.5% holds for the full non-SWE population, approximately 100K+ rows may be misclassified as non-SWE. However, the LLM sample is not random -- it is drawn from the LLM-selected frame which may over-represent borderline cases.

3. **The title_lookup_llm tier's uniform 0.90 confidence is misleading.** All sampled borderline SWE rows have exactly 0.90 confidence, suggesting the tier assigns a fixed score rather than a calibrated probability. This means confidence cannot distinguish good from bad title_lookup_llm classifications.

4. **"AI Engineer" classification is becoming less stable over time.** In 2024 asaniczka data, 98.6% of AI Engineer titles were classified as SWE. In 2026 scraped data, this drops to 85.7%. The emergence of non-coding AI roles (prompt engineering, AI ethics, AI operations) is creating a new boundary that did not exist at the time of the classifier's design.

5. **1.1 million rows have no classification flag at all** (not SWE, not adjacent, not control). These are the general job posting population. This is expected but worth noting -- 82% of the filtered dataset is background.

## Data Quality Caveats

1. **FP/FN rates are estimated from LLM cross-check**, which itself has unknown error rates. The true FP/FN rates may differ.
2. **LLM labels are available for only 9,659 SWE rows** (15.3% of SWE population). The FP/FN estimates assume this subset is representative.
3. **Borderline samples are not strictly random** -- they oversample title_lookup_llm tier postings. The 46% suspicious-title rate applies to this borderline set, not to all SWE rows.
4. **The regex-tier FP rate of 12.6% seems high** for a pattern-matching approach. This may include LLM classification errors rather than regex errors (e.g., the LLM may incorrectly classify some genuine SWE roles as NOT_SWE).

## Recommendation

1. **The current SWE classification is adequate for primary analyses.** 91.4% of SWE labels come from high-confidence regex, and overall accuracy is reasonable. The ~13.5% FP rate is a known limitation that should be reported but does not invalidate the sample.

2. **For sensitivity analyses, consider excluding title_lookup_llm tier** (removes 4,459 rows / 7% of SWE) to test whether findings are robust to the highest-FP-rate tier.

3. **QA/test contamination should be acknowledged** in the methods section. Consider creating a "core SWE" subsample that excludes QA/test titles for robustness checks.

4. **Monitor AI Engineer classification over time.** As AI roles diversify, the current regex patterns may need updating to distinguish coding AI engineers from non-coding AI roles.

5. **Do not use swe_confidence for filtering within tiers** -- the title_lookup_llm tier assigns uniform 0.90, making confidence useless for separating good from bad cases within that tier.

## Action Items

1. **Report the 13.5% FP rate** in any analysis that uses the SWE sample. Frame it as a known limitation.
2. **Run a sensitivity check** excluding title_lookup_llm tier rows from downstream analyses.
3. **Create a "QA-excluded SWE" subsample** for analyses where the distinction between test code and production code matters.
4. **Cross-validate boundary role classification** for AI Engineer in future pipeline runs -- this is the most temporally unstable classification.
5. **Flag the uniform 0.90 confidence issue** for title_lookup_llm as a pipeline improvement opportunity.
