# T01: Data Profile & Column Coverage

## Headline Finding

The unified dataset contains **1,502,404 rows** across three sources. After applying default filters (LinkedIn, English, date_flag=ok), **1,344,441 rows** remain, of which **63,294 are SWE** (4.7%). Column coverage is highly uneven across sources: asaniczka lacks company_industry, company_size, and entry-level seniority labels; scraped has only 20.7% description_core_llm coverage for SWE rows; and LLM classification columns cover only 1-10% of rows depending on source.

## 6. Key Constraint Mapping for Downstream Tasks

### Text analyses (embeddings, topic models, requirement extraction)

**Binding constraint:** description_core_llm coverage at 20.7% for scraped SWE.
**Severity:** HIGH. Only ~7,300 scraped SWE rows have LLM-cleaned text vs ~26,000 from the 2024 sources. This creates an asymmetric comparison base. Using description_core as fallback introduces boilerplate noise.
**Mitigation:** Use description_core_llm where available; run sensitivity on description_core; report text_source distribution. For keyword-presence analyses, raw description is acceptable.

### Seniority analyses

**Binding constraint:** seniority_llm coverage too thin (1.9-10.0% known) to serve as primary. seniority_native missing entry for asaniczka.
**Severity:** HIGH for entry-level trend analysis. The arshkon-only entry-level baseline is 769 SWE rows (seniority_native) or 848 (seniority_final).
**Mitigation:** Use seniority_final as primary with seniority_native ablation. Use arshkon-only for 2024 entry-level baseline. seniority_llm is available for ablation where labeled but cannot drive the analysis.

### Geography analyses

**Binding constraint:** metro_area coverage varies 61-78% across sources.
**Severity:** MODERATE. ~25-40% of SWE rows lack metro assignment.
**Mitigation:** state_normalized at 89-98% is a viable fallback for geographic stratification.

### Company analyses

**Binding constraint:** company_industry is 0% for asaniczka and has different label granularity between arshkon (374 values) and scraped (5,134 values).
**Severity:** HIGH for industry stratification. Company_size limited to arshkon only for SWE.
**Mitigation:** Industry analysis requires harmonization and can only cover arshkon (2024-04) vs scraped (2026). company_name_canonical is usable cross-period.

### Requirement extraction / ghost detection

**Binding constraint:** ghost_assessment_llm thin (10-62% by source). yoe_extracted at 60-72% is the best cross-period requirement signal.
**Severity:** MODERATE for ghost analysis (LLM ghost requires more budget). LOW for YOE analysis.
**Mitigation:** Rule-based ghost_job_risk available at 100% but very conservative (519 non-low). YOE is adequate.

## Surprises & Unexpected Patterns

1. **Scraped data now has two periods.** The scraper has continued into April 2026, producing 69,323 rows in `period=2026-04` in addition to 145,280 in `2026-03`. This was not anticipated in the schema documentation. Downstream analyses should decide whether to pool or separate these.

2. **is_remote_inferred is 0% everywhere.** Despite the description having remote-related text in many scraped postings (22.3% have is_remote=true from the source), the text-inference flag catches none. This suggests the inference logic is either broken or very conservative — it is not a viable replacement for the source flag.

3. **seniority_llm is much thinner than expected.** At 1.9-10.0% known-seniority coverage for SWE, it cannot serve as the primary seniority variable despite being designed for that purpose. The LLM classification budget was insufficient to cover even the SWE sample. This contradicts the schema documentation which positions it as "primary analysis variable."

4. **description_core_llm asymmetry.** Arshkon SWE has 99.2% coverage while scraped SWE has 20.7%. This means text-dependent 2024-vs-2026 comparisons will either use a small scraped subset or fall back to lower-quality text. The 89.7% coverage for asaniczka SWE is better than expected.

5. **Scraped row count is larger than documented.** Schema says ~146K scraped linkedin rows; actual is 214,707 (with ~69K in the new 2026-04 period).

6. **4 undocumented columns exist:** `analysis_group`, `analysis_in_scope`, `eligible_for_extraction`, `llm_text_skip_reason` are present but not in the schema reference.

## Data Quality Caveats

- Row counts for scraped source will grow with continued scraping. Numbers above are as of the 2026-04-08 pipeline run.
- The 9 scraped rows with `date_flag=date_posted_out_of_range` are negligible.
- Description quality is excellent: only 23 SWE rows flagged too_short across all sources.
- The total column count (105) exceeds the documented 101, suggesting the schema needs updating.

## Action Items

1. **Pipeline: Increase LLM budget for scraped SWE.** The 20.7% description_core_llm coverage for scraped SWE is the single biggest data quality gap. Running additional LLM budget would close this.
2. **Pipeline: Investigate is_remote_inferred.** The 0% rate everywhere suggests the inference logic needs repair.
3. **Schema: Document the 4 new columns** and update row counts.
4. **Analysis: company_industry harmonization** needed before any industry-stratified cross-period comparison (374 vs 5,134 distinct values).
5. **Downstream tasks should treat seniority_final as the primary seniority variable**, not seniority_llm, given the coverage gap. seniority_llm is an ablation check only.
