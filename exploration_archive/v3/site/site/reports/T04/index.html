    T04: SWE Classification Audit

    Task: Assess SWE classification quality after preprocessing fixes.

    Date: 2026-04-05 Data: data/unified.parquet (Stage 11 final output, 1.40M rows, 99 columns) Filters: source_platform='linkedin' AND is_english=true AND date_flag='ok'

    2. Dual-Flag Violations

    Violations (is_swe + is_swe_adjacent + is_control > 1): 0

    No rows are simultaneously classified as multiple categories. The mutual exclusivity constraint is perfectly enforced.

    Unclassified rows: 1,061,002 (83.2% of 1,275,355 filtered rows) are neither SWE, SWE-adjacent, nor control. These are expected -- the classification targets specific occupation groups, not the entire job market.

    3. Borderline SWE Sample Audit (title_lookup_llm tier)

    Reviewed 50 SWE postings from the title_lookup_llm tier (all conf=0.900). Manual quality assessment based on title and description snippet:

    Clear true positives (correct SWE classification): ~50%

    • "ReactJS Developer" -- frontend software development
    • "Principal Firmware Engineer" -- embedded software
    • "Senior MLOps Engineer" -- ML infrastructure
    • "Angular developer" / "Angular Lead Developer" -- frontend software
    • "DevSecOps Engineer" / "Senior DevSecOps Engineer" -- security-aware software development
    • "Lead Developer" / "Senior Developer" / "Developer" -- general software development
    • "Embedded Engineer" -- embedded software
    • "Java Architect" -- software architecture
    • "Database Engineer II" -- database software
    • "Programmer" / "Senior Programmer Analyst" -- software development
    • "Distinguished Engineer, Generative AI Systems" -- senior software/AI

    Borderline cases (debatable): ~25%

    • "Quality Assurance Engineer" (multiple instances) -- QA roles involving some automation/code but primarily testing
    • "Senior Quality Assurance Automation Engineer" -- test automation, borderline SWE
    • "QA Lead" (multiple instances) -- QA leadership
    • "Automation Test Analyst" -- test automation
    • "Software Quality Assurance Engineer" / "Senior Software Quality Engineer" -- software QA
    • "QA Automation Lead Engineer" -- test automation leadership
    • "Software Engineering Manager" -- management role, arguably should be excluded

    Likely false positives: ~15%

    • "Senior Automation Engineer" (National Carwash Solutions) -- industrial/manufacturing automation
    • "Test Engineer" (description: "safety and performance standards", "sustaining engineering") -- hardware/product testing
    • "Staff Systems Test Engineer" (Northrop Grumman) -- defense systems testing
    • "Test Development Engineer" -- ambiguous, could be hardware test development
    • "Principal Test Engineer" (BAE/defense context) -- defense systems testing

    Quality estimate for title_lookup_llm tier:

    • True SWE: ~50% (clear software development roles)
    • Borderline/debatable: ~25% (QA/test automation roles involving code)
    • Likely false positives: ~15% (hardware, manufacturing, defense testing roles)
    • Management roles that may warrant separate treatment: ~10%

    4. Borderline Non-SWE Audit

    Reviewed 50 non-SWE rows with "engineer", "developer", or "software" in the title. Assessment:

    Correctly excluded (not SWE):

    • Hardware/traditional engineering: Mechanical Engineer, Electrical Engineer, Civil Engineer, Senior Structural Engineer -- correctly excluded
    • Network/telecom: Senior Telecom / Network Engineer, Network Engineer Wired -- correctly classified as non-SWE (these are SWE-adjacent or control)
    • Industrial/manufacturing: Lean Manufacturing Engineer, Industrial Engineer, Process Line Engineer -- correctly excluded
    • Non-IC roles: Engineering Project Manager, Engineer Supervisor -- correctly excluded
    • Specialty engineering: Audio Visual Engineer, Geotechnical Engineer, Traffic Engineer, Water/Wastewater Engineer -- correctly excluded
    • Sales-adjacent: Workday Sales Engineer -- correctly excluded

    Potential false negatives identified:

    • "Ruby on Rails Developer" (conf=0.637, unresolved) -- this appears to be a genuine SWE role that was missed
    • "Principal Engineer, YouTube ML Recommendations Systems" (conf=0.601, unresolved) -- strong SWE signals in description
    • "Senior Deep Learning Engineer" (conf=0.814, embedding_adjacent) -- classified as adjacent rather than SWE
    • "Senior Business Intelligence Developer" (conf=0.605, unresolved) -- borderline SWE/adjacent

    Estimated false negative rate from this sample: ~4-8% (2-4 of 50)

    The exclusion rules for non-software engineering disciplines appear robust. Most correctly-excluded rows are hardware engineering, civil/structural, or management roles.

    5. False Positive Estimation

    Method: Software keyword scan in description

    Checked whether SWE-classified descriptions mention software/code/programming-related terms (software, programming, coding, developer, python, java, javascript, typescript, c++, api, frontend, backend, full-stack, devops, deploy, repository, git, codebase):

    Tier Total Has SW Signal No SW Signal No-SW %
    regex 47,585 45,367 2,218 4.7%
    title_lookup_llm 3,948 3,482 466 11.8%
    embedding_high 794 741 53 6.7%
    Total 52,327 49,590 2,737 5.2%

    Not all rows without software keywords are false positives -- some postings use domain-specific language. But this provides a reasonable upper bound on false-positive risk.

    Suspicious title categories within SWE:

    Title Pattern SWE Count Mentions "software" Mentions mfg/hardware
    %automation engineer% 545 56% 44%
    %reliability engineer% 989 56% 32%
    %test engineer% 699 73% 68%
    %quality assurance% 278 60% 41%
    %qa engineer% 162 77% 22%

    Key findings: - Test engineer has the highest manufacturing keyword co-occurrence (68%), meaning nearly 2/3 of SWE-classified test engineers mention manufacturing or hardware in their descriptions. Many of these are likely product/hardware test roles, not software testers. - Automation engineer has 44% manufacturing overlap, indicating a substantial fraction are industrial automation roles. - Reliability engineer has 32% manufacturing overlap, lower than test/automation but still indicating meaningful contamination. - QA engineer is the cleanest of the suspicious categories at 22% manufacturing overlap.

    Estimated false positive rate:

    • Regex tier (90.9% of SWE): ~3-5% false positives. Regex patterns like "software engineer" are highly precise.
    • Title_lookup_llm tier (7.5%): ~15-25% false positives. This tier includes "automation engineer", "reliability engineer", "test engineer" which conflate software and hardware/manufacturing roles.
    • Embedding_high tier (1.5%): ~7-10% false positives. Slightly elevated but small absolute count.
    • Overall weighted estimate: ~4-6% false positives (~2,100-3,100 rows out of 52,327).

    6. False Negative Estimation

    893 non-SWE rows match strong SWE title patterns (software engineer, software developer, full-stack, backend engineer, frontend engineer, web developer).

    Breakdown by category:

    Category Count Notes
    Management 610 Director/Manager of SW Engineering -- intentional exclusion
    Other 159 Mixed -- includes teaching, editor, client technology roles
    Intern 61 Internship roles
    Senior IC 58 Senior/Staff/Principal-prefixed roles caught by edge cases
    Advisory 5 Consultants

    Key insight: 610/893 (68.3%) of potential false negatives are management roles (e.g., "Director, Software Engineering - Card Core", "Manager, Software Engineering, Back End"). These are intentionally excluded from SWE because they are not IC development roles. This is a correct design decision for a study of SWE posting requirements.

    Top management false-negative titles are dominated by Capital One's naming convention ("Director, Software Engineering - Card Core", "Senior Director, Software Engineering - Card Authorizations", etc.), which inflates the count due to multi-posting.

    The remaining ~283 potential false negatives include: - Interns (61): Software engineering internships -- excluded by level, arguably correct - Teaching (21): "College of IT: FT Instructor - Software Engineering" -- correctly excluded - Senior IC edge cases (58): Roles with unusual titling that the regex doesn't capture - Other (159): Titles like "Client Technology - Software Engineering" where the role may or may not be IC development

    Estimated false negative rate: ~283 genuine misses / ~53,000 target = ~0.5%, which is negligible.

    This is the analytically critical part of the audit: do boundary roles shift classification across periods?

    Machine Learning Eng. 100% (87/87) 91.5% (375/410) 98.3% (738/751) Yes
    DevOps 97.7% (250/256) 99.3% (754/759) 99.1% (994/1,003) Yes
    Site Reliability 100% (91/91) 99.3% (277/279) 99.8% (412/413) Yes
    Platform Engineer 90.7% (49/54) 86.3% (314/364) 85.0% (340/400) Mostly
    Cloud Engineer 98.1% (105/107) 96.2% (205/213) 90.7% (353/389) Mild shift
    Security Engineer 1.6% (4/249) 0.4% (3/677) 3.1% (40/1,272) Stable (non-SWE)
    QA Engineer 32.3% (10/31) 59.5% (72/121) 42.8% (80/187) Unstable
    Test Engineer 32.3% (54/167) 33.0% (414/1,256) 24.5% (231/942) Mostly
    Solutions Architect 2.3% (3/133) 0.5% (4/754) 0.1% (1/688) Stable (non-SWE)

    Key findings:

    1. Core SWE roles are stably classified. Data Engineer, ML Engineer, DevOps, SRE all classify as SWE at >85% across all periods. These are safe to include in SWE analyses.

    2. Security Engineer is consistently non-SWE. 0.4-3.1% SWE classification. The boundary is stable and correctly placed on the SWE-adjacent side. The small uptick in scraped (3.1%) is worth monitoring but not concerning.

    3. QA Engineer is the most unstable boundary. Classification as SWE ranges from 32.3% to 59.5% across sources. This means QA engineers are being classified inconsistently, and any temporal analysis comparing QA inclusion rates could create spurious trends. The low base count (31-187 per source) amplifies the instability.

    4. Cloud Engineer shows mild reclassification drift. From 98.1% SWE (arshkon) to 90.7% SWE (scraped). This 7.4pp shift could reflect real changes in how cloud engineer roles are described (more infrastructure-focused in 2026), or could be a classifier artifact.

    5. Platform Engineer is borderline stable with 85-91% SWE classification. The boundary is placed correctly (mostly SWE), but ~10-15% of platform engineers are classified as adjacent.

    6. Test Engineer classification is declining. From 32-33% SWE (2024) to 24.5% (2026). This 8pp drop could reflect real role evolution (more specialized testing, less code) or title convention changes.

    Risk assessment for temporal analysis:

    The unstable QA and drifting test engineer boundaries together affect: - QA Engineer: 10-80 SWE rows per source (very small) - Test Engineer: 54-414 SWE rows per source (moderate)

    Combined, these represent ~1,000-1,500 rows within SWE, or roughly 2-3% of the 52K SWE sample. A 20-30pp swing in their classification rate could shift SWE counts by ~200-400 rows, which is small relative to total SWE but could matter for compositional analyses within specific role subcategories.

    Recommendation: For sensitivity analysis, report results with and without QA/test engineer titles.

    9. Embedding_high Tier Audit

    The 794 embedding_high SWE rows show distinctive title patterns:

    Top titles: - "ENGINEER SYSTEMS 3" (10), "ENGINEER SOFTWARE 3" (4) -- reversed title formats, likely parsing artifacts - "Principal Engineer Software" (7), "Sr Principal Engineer Software" (6) -- valid but unusual formatting - "AI Research Engineer" (6), "AI Native Engineer" (5) -- genuine SWE roles with non-standard titles - "ML Ops Engineer" (5), "Senior ML Ops Engineer" (4) -- valid MLOps roles - "Test Automation Lead" (4), "Software Test Automation Engineer" (4) -- QA/test automation - "QA/QC Engineer" (3), "QA Test Lead" (3), "QA Engineer Automated" (3) -- QA roles

    Assessment: The embedding_high tier contains a mix of: - Genuine SWE roles with unusual title formatting (~50-60%) - QA/test automation roles (~20-25%) - Parsing artifacts with reversed name/title fields (~10-15%) - Edge cases (~10%)

    The reversed-title artifacts (e.g., "ENGINEER SYSTEMS 3") suggest a data quality issue upstream of classification. These rows should be audited for title-field swapping.

    The classifier is conservative -- it has a low false-negative rate but a moderate false-positive rate. The false positives are concentrated in the title_lookup_llm tier (7.5% of SWE), where industrial automation, reliability, and hardware test engineers bleed in. The regex tier (90.9%) is highly precise.

    11. Surprises and Unexpected Patterns

    1. Test engineer has the highest manufacturing co-occurrence (68%). Nearly 2/3 of SWE-classified test engineers mention manufacturing or hardware keywords. This is the single highest-risk boundary for false positives and the most compelling case for targeted exclusion rules.

    2. "Ruby on Rails Developer" is classified as non-SWE. This is a clear false negative (conf=0.637, unresolved tier). A dedicated regex pattern for "Rails Developer" would fix this and similar framework-named-developer titles.

    3. "Senior Deep Learning Engineer" is SWE-adjacent, not SWE. Classified via embedding_adjacent (conf=0.814). Given that deep learning engineering is heavily code-based, this seems like a misclassification that an expanded regex could fix.

    4. Capital One dominates false negatives. The management false-negative list is heavily skewed by Capital One's titling convention ("Director, Software Engineering - Card Core", "Manager, Software Engineering, Android"). This is an artifact of Capital One posting many distinctly-titled management roles, not a systemic classifier problem.

    5. Embedding_high contains reversed title artifacts. "ENGINEER SYSTEMS 3" and "ENGINEER SOFTWARE 3" appear to be cases where name/title fields were swapped during ingestion. These 20+ rows have correct classification despite garbled titles, suggesting the embedding classifier is robust to title noise.

    6. Scraped data has 94.2% regex purity vs 87.9% for asaniczka. The scraper's search queries naturally select for postings with explicit SWE title keywords, reducing reliance on noisier classification tiers. This is a beneficial side effect that improves classification quality for the scraped period.

    12. Recommendations

    1. For primary SWE analyses: Use is_swe = true with the understanding that 4-6% false positives are present. This is acceptable for most analyses.

    2. For precision-sensitive analyses: Add filter swe_classification_tier = 'regex' to restrict to high-precision titles (90.9% of sample). This drops title_lookup_llm and embedding_high rows.

    3. For robustness checks: Report results with and without:

    4. QA/test engineer titles (unstable boundary)
    5. Automation/reliability engineer titles (manufacturing contamination)

    6. title_lookup_llm tier rows

    7. Pipeline improvement (regex expansion): Add regex patterns for "firmware engineer", "MLOps engineer", "DevSecOps engineer", "AI engineer", "Rails developer", "deep learning engineer" to promote these from title_lookup_llm/embedding to the more precise regex tier.

    8. Pipeline improvement (manufacturing exclusion): Add manufacturing/hardware exclusion rules for "test engineer", "reliability engineer", and "automation engineer" -- exclude when title lacks software/code qualifiers and description contains manufacturing/hardware signals.

    9. Data quality: Investigate reversed-title artifacts in embedding_high tier (e.g., "ENGINEER SYSTEMS 3"). These likely reflect upstream ingestion issues.
